Agent Harness Architectural Dimensions

A taxonomy of the design axes that define an agent system's architecture.

Reference document, 2026-06-13. Based on analysis of 7 production codebases.

1. Core Loop

The fundamental execution cycle: how the agent receives input, calls the LLM, executes tools, and decides to continue or stop. Every agent has this.

Key Design Decisions

  • Loop nesting depth: Single-level (claude-code) to 4-level (codex). More levels = more separation of concerns but more control flow to trace.
  • State representation: Mutable state passed through iterations vs. event stream vs. in-place message list mutation.
  • Tool execution timing: After stream completes (batch) vs. during stream (streaming executor).
  • Turn gating: What constitutes a "turn"? Some agents distinguish "turns" (LLM call + tool results) from "iterations" (single LLM call).

2. Tool System

How tools are defined (schema), discovered (registry), called (dispatch), and how their results are normalized and fed back to the LLM.

Key Design Decisions

  • Concurrency model: Sequential-only vs. batch-parallel (Promise.all / asyncio.gather / FuturesOrdered) vs. streaming-executor (tools run during LLM streaming).
  • Tool registration: Hardcoded enum vs. plugin/MCP-based discovery vs. file-system scanning.
  • Result normalization: Size truncation, file offloading for large results, "terminate" flag for short-circuiting the loop.
  • Sandboxing: At what layer are tools sandboxed? Tool-level (bash sandbox), process-level, or container-level?

3. Context Management

How the conversation context (messages sent to the LLM) is composed, managed, and compacted to stay within the model's context window.

Key Design Decisions

  • Compaction strategy: Proactive (before overflow) vs. reactive (after 413 error). Summarization vs. truncation vs. micro-compaction.
  • System prompt composition: Static base + dynamic injection (memory, skills, runtime context) vs. monolithic template.
  • Message budget: Per-message size caps, tool result truncation, conversation history snipping from head.
  • Cache control: Which parts of the prompt are cached? Varies by provider (Anthropic breakpoints vs. automatic).

4. Permission & Safety Model

How the agent decides what actions are allowed, and what mechanisms enforce those decisions.

Key Design Decisions

  • Approval model: Always-ask (interactive) vs. auto-approve (sandbox) vs. ACL-based (permissions per tool/path).
  • Sandboxing: None (trust model) vs. directory-restricted vs. network-disabled vs. full container isolation.
  • Permission persistence: Session-scoped vs. durable across sessions vs. hierarchical (project then user then global).
  • Guardrails: Pre-tool hooks, post-tool validation, content filters, workspace-boundary enforcement.

5. Memory Architecture

How the agent persists state across sessions — user preferences, project conventions, learned skills, conversation history.

Key Design Decisions

  • Memory tiers: Working memory (in-context) vs. short-term (session history) vs. long-term (durable files/DB).
  • Storage format: Markdown files (MEMORY.md, USER.md) vs. SQLite/vector DB vs. JSON blobs.
  • Consolidation: When and how is memory reviewed, summarized, and pruned? Automatic (background) vs. manual.
  • Skill creation: Can the agent create new reusable skills from experience? (hermes-agent's defining feature).

6. Sub-agent & Delegation

How work is split across multiple agent instances — spawning, communication, result collection.

Key Design Decisions

  • Spawn model: Same-process (function call) vs. child process vs. remote/container.
  • Isolation: Shared context (messages visible to parent) vs. sandboxed (only final result returned).
  • Concurrency: Sequential delegation vs. parallel sub-agent fan-out.
  • Budget: Do sub-agents have their own token/turn budgets? (hermes-agent: yes, separate IterationBudget per sub-agent).

7. UI / Transport Layer

How the user interacts with the agent — terminal UI, web, messaging channels, API.

Key Design Decisions

  • Interface paradigm: REPL (interactive terminal) vs. one-shot (CLI arg) vs. daemon (background) vs. chat (messaging).
  • Rendering: TUI library (Ink, Bubble Tea, Textual, custom) vs. plain stdout vs. web UI vs. mobile.
  • Multi-channel: Single interface vs. unified gateway (openclaw: 25+ channels; hermes-agent: multi-platform).
  • Client/server split: Monolithic process vs. separate server + client (opencode: Go TUI + TypeScript server).

8. Error Recovery & Resilience

How the agent handles failures — API errors, context overflow, malformed responses, tool failures.

Key Design Decisions

  • LLM error recovery: Retry with backoff vs. model fallback vs. provider fallback.
  • Context overflow: Auto-compaction vs. reactive compact (after 413) vs. collapse drain (remove oldest turns).
  • Malformed tool calls: Validate and retry (hermes-agent: 3 attempts for invalid JSON, invalid tool names) vs. silently inject error results.
  • Truncated output: Max-tokens recovery (inject continuation message) vs. escalate token limit vs. accept partial.
