Agent Harness Glossary

Canonical terminology for the agent architecture workspace. Definitions are opinionated — they resolve ambiguities in the wider field.

Core Loop

Agent harness
The surrounding infrastructure that manages an LLM-based agent's execution: the loop, tool system, context management, permissions, and session lifecycle. The harness is what remains when you remove the LLM, the tools, and the UI.
Avoid: agent framework, agent runtime (too broad).
Core loop
The fundamental execution cycle: receive input → call LLM → execute tools → feed results back → repeat or stop. Also called the "think-act-observe loop" or "ReAct loop."
Avoid: agent loop (ambiguous with message pump).
Turn
One complete cycle of the core loop: one LLM call plus the execution of any tool calls it produced, ending when either (a) the model returns text-only, (b) stop hooks fire, or (c) an error/abort occurs. A single user request may span multiple turns.
Avoid: step, iteration (these are ambiguous — some codebases use "step" for one LLM call regardless of tools).
Iteration
One LLM API call within a turn. A turn with N rounds of tool execution contains N+1 iterations (1 initial call + N follow-up calls after each tool batch).
Avoid: step (claude-code uses "step" for what we call "turn").
Nested two-loop architecture
A core loop structure with an outer while (true) handling follow-up messages and an inner while (hasToolCalls) handling tool turns. Used by pi-mono and openclaw.
Avoid: double loop (too vague).
Flat while-true architecture
A core loop structure with a single while (true) where all concerns — context prep, LLM call, tool execution, recovery — are sequential phases within one iteration. Used by claude-code and hermes-agent.
Avoid: single loop (ambiguous).

Tool System

Batch execution
Executing all tool calls from an LLM response after the stream completes, in parallel or sequentially. Simpler but higher latency.
Avoid: post-stream execution.
Streaming execution
Executing tool calls as the LLM emits them in the stream, before the full response is complete. Lower latency but requires careful state management.
Avoid: eager execution (not specific enough).
Tool result termination
A tool result flag (terminate: true) that signals the core loop should stop after this tool batch. Used when a tool's output is itself the final answer.
Avoid: final tool, terminal tool.

Context Management

Compaction
Reducing the size of conversation context to stay within the model's context window. May use summarization (LLM generates a summary), truncation (drop oldest messages), or micro-compaction (replace individual tool results with short summaries).
Avoid: compression (implies lossless, which compaction rarely is).
Proactive compaction
Compacting context before an overflow occurs, typically when approaching a token threshold. Prevents wasted API calls that would fail with 413.
Avoid: pre-emptive compaction.
Reactive compaction
Compacting context after receiving a 413 (context too large) error. Simpler to implement but wastes the failed API call.
Avoid: emergency compaction.

Lifecycle

Session
A durable, addressable conversation between a user and an agent. Has an ID, persists across agent restarts, and accumulates message history. May span many user requests.
Avoid: conversation (too generic).
Steering
A user message injected mid-run to interrupt and redirect the agent. The agent finishes its current tool execution, then processes the steering message before the next LLM call.
Avoid: interruption (steering is a specific kind of interruption with content).
Follow-up
A message queued during an agent run that should be processed after the agent would otherwise stop. Used for sub-agent results, notification commands, and multi-step orchestration.
Avoid: deferred message, queued message.
