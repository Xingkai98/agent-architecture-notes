Lesson 0010: Checkpoints, Interrupt/Resume & Streaming

How LangGraph persists state, enables human-in-the-loop, and streams execution at every granularity.

Course 2: Agent Infrastructure · June 2026 · Permalink

Lesson 0009 covered the Pregel execution model — how LangGraph replaces while-loops with graph supersteps. This lesson covers what the runtime does between supersteps: persisting state, pausing for human input, and emitting streaming events. These three capabilities are what make LangGraph more than a graph executor — they make it a production-grade agent runtime.

Checkpoints: state persistence at every step

Without checkpointing, a crashed agent loses everything. The user has to restart from the beginning, repeating work and burning tokens. With checkpointing, execution can resume from exactly where it stopped, fork into parallel branches from any point, or time-travel back to a previous state.

LangGraph's checkpoint system is defined at checkpoint/base/__init__.py. The Checkpoint TypedDict at line 92 captures a complete state snapshot:

class Checkpoint(TypedDict):
    v: int                        # Schema version (currently 4)
    id: str                       # UUID6 checkpoint ID
    ts: str                       # ISO timestamp
    channel_values: dict          # {channel_name: current_value}
    channel_versions: dict        # {channel_name: version_number}
    versions_seen: dict           # {node_name: {channel: version}}
    updated_channels: list[str]   # Channels changed in this step

The versions_seen field is the key to deterministic replay. It records exactly which version of each channel each node consumed. When replaying, the runtime reads the same versions, producing the same outputs.

BaseCheckpointSaver at line 176 defines the persistence interface with four methods: get_tuple(config) fetches a checkpoint, put(config, checkpoint, metadata, new_versions) stores one, put_writes(config, writes, task_id) stores intermediate task-level writes, and list(config) enumerates checkpoints for a thread.

Checkpoints are written at three points in the execution lifecycle (pregel/_loop.py): an input checkpoint (source="input") after mapping input to channels, loop checkpoints (source="loop") after each superstep's after_tick(), and fork checkpoints (source="fork") when branching from history.

The durability model supports three modes: sync (checkpoint written before next step — safest), async (written in background while next step executes — default), and exit (written only when the graph exits — used with DeltaChannel for large histories).

DeltaChannel: O(N²) → O(N)

A naive checkpoint would serialize the entire message list at every superstep. For a run with N steps, this means O(N²) total storage — a 100-step conversation with 1KB messages would produce ~5MB of checkpoints. DeltaChannel fixes this by only snapshotting every Nth step (default 50) and reconstructing intermediate states via ancestor walk:

# DeltaChannel reconstruction (checkpoint/_checkpoint.py:136)
# To read step 73: walk back to checkpoint at step 50, replay writes 51→73
# Storage: O(N) instead of O(N²)

This is the mechanism deepagents uses at graph.py:67 for its DeepAgentState.messages field. Each superstep writes only the new messages; the DeltaChannel reducer stores the delta and periodically snapshots the full history.

interrupt(): the pause button

LangGraph's interrupt mechanism, defined at langgraph/types.py:811, is how human-in-the-loop works. When a node calls interrupt(value), three things happen:

1. Raise GraphInterrupt
Execution stops at this node
→
2. Persist checkpoint
Current state saved to checkpointer
→
3. Emit interrupt event
Client receives the interrupt payload
→
4. Resume with Command
graph.invoke(Command(resume=value))

The Interrupt dataclass at line 533 carries a value (the interrupt payload) and an id (a hash-based unique ID for targeted resumption). The Command dataclass at line 758 provides the resume mechanism: Command(resume=value) answers a pending interrupt, Command(update={...}) modifies graph state before resuming, and Command(goto="node_name") jumps to a specific node on resume.

This is the infrastructure beneath two Course 1 patterns: deepagents' HumanInTheLoopMiddleware (which uses interrupt_on config to gate tool calls) and deer-flow's ClarificationMiddleware (which uses Command(goto=END) to intercept clarification requests).

8 streaming modes: every granularity of observability

LangGraph supports 8 stream modes (defined at langgraph/types.py:120):

The v3 streaming protocol, GraphRunStream at stream/run_stream.py:31, uses a StreamMux to multiplex events to multiple projections. The StreamTransformer abstraction at stream/_types.py:44 enables custom projections — for example, the MessagesTransformer extracts token-by-token deltas from AIMessage chunks for chat UIs.

Deer-flow's worker.py:312 uses agent.astream(input, stream_mode=["values", "messages"]) — values mode for full state snapshots, messages mode for incremental text. The StreamBridge.publish() method converts each LangGraph stream event into an SSE event for the frontend.
