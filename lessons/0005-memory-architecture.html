Lesson 0005: Memory Architecture

Working, short-term, long-term — the three memory tiers that every agent must manage, and how seven codebases do it (or don't).

June 2026 · 7 primary source codebases · Permalink

Memory is what separates a stateless API call from an agent that learns and improves over time. Every agent faces the same fundamental problem: the LLM context window is finite, but conversations, files, and experiences are unbounded. The solution is always a tiered architecture — but the implementations range from full 3-layer orchestration to "we just dump JSONL and call it a day."

In Lesson 0003 we covered how agents manage the LLM context window. In Lesson 0004 we examined access control. This lesson zooms out to the broader question: what does the agent remember, where does it store it, and how does it retrieve the right memory at the right time?

The three tiers of memory

Every agent memory architecture decomposes into exactly three levels:

Working Memory In-context
Active conversation only
~100K tokens
→
Short-Term Memory Session-scoped
Messages, files, notes
Hours to days
→
Long-Term Memory Durable storage
Across sessions, across days
Weeks to months

Working memory is the LLM context window itself — the messages the model can "see" right now. Short-term memory is the session history that survives individual turns but not individual sessions. Long-term memory is durable: it spans sessions, days, and eventually months.

The hard problem is not storage — it's retrieval. Storing everything is easy. Getting the right memory injected into the right context window at the right time is the entire game.

hermes-agent: The defining 3-layer architecture

Hermes-agent is the most sophisticated memory system in the set. It implements all three tiers with a background daemon, full-text search, skill lifecycle management, and cron-driven suggestion proposals. Every other codebase either omits a tier or implements it in a simpler form.

MemoryManager: prefetch → inject → sync

The MemoryManager in agent/memory_manager.py:252 orchestrates a three-phase pipeline on every turn:

# agent/memory_manager.py:252 — MemoryManager orchestration
class MemoryManager:
  def __call__(self, agent: Agent, ctx: AgentContext, turns: list[Turn]):
    # Phase 1: Prefetch — pull relevant memories before the LLM call
    relevant = self._prefetch_memories(ctx, turns)
    # Phase 2: Inject — insert memories into the system prompt
    ctx = self._inject_memories(ctx, relevant)
    # Phase 3: Sync — background thread for new memory extraction
    self._start_sync_thread(agent, turns)

The prefetch phase is the critical one. Before each LLM call, the MemoryManager queries SQLite with FTS5 full-text search to find memories relevant to the current conversation. It uses keyword extraction from recent turns and matches against stored memory entries.

The inject phase inserts retrieved memories into the system prompt in a reference-only block — the agent can see them but cannot treat them as active instructions. This distinction is crucial: user-managed files like MEMORY.md must inform, not command.

The sync phase spawns a background daemon thread (via background_review.py) that replays the conversation and asks the LLM: "should any skill or memory be saved from this exchange?" This happens asynchronously, never blocking the main agent loop.

Storage: SQLite with FTS5 + CJK trigram

The storage layer in hermes_state.py uses SQLite with WAL mode for concurrent reads during writes. The schema includes FTS5 full-text search with both standard tokenization and CJK trigram support (for Chinese, Japanese, Korean text). Triggers keep the FTS index synchronized with the main table on insert/update/delete.

# hermes_state.py — FTS5 with CJK trigram support
CREATE VIRTUAL TABLE memories_fts USING fts5(
  title, content, tags,
  tokenize='trigram'  -- CJK + standard text search
);
CREATE TRIGGER memories_ai AFTER INSERT ON memories BEGIN
  INSERT INTO memories_fts(rowid, title, content, tags)
  VALUES (new.rowid, new.title, new.content, new.tags);
END;

The human-readable layer lives at ~/.hermes/memories/ as MEMORY.md and USER.md files. These are the ground truth — the SQLite database is a derived index, not the source of truth. Users can edit these files directly, and the agent respects them on next load.

Background review and skill curation

Two background systems maintain memory health over time:

Background Review (agent/background_review.py): A daemon thread that replays the conversation after each turn and asks the LLM a single question: "should any skill or memory be saved from this conversation?" This is the extraction mechanism — the LLM decides what's worth keeping, not a heuristic.

Skill Curator (agent/curator.py): Manages the lifecycle of saved skills. Skills auto-transition through states: active (used recently) → stale (unused for 30 days) → archive (unused for 90 days). This prevents memory bloat and keeps the active set relevant.

Cron Suggestions (cron/suggestions.py): An experience-driven system that analyzes past interactions and proposes new cron jobs. If the agent notices a pattern — "you always ask for a standup summary on Monday mornings" — it can suggest automating it.

Claude Code: File-based memory hierarchy

Claude Code takes a file-oriented approach. Instead of a database, memory is structured as markdown files at well-known paths, loaded in a strict precedence order.

4-tier CLAUDE.md loading

The src/utils/claudemd.ts loader resolves CLAUDE.md files from four levels, merged in order of increasing specificity:

  1. Managed: /etc/claude/claudemd/ — system-wide, admin-controlled
  2. User: ~/.claude/CLAUDE.md — personal, cross-project
  3. Project: CLAUDE.md in repo root, plus .claude/CLAUDE.md — team-shared
  4. Local: CLAUDE.local.md — per-developer, gitignored

Higher tiers set policy; lower tiers add specifics. A project CLAUDE.md can't override a managed-level directive, but it can add project-specific instructions.

MEMORY.md system

The src/memdir/ module manages MEMORY.md files with strict limits: max 200 lines or 25KB, whichever is hit first. It supports an @include directive for splitting large memory into multiple files, and automatically writes memory files when the agent accumulates enough context.

Session memory

src/services/SessionMemory/sessionMemory.ts maintains auto-generated markdown notes per session. These notes capture what happened during the session — files touched, decisions made, problems encountered — and persist them to disk for later review.

Memory consolidation

Consolidation in Claude Code is triggered by token thresholds: at minimum 10,000 tokens to initiate, and at least 5,000 tokens between updates. A forked subagent extracts key information from the conversation — this runs as a separate process so it never blocks the main agent. The consolidation agent is given a focused prompt: extract facts, decisions, and patterns, not raw conversation.

// src/services/SessionMemory/consolidation.ts — thresholds

Nanobot: File I/O with Consolidator

Nanobot's memory system is file-based but adds an incremental consolidation mechanism that Claude Code's simpler threshold approach lacks.

MemoryStore: four files + history

nanobot/agent/memory.py:40 defines the MemoryStore with five persistent artifacts:

# nanobot/agent/memory.py:40 — MemoryStore
class MemoryStore:
  def __init__(self, root_dir: Path):
    self.memory_md    = root_dir / "MEMORY.md"     # persistent memory
    self.user_md      = root_dir / "USER.md"       # user preferences
    self.soul_md      = root_dir / "SOUL.md"       # agent identity
    self.history_jsonl = root_dir / "history.jsonl" # consolidated log
    self.cursor        = 0                          # last consolidated position

The SOUL.md file is unique among all 7 codebases — it defines the agent's personality and behavioral constraints, separate from factual memory (MEMORY.md) and user preferences (USER.md). This tripartite split (identity, facts, preferences) is a clean separation that other codebases conflate into fewer files.

Consolidator: incremental LLM summarization

The Consolidator at memory.py:602 does what its name says: it incrementally summarizes old messages into history.jsonl. It uses a cursor-based approach — tracking which messages have already been consolidated via the cursor field, so only new messages are processed each run.

The consolidation boundary algorithm picks safe cut points at user-turn boundaries. It will never split a conversation in the middle of an agent response — the cut always lands between a user message and the next agent message. This preserves conversational coherence in the summary.

AutoCompact: idle-session archiving

nanobot/autocompact.py:17 provides AutoCompact, a TTL-based archiving mechanism. If a session goes idle beyond a configured duration, AutoCompact triggers consolidation and archives the full transcript. This prevents disk bloat for long-running but infrequently-used agents.

Git-based versioning

SOUL.md, USER.md, and MEMORY.md are versioned through a GitStore wrapper. Every change to these files is committed with a descriptive message, giving the user an audit trail of how the agent's memory evolved. The history.jsonl is NOT versioned — it's treated as a cache.

Codex: SQLite + 2-stage extraction

Codex uses a relational approach with four separate SQLite databases and a two-stage memory extraction pipeline.

Four databases

Codex maintains four SQLite databases, each with a distinct concern:

  • State DB: Runtime state, tool results, conversation metadata
  • Logs DB: Structured log entries for debugging and observability
  • Goals DB: Persistent goal tracking across sessions
  • Memories DB: memories_1.sqlite — the long-term memory store

2-stage extraction pipeline

codex-rs/state/src/runtime/memories.rs:28 implements the MemoryStore with a two-phase extraction pipeline:

// codex-rs/state/src/runtime/memories.rs:28
// Stage 1: Thread-level extraction — runs per conversation thread
// Phase 2: Global consolidation — merges across threads, 6-hour cooldown
pub struct MemoryStore {
  db: SqlitePool,
  last_global_consolidation: Instant,  // 6-hour cooldown
  // ...
}

Stage 1 extracts memories from a single conversation thread — what was decided, what was learned. This runs frequently, after substantial turns.

Phase 2 performs global consolidation — merging memories from multiple threads into a unified knowledge base. This runs on a 6-hour cooldown to avoid redundant LLM calls. The cooldown is per-process (not persisted), so it resets on restart.

The two-stage design recognizes a key insight: thread-local extraction is cheap and can run often, but cross-thread consolidation is expensive and should be batched.

Minimal memory: pi-mono, openclaw, and opencode

pi-mono: JSONL sessions only

pi-mono has the simplest memory story of any codebase: JSONL session files. That's it. No MEMORY.md, no consolidation, no skill learning, no background extraction. Each session is written to a .jsonl file for replay, but there is no mechanism for information to cross session boundaries. Why? pi-mono was designed as a research framework where memory is the consumer's responsibility — the framework provides the loop, hooks, and tools, and the app developer decides what to remember.

openclaw: SDK available, not default

openclaw ships a Memory Host SDK with SQLite storage and optional embeddings for semantic search. However, it is not used by default. The SDK is available for consumers who want to wire it in, but the framework itself runs without any long-term memory. This reflects openclaw's design philosophy: provide the capability, don't force the integration.

opencode: localStorage only

OpenCode's packages/app/src/utils/persist.ts persists UI state to localStorage — things like panel widths, sort order, and filter preferences. There is no agent memory system at all. The persistence is purely for the human-facing UI, not for the agent's knowledge. This is consistent with OpenCode's positioning as a lightweight coding assistant rather than a persistent agent.

Design heuristics

"Memory is only useful if it's retrievable at the right time — prefetch is everything." Storing memories is the easy part. The hard part is surfacing the right memory in the agent's context window exactly when it's needed. Hermes-agent's prefetch phase, Claude Code's tiered CLAUDE.md loading, and Codex's SQL queries all solve the same problem: retrieval timing.
"Consolidation should happen in the background, never blocking the main loop." Every codebase with consolidation (hermes-agent, claude-code, nanobot, codex) runs it asynchronously. Hermes-agent uses a daemon thread. Claude Code forks a subagent process. Codex gates on a 6-hour cooldown. The main agent loop must never wait on a consolidation run — the user's turn time is sacred.
"User-managed files (MEMORY.md) must be 'reference-only' in context — never treated as active instructions." Hermes-agent injects memories into a reference block in the system prompt. Claude Code separates CLAUDE.md (instructions) from MEMORY.md (knowledge). The distinction matters: the user writes instructions; the agent writes memories. Conflating them causes the agent to treat its own stale memories as commands.
"Staleness decays — memory without lifecycle management becomes noise." Hermes-agent's curator is the only system with explicit lifecycle management: active → stale (30d) → archive (90d). Without this, long-lived agents accumulate cruft — old facts, resolved TODOs, and irrelevant context that dilutes the retrieval signal.
"Safe cut points matter for coherence." Nanobot's consolidator only splits at user-turn boundaries. Codex separates thread-local from global consolidation. Both recognize that arbitrary truncation produces incoherent summaries. A cut point must respect conversational structure.

Why pi-mono has no memory

The absence of memory in pi-mono is a design decision, not an omission. pi-mono is a research framework — it provides the agent loop, hooks, and tool interface, and leaves memory to the consumer. This is consistent with the "primitives, not features" philosophy: give developers building blocks, not a prescribed memory architecture.

But there's a real cost: every pi-mono consumer must build their own memory system. If they don't, their agent has no persistent knowledge. This is fine for one-shot coding tasks; it's crippling for long-running agents.
