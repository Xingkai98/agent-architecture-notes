Lesson 0009: The Pregel Execution Model

Graphs, not loops — how LangGraph replaces imperative control flow with declarative state machines.

Course 2: Agent Infrastructure · June 2026 · Permalink

Every agent in Course 1 is built around a loop. while (should_continue), for iteration in range(max), state machines — all imperative. The codebase author writes the iteration logic, handles the state transitions, and manages termination. This lesson introduces a fundamentally different approach: define a graph, compile it, and let a runtime execute it.

Pregel was introduced by Google in 2010 as a system for large-scale graph processing. The key insight — Bulk Synchronous Parallel execution — maps naturally to agent workflows: each node is a computation, edges define dependencies, and a barrier synchronizes state between "supersteps."

LangGraph's Pregel runtime is the execution engine beneath deepagents, deer-flow, and LangChain's create_agent. When you call agent.astream() on any of these, you're not running a while-loop — you're stepping through a graph one superstep at a time. Understanding this runtime is understanding the "operating system" that half the modern agent ecosystem runs on.

From loops to graphs

In Course 1, we saw four loop architectures:

A: Nested 2-Loop
pi-mono, openclaw
B: Flat While-True
claude-code, hermes-agent
C: State Machine
nanobot, codex
D: Delegated Graph
deepagents, deer-flow

Architecture D delegates to LangGraph. But what does LangGraph actually do? It compiles a StateGraph definition into a CompiledStateGraph (which extends Pregel), then executes it using the Bulk Synchronous Parallel model. Each iteration is a superstep with three phases:

1. PLAN
prepare_next_tasks() determines which nodes fire based on channel versions
→
2. EXECUTE
PregelRunner runs all selected nodes concurrently via asyncio.gather
→
3. UPDATE
apply_writes() commits results to channels, bumps versions

Plan: prepare_next_tasks() at pregel/_algo.py:392 consumes pending Send tasks (PUSH tasks) and checks trigger_to_nodes for channel-driven triggers (PULL tasks). For each triggered node, _proc_input() at line 1348 builds the node's input from current channel state. The function returns a dict of PregelExecutableTask objects — one per triggered node.

Execute: PregelRunner at pregel/_runner.py:135 runs all tasks concurrently. For sync loops, it uses executor.submit(). For async, asyncio.gather(). Nodes that don't share channel dependencies execute in parallel — this is the "bulk synchronous" part of BSP.

Update: apply_writes() at pregel/_algo.py:232 sorts tasks by path, updates versions_seen, applies writes to channels, bumps channel versions, and notifies unchanged channels. Each channel has a reducer that controls how concurrent writes merge — a LastValue channel keeps the last write, a BinaryOperatorAggregate channel sums values, a Topic channel appends to a list.

Channels: the state primitive

In Course 1's agent loops, state is a mutable object — a struct, a dict, a message list. In LangGraph, state is distributed across channels. Each channel has a type that defines its merge behavior:

ChannelReducerUse case
LastValueLatest write winsSystem prompt, model name, flags
TopicAppend to listMessage history (with add_messages reducer)
EphemeralValueRead once, then clearedJump-to-node signals, transient flags
BinaryOperatorAggregateApply binary op (sum, max, etc.)Token counters, iteration counts
DeltaChannelSnapshot every N stepsLarge message histories (O(N²)→O(N) checkpoint)

DeltaChannel is the most architecturally interesting. A naive checkpoint would serialize the entire message list at every step — O(N²) storage. DeltaChannel snapshots only every 50 steps, and reconstructs intermediate states by walking back to the nearest snapshot and replaying writes since. This is what deepagents uses at graph.py:67 for its DeepAgentState.messages.

Channels are a form of state machine with merge semantics. They solve the same problem as CRDTs — how to merge concurrent writes from parallel node execution — but for a much more constrained domain.

Conditional edges: the graph's decision points

A StateGraph is built in two steps. First, you add nodes with add_node("name", callable) at graph/state.py:662. Then you wire them together: add_edge("a", "b") for unconditional transitions, add_conditional_edges("a", branch_fn, path_map) at line 969 for branching.

The conditional edge is where LangGraph's "graph as program" nature becomes visible. The branch_fn receives the current state and returns a string — the next node to execute. LangGraph wraps this in a BranchSpec (graph/_branch.py) that, after the source node executes, evaluates the branch function and writes to branch:to:{target} channels. The next superstep's Plan phase reads these channels to determine which nodes fire.

This is how the standard agent graph works:

# Every LangGraph-based agent (deepagents, deer-flow, LangChain create_agent)
# compiles to this topology:

StateGraph(AgentState)
  .add_node("model", call_llm)         # THINK
  .add_node("tools", ToolNode(tools))  # ACT
  .add_edge(START, "model")
  .add_conditional_edges(
      "model",
      lambda state: "tools" if state["messages"][-1].tool_calls else END,
      {"tools": "tools", END: END}
  )
  .add_edge("tools", "model")          # OBSERVE → back to THINK
  .compile()

The compile() method at graph/state.py:1164 validates the graph, resolves channels, wraps each node as a PregelNode with channel readers/writers, and returns a CompiledStateGraph (which extends Pregel). From this point on, invoke() / stream() / astream() all go through the Pregel loop.

Why graphs beat loops

The Pregel model gives you three things that imperative loops don't:

  1. Parallelism for free. Nodes without channel dependencies execute in the same superstep. In the agent graph above, "model" and "tools" are always sequential (they share the messages channel), but a graph with independent branches — e.g., a research agent that searches the web AND reads local files simultaneously — gets parallelism automatically.
  2. Deterministic replay. Because every channel write is versioned and ordered, replaying a graph from a checkpoint produces exactly the same sequence of states. This enables time-travel debugging, branching from any checkpoint, and audit trails — all covered in Lesson 0010.
  3. Durability at the framework level. The loop author doesn't write save/restore logic — the Pregel runtime checkpoints after every superstep. If the process crashes, the next invocation resumes from the last checkpoint. This is what deer-flow's worker.py relies on for its pre-run checkpoint rollback.

The tradeoff: opacity

There is a real cost to the graph model. When a Course 1 agent (claude-code, pi-mono) misbehaves, you set a breakpoint in the while-loop and step through. When a LangGraph agent misbehaves, the stack trace passes through PregelLoop._tick() → prepare_next_tasks() → _apply_writes() — three levels of framework indirection before you reach your node code. Debugging requires understanding the channel version system and the trigger graph.

Design heuristic: If your agent has complex branching (parallel research, sub-agent fan-out), the graph model pays for itself in free parallelism and durability. If your agent is a simple linear think→act→observe loop, an imperative while-loop is simpler to debug. The tipping point is when you have more than one conditional branch or more than three tools — at that point, the channel/reducer model starts preventing bugs that would be manual state management in a flat loop.
