Lesson 0011: From AgentExecutor to Middleware Protocol

How LangChain's agent system evolved from a callback-driven while-loop to a composable middleware stack on LangGraph.

Course 2: Agent Infrastructure · June 2026 · Permalink

LangChain's repository contains two generations of agent system side by side. The classic AgentExecutor (now deprecated) is a callback-driven while-loop — the same pattern we saw in Course 1's pi-mono and hermes-agent. The V1 create_agent (current) is a middleware stack compiled to a LangGraph StateGraph — the same pattern that deepagents and deer-flow extend. This lesson traces the evolution and extracts the architectural lesson: callbacks couple, middleware composes.

The fact that both systems coexist in the same repo is unusual and valuable — it's a fossil record of the architectural transition from imperative agent loops to declarative middleware stacks. Few production codebases preserve their deprecated architecture alongside the replacement.

AgentExecutor: the while-loop era

The AgentExecutor class at langchain_classic/agents/agent.py:1012 is a textbook while-loop agent. Its _call() method at line 1570 iterates until AgentFinish or max_iterations (default 15):

# AgentExecutor._call() — langchain_classic/agents/agent.py:1570
while self._should_continue(iterations, time_elapsed):
    next_step = self._take_next_step(...)        # agent.plan() → execute tool → observe
    if isinstance(next_step, AgentFinish):
        return self._return(next_step, ...)       # terminal state
    intermediate_steps.extend(next_step)          # (action, observation) pairs
    iterations += 1
    time_elapsed = time.time() - start_time

Each iteration calls agent.plan() to get the next AgentAction, then _perform_agent_action() at line 1380 to execute the tool via tool.run(), generating an observation. The loop continues until either the LLM returns a final answer or the iteration/time budget is exhausted.

Instrumentation is callback-based: CallbackManager fires on_agent_action, on_agent_finish, on_tool_start, on_tool_end at fixed points. There is no way to intercept or modify the execution flow from a callback — callbacks are observers, not participants. If you want to add retry logic, context summarization, or permission checks, you must modify the executor itself or wrap it externally.

create_agent: the graph era

The V1 create_agent() at langchain_v1/langchain/agents/factory.py:787 takes a completely different approach. Instead of a while-loop, it builds a StateGraph with two nodes — "model" and "tools" — connected by conditional edges:

# create_agent() — factory.py:787
StateGraph(AgentState)
  .add_node("model", model_node)           # THINK: call LLM
  .add_node("tools", ToolNode(tools))      # ACT: execute tools
  .add_edge(START, "model")
  .add_conditional_edges("model", model_to_tools, {"tools": "tools", END: END})
  .add_edge("tools", "model")              # OBSERVE: feed results back
  .compile(checkpointer=checkpointer)

The recursion limit is 9999 (vs. AgentExecutor's 15) — the graph model assumes LangGraph's checkpointing and streaming will handle the unbounded execution safely. The real architectural innovation is not the graph itself, but how middleware hooks become graph nodes.

The 6-hook AgentMiddleware protocol

AgentMiddleware at middleware/types.py:383 defines six hook points:

HookWhen it firesPurpose
before_agentOnce before the loop startsSetup, load state, initialize resources
before_modelBefore each LLM callModify system prompt, inject context
after_modelAfter each LLM callInspect response, detect tool calls
after_agentOnce after the loop endsCleanup, save state, emit events
wrap_model_callWraps the LLM invocationRetry, fallback, rewrite request
wrap_tool_callWraps tool executionRetry, monitor, modify input/output

The critical architectural difference from callbacks: wrap hooks can intercept. A wrap_model_call handler receives the ModelRequest and a handler callback that invokes the actual model. The middleware can modify the request before passing it to the handler, catch exceptions and retry, or even skip the handler entirely. This is the difference between observing execution and participating in it.

In the compiled graph, the before_agent, before_model, after_model, and after_agent hooks become separate graph nodes. The wrap_model_call and wrap_tool_call hooks are composed as nested onion layers via _chain_model_call_handlers() at factory.py:234 — first-in-list is outermost, meaning it executes first on the way in and last on the way out.

Each middleware is self-contained — a single file, typically 100-300 lines, implementing one or two hooks. This modularity is what deepagents and deer-flow inherit and extend. DeepAgents adds FilesystemMiddleware and SubAgentMiddleware. DeerFlow adds 8 more, including SandboxMiddleware, LoopDetectionMiddleware, and DeferredToolFilterMiddleware — each as a composable middleware, not a modification to the agent loop.

Why this evolution matters

The AgentExecutor → create_agent transition is not unique to LangChain. It's the same pattern we observed across Course 1: ad-hoc logic embedded in the loop → explicit, composable interceptors.

Course 1: Implicit
claude-code's 12 terminal reasons embedded in queryLoop()
→
Course 1: Explicit
deepagents' 11 middleware, each a named class

LangChain's repo captures the exact same transition within a single codebase. The classic AgentExecutor represents the "implicit" pole — all behavior (retry, summarization, tool execution) is either in the loop or in separate callbacks that can't modify execution. The V1 create_agent represents the "explicit" pole — every behavior is a middleware with a defined interface and composable ordering.

Architectural lesson: Callbacks observe. Middleware participates. If you're building an agent framework today, start with a middleware stack — you can always add callbacks as observers, but retrofitting middleware into a callback-based system requires rewriting the execution loop.
