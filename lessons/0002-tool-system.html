Lesson 0002: Tool System

Define, register, dispatch, normalize — the four stages every agent tool passes through.

June 2026 · 9 primary source codebases · Permalink

Every tool in every agent goes through exactly four stages: it must be defined (so the LLM knows its schema), registered (so the harness can find it), dispatched (so it actually runs), and normalized (so results feed back into the conversation). The differences between codebases are all in how they implement these four stages.

In Lesson 0001, we saw that every agent is built around a loop. This lesson zooms into the inner engine of that loop: how tools work. We'll trace all four stages across all nine codebases, and identify the four fundamental design philosophies that divide them.

The four stages

Every tool interaction follows this pipeline:

┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ DEFINE │ → │ REGISTER │ → │ DISPATCH │ → │ NORMALIZE │ │ │ │ │ │ │ │ │ │ Schema, │ │ Registry, │ │ Execute, │ │ Result → │ │ params, │ │ discovery, │ │ sandbox, │ │ LLM-ready │ │ description │ │ availability │ │ permissions │ │ message │ └──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘

Four design philosophies

Across the nine codebases, four distinct philosophies emerge:

Maximally minimal Fully orchestrated
  1. Minimalist: pi-mono, openclaw, nanobot. Tools are simple objects with an execute function. No central registry or discovery machinery — tools live in arrays or dicts. The framework doesn't even ship with built-in tools; tools are provided by the consumer.
  2. Type-safe / DI: opencode (Effect-TS). Tools are defined with runtime-validated schemas, registered via dependency injection into scoped containers, and dispatched with automatic cleanup. The type system guarantees correctness.
  3. Fully orchestrated: claude-code, codex, hermes-agent. Tools go through a multi-stage pipeline: permission checks, sandbox selection, execution, retry, result normalization, and (for claude-code) UI rendering. Each stage is a separate concern with pluggable implementations.
  4. Middleware-driven: deepagents, deer-flow. Tools are not registered directly by the developer — they are contributed by middleware. Each middleware in the stack (Filesystem, SubAgent, Skills, MCP) exposes its own tools. The tools available to the LLM at any moment are the union of all middleware-contributed tools plus user-provided tools. This inverts the ownership model: tool lifecycle is managed by interceptors, not by a central registry. (Contrast with Lesson 0001's Architecture D — the middleware stack IS the architecture, and tool ownership follows the same pattern.)

Stage 1: Define — how tools describe themselves

Schema-first (pi-mono, openclaw)

Tools implement the AgentTool<TParameters> interface, which extends a generic Tool with a label, execute(), and executionMode. The schema is declared as a TypeScript generic — the type is the schema. No runtime schema validation beyond what the tool's own execute() does internally.

// pi-mono: packages/agent/src/types.ts:365
export interface AgentTool<TParameters = any, TDetails = any>
  extends Omit<Tool<TParameters>, 'execute'> {
  label: string;
  execute(toolCall, context, options?): Promise<AgentToolResult<TDetails>>;
  executionMode: 'sequential' | 'parallel';
}

// openclaw: packages/agent-core/src/types.ts:438 — identical structure
export interface AgentTool<TParameters = any, TDetails = any>
  extends Omit<Tool<TParameters>, 'execute'> {
  label: string;
  execute(toolCall, context, options?): Promise<AgentToolResult<TDetails>>;
  executionMode: 'sequential' | 'parallel';
}

Zod-typed objects (claude-code)

Claude Code tools are objects implementing the Tool<Input, Output> interface, with Zod schemas for inputSchema and outputSchema. The buildTool() factory fills in safe defaults for 7 lifecycle methods.

// claude-code: src/Tool.ts:783
buildTool({
  name: ToolNames.BASH,
  description: 'Execute a bash command',
  inputSchema: BashInputSchema,  // Zod schema
  async *call(input, context) { /* ... */ },
  // buildTool provides defaults for:
  // isConcurrencySafe, isReadOnly, isEnabled,
  // checkPermissions, validateInput,
  // renderToolUseMessage, renderToolResultMessage
})

JSON Schema dicts (hermes-agent, nanobot)

Both Python codebases use JSON Schema dicts. Hermes-agent's ToolEntry stores a schema dict; nanobot's Tool ABC has a parameters property returning a JSON Schema dict. The difference: nanobot uses a Python ABC with @tool_parameters decorator for declarative schema attachment, while hermes-agent passes the schema dict directly to registry.register().

# hermes-agent: tools/registry.py:77
class ToolEntry:
  name: str
  toolset: str
  schema: dict            # OpenAI-format JSON Schema
  handler: Callable       # sync or async
  check_fn: Callable | None  # availability gate

# nanobot: nanobot/agent/tools/base.py:124
class Tool(ABC):
  name: str
  description: str
  parameters: dict        # JSON Schema
  async def execute(self, **kwargs): ...

  @tool_parameters(StringParam("query", "..."))
  async def search(self, query: str): ...

Rust trait-based (codex)

Codex uses Rust traits: CoreToolRuntime for tool handlers, ToolOutput for results, ToolSpec for schema definition. The type system enforces correctness at compile time.

Stage 2: Register — how tools are discovered

The registration strategy correlates strongly with language and framework choice:

No registry (pi-mono, openclaw): Tools are passed as an AgentTool[] array on AgentContext. The consumer owns tool lifecycle entirely. This is the simplest possible approach — and the most flexible, since there's no framework to fight.

pi-mono's approach: "primitives, not features." The framework provides zero built-in tools. The four tools (read, write, edit, bash) are defined at the application layer, not in the framework.

Flat array with conditional inclusion (claude-code): getAllBaseTools() in src/tools.ts:193 returns a flat array of 30+ tools. Tools are conditionally included based on feature flags (REPLTool, WebBrowserTool) and permission context. MCP tools are merged in via assembleToolPool() (line 345) with deduplication.

Singleton registry with AST discovery (hermes-agent): The ToolRegistry singleton in tools/registry.py:151 stores all tools. Discovery in discover_builtin_tools() (line 57) scans tools/*.py with the ast module — it parses Python source to find registry.register(...) calls before importing the module. This avoids side effects from importing tools that require unavailable dependencies (Docker, Playwright, etc.).

Package scanning (nanobot): ToolLoader in nanobot/agent/tools/loader.py:21 uses pkgutil.iter_modules to discover Tool subclasses, plus entry_points(group="nanobot.tools") for plugin discovery. The _plugin_discoverable class attribute prevents abstract base classes from being auto-discovered.

Per-turn build (codex): build_tool_router() in spec_plan.rs constructs the ToolRouter fresh each turn. It incorporates MCP tools, dynamic tools (discovered at runtime), extension tools, and standard tools. The router maps ToolName → ToolRegistry entry.

DI + Scope (opencode): ToolRegistry.Service in registry.ts uses Effect's Layer/Scope for dependency injection. Tools are registered into a scoped container and automatically cleaned up when the scope exits. The materialize() method snapshots definitions for the LLM.

Middleware-driven tools (deepagents, deer-flow)

A new paradigm emerges in these two codebases: tools are not declared by the developer and registered into a central list. Instead, each middleware contributes its own tools, and the final tool list is the union of all middleware contributions plus user-provided tools.

deepagents: create_deep_agent() in graph.py:236 assembles a middleware stack where each middleware exposes tools: TodoListMiddleware → write_todos, FilesystemMiddleware → ls/read_file/write_file/edit_file/glob/grep/execute, SubAgentMiddleware → task. The _ToolExclusionMiddleware in middleware/_tool_exclusion.py can filter tools based on HarnessProfile.excluded_tools. Tool descriptions can be overridden without modifying the tool object via _apply_tool_description_overrides() in _tools.py:30-65.

deer-flow: get_available_tools() in tools/tools.py:44 assembles tools in a strict priority order: config-defined tools → built-in (present_files, ask_clarification) → subagent task tool → vision tool → MCP tools (lazy-init, cache with mtime) → ACP tools → skill management. The DeferredToolFilterMiddleware hides MCP tool schemas from the LLM until the tool_search promotion triggers them — a unique "lazy tool loading" pattern that keeps context windows lean when MCP servers expose hundreds of tools.

This middleware-driven approach inverts control: instead of a tool registry being configured then executed, tool availability itself becomes a function of the middleware stack's current state. A skill activation can add tools. A permission change can remove them. The tool system is dynamic, not static.

Stage 3: Dispatch — how tools are executed

Execution timing

As covered in Lesson 0001, two camps:

  • Batch (after stream): pi-mono, openclaw, nanobot, hermes-agent — wait for LLM stream to complete, collect tool calls, execute together
  • Streaming (during stream): claude-code — StreamingToolExecutor executes tools as the LLM emits them
  • Hybrid: codex — FuturesOrdered drains in-flight tool futures during streaming; opencode — configurable

Sandboxing

Only one codebase has a serious sandboxing story:

codex: The ToolOrchestrator (orchestrator.rs) provides a multi-layered sandbox: Landlock (Linux), Windows sandbox, macOS sandbox, elevated/restricted token. On sandbox denial, the orchestrator can retry with escalated sandboxing — if a restricted sandbox blocks a file write, the tool can be retried with the elevated sandbox. Network access approval is a first-class concept with immediate and deferred modes.

deer-flow: The second codebase with a serious sandboxing story. The SandboxProvider pattern in sandbox/sandbox_provider.py abstracts execution across Local, Docker, and AioSandbox backends. Tools see virtual paths (/mnt/user-data/workspace) that are translated to host paths. Sandbox tools (bash, ls, read_file, write_file, glob, grep, str_replace) go through virtual path translation (validate_local_tool_path()) and path traversal detection. Write size capped at 80KB. This is sandboxing as a first-class architectural dimension, not an afterthought — and it's integrated into the tool system rather than wrapped around it.

Other codebases delegate sandboxing to external mechanisms: pi-mono/openclaw use beforeToolCall hooks, hermes-agent uses environment-based terminals, claude-code uses permission checks on tool access (not execution sandboxing), deepagents delegates to the backend protocol (SandboxBackendProtocol).

Stage 4: Normalize — how results feed back

After execution, tool results must become messages the LLM can consume. This is where the most divergence occurs:

Size truncation (all): Every codebase has a maximum result size. Claude Code's applyToolResultBudget() persists large results to disk and replaces them with truncated previews. Hermes-agent's max_result_size_chars truncates per-tool. Nanobot exempts read_file from offloading to prevent persist-to-disk → read-from-disk loops.

Error normalization (all): All codebases catch tool exceptions and convert them to error messages that the LLM can understand. Hermes-agent and nanobot return {"error": "..."} JSON strings. pi-mono/openclaw create ToolResultMessage objects with isError: true. Codex uses FunctionCallError variants.

Progress streaming (pi-mono, openclaw): The AgentToolResult can include a progress callback for partial streaming, allowing the UI to show incremental tool output before completion. OpenClaw added this on top of the pi-mono base.

Per-tool rendering (claude-code): Uniquely, Claude Code tools have 7+ rendering methods (renderToolUseMessage, renderToolResultMessage, renderToolUseProgressMessage, etc.) — the tool result is not just a message for the LLM, but a UI artifact for the TUI.

The fundamental tradeoff: complexity budget

The tool system is where the codebases diverge most dramatically. pi-mono has zero lines of tool registry code. Codex has an entire ToolOrchestrator with approval, sandbox selection, and retry escalation. Why?

More tool infrastructure = more safety guarantees, but more code to maintain. Codex's sandbox retry logic means a tool that fails due to permission denial in the restricted sandbox can automatically retry with elevated permissions — the user doesn't need to re-run. But it also means a multi-stage pipeline with 10+ files just for tool orchestration.

Less tool infrastructure = more flexibility, but more burden on the consumer. pi-mono gives you AgentTool[] and says "go." You can implement any tool system you want. But you also have to — the framework won't help you with permission checks, sandboxing, result normalization, or error recovery.

Design heuristic: If you control the execution environment and trust your tools, start minimalist — an array of objects with an execute function. If you need sandbox isolation, permission gating, and observability, the orchestrated approach (codex, claude-code) pays off — but expect the tool system to become the largest subsystem in your codebase.
