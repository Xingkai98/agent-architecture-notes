Lesson 0007: UI / Transport Layers

TUI frameworks, client/server splits, and multi-channel adapters — how agents talk to the world.

June 2026 · 9 primary source codebases · Permalink

Every agent needs a surface. For some that surface is a terminal (Claude Code, Pi-mono, Codex, Opencode, DeepAgents). For others it is a messaging app (OpenClaw with 24 channels, Nanobot with 18, DeerFlow with 7). For still others it is a desktop app, web UI, or API server. The choice of transport layer shapes everything above it: how tools render results, how streaming works, how errors surface to the user.

In Lesson 0002, we saw how tools are defined, registered, dispatched, and normalized. This lesson moves one layer up: how those tool results reach the user, and how the user's input reaches the agent. We'll survey the full interface paradigm spectrum, compare six TUI implementations in detail, examine client/server architectures, and map the multi-channel landscape.

The interface paradigm spectrum

Agents expose themselves to users along a spectrum of increasing complexity and reach:

REPL/TUI API
  1. REPL / TUI: The agent is a terminal application. The user types, the agent responds, the loop is visible. Claude Code, Pi-mono, Codex, and Opencode all live here. This is the dominant paradigm for developer-facing agents.
  2. CLI one-shot: The agent runs a single turn and exits. Input comes from args or stdin, output goes to stdout. Useful for scripting and CI. Codex and Pi-mono both support this mode.
  3. Daemon: The agent runs as a long-lived process, often headless. Codex's app-server and OpenClaw's gateway both operate this way — the agent process is separate from the interface.
  4. Multi-channel: One agent, many surfaces. OpenClaw (24 channels) and Nanobot (18 channels) are the exemplars here. A single agent loop can receive input from Telegram, Discord, Slack, Feishu, and Matrix simultaneously.
  5. API: The agent exposes an HTTP endpoint (often OpenAI-compatible /v1/chat/completions). Nanobot's api/server.py is the clearest example — a thin aiohttp wrapper that lets any HTTP client drive the agent.

TUI technologies compared

Six codebases implement terminal UIs, but they make radically different technology choices. The framework selection is not just an implementation detail — it determines how streaming output renders, how scrolling works, how key bindings are defined, and how much maintenance burden the project carries.

Claude Code: Custom React + Ink fork

Claude Code's TUI is built on a custom fork of Ink (React for terminals). The fork lives at src/ink/ with a 251KB ink.tsx that includes a custom fiber reconciler and terminal layout engine. The two largest components — Messages.tsx (147KB) and VirtualMessageList.tsx (148KB) — show how much UI logic a terminal agent accumulates. Virtual scrolling is mandatory: agent conversations produce hundreds of messages, and naively rendering them all would overwhelm the terminal.

The Ink fork is a multi-year commitment. Claude Code can't simply upgrade to upstream Ink — the fork diverged to support custom layout, virtual scrolling, and streaming tool rendering. This is the cost of the "TUI as application" philosophy.

Codex: ratatui + crossterm

Codex uses Rust-native terminal libraries: ratatui for widget-based rendering and crossterm for terminal control. The main TUI lives at codex-rs/tui/src/tui.rs and app.rs (58KB). Ratatui provides a retained-mode widget tree with layout constraints — a different paradigm from React's declarative diffing. Crossterm handles raw mode, cursor control, and input events.

Pi-mono: Custom TUI from scratch

Pi-mono's TUI at packages/tui/src/tui.ts (53KB) has zero framework dependencies. It implements components from scratch: box, editor, image, input, loader, and markdown. No Ink, no ratatui, no prompt_toolkit — just raw ANSI escapes and terminal IO. This gives maximal control and minimal bloat, at the cost of reimplementing every widget behavior.

Hermes-agent: prompt_toolkit

Hermes-agent uses Python's prompt_toolkit in cli.py, composing Application, Layout, TextArea, and KeyBindings. Prompt_toolkit provides a mature widget system with syntax highlighting, auto-completion, and key binding DSL out of the box — no need to build from scratch.

Opencode: OpenTUI + Solid.js

Opencode's TUI at packages/tui/src/app.tsx (36KB) is built on OpenTUI with Solid.js for reactive rendering. This is a novel combination: terminal UI with fine-grained reactive JSX. Solid's signal-based reactivity means only changed parts of the terminal output re-render, without a virtual DOM.

Nanobot: prompt_toolkit + rich

Nanobot combines prompt_toolkit for the interactive CLI with rich for formatted output in cli/commands.py (67KB). Rich provides tables, panels, syntax-highlighted code blocks, and markdown rendering — all styled for terminal output.

Client/Server splits

Four codebases separate the agent process from the UI process — a design choice that enables remote operation, headless deployment, and multiple frontends:

Opencode: Web Worker server + RPC + TUI client

Opencode runs the agent core in a Web Worker, communicating with the TUI client via RPC. The TUI can connect to a local worker (in-process) or a remote server over HTTP. This means the same agent can drive a terminal UI, a web UI, or an IDE plugin — all through the same RPC interface.

Codex: App Server + TUI client

Codex splits into app-server/ (the agent process) and app-server-client/ (the TUI). The two can run in-process (zero network overhead) or the TUI can connect to a remote app-server over HTTP. This is the same pattern as a language server: the server does the work, the client handles display.

OpenClaw: Gateway protocol

OpenClaw defines a formal gateway protocol at packages/gateway-protocol/ with a corresponding gateway-client/ package. The protocol specifies message formats, authentication, and transport — any client that speaks the protocol can drive the agent. This is a step beyond Codex's ad-hoc client/server split: the protocol is versioned, documented, and reusable.

DeepAgents: CLI + LangGraph Server

DeepAgents takes a modular approach to transport, offering three distinct surfaces built on LangGraph's infrastructure. The primary developer interface is dcode, a CLI tool installable via curl -LsSf https://langch.in/dcode | bash. Its standout feature is the interactive /model command, which lets users switch LLM providers mid-conversation without restarting the session. Configuration lives at ~/.deepagents/config.toml.

For programmatic use, the Python SDK's create_deep_agent() returns a CompiledStateGraph — a LangGraph graph that can be invoked directly in-process or served via LangGraph Server. The server exposes a standard LangGraph-compatible REST API for remote invocation. Streaming inherits all 8 LangGraph stream modes: values, updates, messages, custom, checkpoints, tasks, debug, and tools. This is the simplest transport model in the set — DeepAgents doesn't build its own protocol or message bus. It inherits LangGraph's battle-tested streaming and multi-mode infrastructure, trading architectural complexity for a mature, well-documented foundation.

Multi-channel (messaging bots)

Three codebases go far beyond terminal UIs — they are messaging platforms first, agents second:

OpenClaw: 24 channels

OpenClaw supports 24 channels: Discord, Slack, Telegram, Feishu, Matrix, WhatsApp, Signal, IRC, iMessage, LINE, Zalo, and more. Channels are registered declaratively in each extension's package.json. The channel plugin contract standardizes adapters for: configuration, authentication, outbound message sending, connection status, device pairing, and security policies. Build-time channel discovery generates dist/channel-catalog.json — a manifest of all available channels without runtime scanning.

The key architectural insight: each channel is an adapter, not an agent. There is one agent loop. Messages from all channels route through a unified message bus. The agent does not know or care which channel a message came from.

Nanobot: 18 channels

Nanobot supports 18 channels with substantial per-channel implementations: Telegram (64KB), Feishu (78KB), Discord (33KB), Slack (30KB), Signal (55KB), Matrix (43KB), Weixin (63KB), and more. All channels follow the BaseChannel ABC with three core methods: start(), stop(), and send(). A message bus handles pub/sub routing between channels and the agent core.

# Nanobot channel contract (conceptual): every channel implements these three
class BaseChannel(ABC):
    async def start(self) -> None: ...
    async def stop(self) -> None: ...
    async def send(self, target, message) -> None: ...

# Each channel is an adapter — the agent loop never imports Telegram or Discord.
# Messages arrive on the bus as a normalized type; the bus routes them to the agent.
# Responses go back through the bus; the channel adapter formats them for its platform.

DeerFlow: 7 IM platforms + Web UI + embedded client

DeerFlow has the most sophisticated multi-channel transport architecture of all nine codebases. The transport layer is a first-class architectural dimension — not a bolt-on — with four independent surfaces:

  • Message Gateway: Native support for Telegram, Slack, Feishu/Lark, Discord, DingTalk, WeChat, and WeCom — no public IP or webhook tunnel required.
  • Web UI: A Next.js frontend served through nginx at port 2026, layered over a FastAPI Gateway API at port 8001 with REST endpoints for threads, runs, models, MCP, skills, memory, uploads, artifacts, suggestions, feedback, and channels.
  • Embedded client: DeerFlowClient provides direct in-process access without standing up any HTTP services — useful for scripting, testing, and programmatic orchestration.
  • LangGraph-compatible endpoint: Nginx routes /api/langgraph/* to the Gateway runtime at port 8001, enabling standard LangGraph clients to connect.

The IM Channels system (app/channels/) is architecturally the most interesting part. It is built on a MessageBus — an async pub/sub layer that decouples the agent from any specific platform. Per-platform implementations handle platform-specific behaviors: Feishu/Lark adapters patch cards in-place (updating a single message as the agent progresses rather than posting new ones), while the Telegram adapter edits a "Working on it..." placeholder message. User-owned channel bindings are managed through channel_connections, allowing each user to connect their own accounts without sharing a single bot identity.

SSE streaming uses a StreamBridge (worker.py line 317): bridge.publish(run_id, sse_event, ...) delivers events to all connected clients. The full nginx routing table is: /api/langgraph/* → Gateway runtime (8001), /api/* → Gateway API (8001), / → Frontend (3000).

Hermes-agent: Desktop + CLI

Hermes-agent takes a different approach: a desktop Electron app at apps/desktop/ built with React + Vite, plus a cli.py for terminal use. The desktop app provides a graphical chat interface while the CLI serves scripting and power users. This is the only codebase in the set that ships a GUI.

Single-interface agents

Four codebases commit to a single interface — no channels, no desktop app, no API server:

  • Claude Code: Terminal-only (React+Ink). Supports structured IO for IDE integration (VS Code, JetBrains) but the agent itself is a terminal application.
  • Pi-mono: Terminal-only (custom TUI). No API, no channels, no desktop. The terminal is the entire product surface.
  • Codex: Terminal-only (ratatui) with a remote control daemon. The daemon enables headless operation, but user interaction is through the TUI.
  • Opencode: Terminal-only (OpenTUI). The Web Worker architecture allows a web UI, but the primary interface is the terminal.

Interesting design choices

Five implementation details worth studying:

Claude Code's Ink fork: The TUI IS the application. Messages.tsx at 147KB is not a "view layer" — it is the core user experience. Tool results, streaming text, permissions prompts, and mode indicators all render through React components. The fork exists because no off-the-shelf terminal UI library handles streaming agent conversations at this scale.

Pi-mono's from-scratch TUI: At 53KB with zero dependencies, Pi-mono's TUI is a statement. No framework churn, no upgrade treadmill, no breaking changes from upstream. The cost is that every feature — syntax highlighting, scrollback, resize handling — must be built and maintained. For a project that values minimalism, this tradeoff makes sense.

OpenClaw's channel catalog: dist/channel-catalog.json is generated at build time from extension manifests. This means the agent knows its available channels before starting — no runtime scanning, no dynamic imports, no race conditions. It is a small touch that eliminates an entire class of startup bugs.

Nanobot's OpenAI-compatible API server: api/server.py is a thin aiohttp wrapper that exposes /v1/chat/completions. This means any tool that can call the OpenAI API — langchain, openai-python, curl — can drive Nanobot without knowing it is not talking to OpenAI. The agent becomes a drop-in replacement.

DeerFlow's four-surface architecture: No other codebase exposes the agent through four independent transport surfaces simultaneously — IM gateway (7 platforms), web UI (Next.js), REST API (FastAPI), and embedded client (in-process). The IM channel system is not a plugin directory but a structured MessageBus with per-platform adapters that handle platform-specific rendering (in-place card patching for Feishu, placeholder editing for Telegram). This design treats the channel layer as an architectural peer to the agent core, not as an add-on.

Design heuristics

Heuristic 1: Terminal agents need virtual scrolling — messages grow unbounded. Claude Code's VirtualMessageList (148KB) exists because naive rendering of 500+ messages freezes the terminal. Every TUI agent hits this eventually, regardless of framework.
Heuristic 2: Multi-channel agents need a unified message bus — each channel is an adapter, not an agent. OpenClaw and Nanobot both converge on this pattern independently. The agent loop processes one message type; channel adapters translate to and from it. Don't write an agent per channel.
Heuristic 3: The TUI framework choice is a multi-year commitment — custom forks give control but cost maintenance. Claude Code's Ink fork can't track upstream. Pi-mono's from-scratch TUI needs every widget built by hand. Ratatui upgrades require Rust refactors. Choose the surface of your maintenance burden carefully.

Recommended next reading

  • If you want to understand virtual scrolling for terminals: trace Claude Code: src/components/VirtualMessageList.tsx — the 148KB file is the definitive implementation of rendering only visible messages in a terminal.
  • If you want the cleanest multi-channel pattern: study OpenClaw: packages/gateway-protocol/ — the protocol defines message formats, auth, and transport independent of any specific channel.
  • If you want the most practical TUI pattern: read Pi-mono: packages/tui/src/tui.ts — 53KB of from-scratch terminal rendering shows exactly what a TUI needs and nothing more.
  • If you want channel implementation depth: compare Nanobot's Feishu channel (78KB) with its Telegram channel (64KB) — two production-quality adapters with different authentication and event models.
  • If you want the most ambitious transport architecture: study DeerFlow: app/channels/ — the MessageBus + per-platform adapter pattern with in-place card editing, plus the nginx-routed four-surface design (IM gateway, web UI, REST API, embedded client).
