Lesson 0008: Error Recovery

From crash to continuation — how 7 agent codebases classify, recover from, and survive failure.

June 2026 · 7 primary source codebases · Permalink

Every agent loop will encounter errors. The difference between a brittle agent and a resilient one is not whether errors occur — it's whether the agent can see the error, classify it, choose a recovery strategy, and continue without losing state. This lesson traces the full recovery pipeline across all 7 codebases.

In Lesson 0002, we saw tools as the engine of the agent loop. In Lesson 0003, we saw how context is the loop's memory. This lesson covers what happens when the loop breaks — and how each codebase picks it back up.

Error categories

Every agent error falls into one of five categories. Each codebase handles them differently, but the categories are universal:

Two poles of recovery philosophy

Across the seven codebases, two poles emerge:

Caller-delegated (pi-mono, openclaw) Full self-recovery (claude-code, hermes-agent)
  1. Caller-delegated: pi-mono, openclaw. The agent loop stops on error and signals the reason upward. Recovery — retry, restart, model switching — is the caller's responsibility. The framework provides hooks (config.transformContext, config.prepareNextTurn) but no built-in retry logic.
  2. Full self-recovery: claude-code, hermes-agent, nanobot. The agent loop includes retry with exponential backoff, model fallback chains, context compaction on overflow, and checkpoint restore. The loop itself is a state machine with error transitions.
  3. Hybrid: codex, opencode. Some errors are handled internally (SQLite recovery, step limits), others are surfaced to the caller.

Claude Code: The most complex recovery system

Claude Code's queryLoop() in src/query.ts:241 is a recursive generator that runs the agent turn-by-turn against mutable State. The loop has 12 terminal reasons (conditions that end the loop) and 7 recovery transitions (conditions that retry the current turn with modified state).

The 7 recovery transitions

Before reaching a terminal reason, the loop may take one of 7 recovery paths. Each path modifies State and yields back to the top of the loop for another attempt:

// claude-code: src/query.ts — recovery transitions in queryLoop()

// 1. collapse_drain_retry — after drain-and-collapse, retry the same turn
// 2. reactive_compact_retry — compaction triggered by overflow, then retry
// 3. max_output_tokens_escalate — first 'max_tokens' hit → escalate budget
// 4. max_output_tokens_recovery — subsequent hits → inject recovery message
// 5. stop_hook_blocking — stop hook said "don't stop", continue the loop
// 6. token_budget_continuation — exact budget hit, append continuation request
// 7. next_turn — normal transition: collect results, start next turn

Max-tokens recovery: a two-phase strategy

When the model hits its output token limit, Claude Code uses a two-phase escalation:

// Phase 1: Escalate budget from 8K → 64K (ESCALATED_MAX_TOKENS)
// If it hits again at 64K, Phase 2 kicks in:
//
// Phase 2: Inject a recovery message into the conversation
// "You hit the output token limit. Summarize your progress and continue."
//
// This repeats up to 3 times. After the third recovery message,
// if the limit is still hit, the loop terminates.

Model fallback

API errors are caught in src/services/api/withRetry.js. When the primary model fails with a retryable error, a FallbackTriggeredError is thrown. The caller catches this and switches to fallbackModel — typically a smaller, faster model that can at least complete the current turn before the user is asked to intervene.

Design observation: Claude Code packs the most recovery logic of any codebase into a single function. The result is a loop where every exit and retry path is visible in one place — no hidden recovery scattered across files. This is intentional: "flatten the loop to see all terminal conditions in one function."

Hermes-agent: Classify-first recovery with 22 failure reasons

Hermes-agent takes a fundamentally different approach: instead of a flat loop with inline recovery, it uses a dedicated error classification pipeline that runs before any recovery decision is made.

FailoverReason: 22 reasons

agent/error_classifier.py:24 defines the FailoverReason enum with 22 members:

The classification pipeline in classify_api_error() runs through a priority-ordered chain: provider-specific patterns (checking the provider's error wrapper) → HTTP status code → error code string → message match → TLS transient detection → transport heuristics. Each stage can short-circuit the pipeline with a definitive classification.

TurnRetryState: 14 one-shot recovery guards

agent/turn_retry_state.py:33 defines the TurnRetryState dataclass — a collection of 14 boolean flags, each acting as a one-shot guard to prevent infinite retry loops:

  • 5 auth refresh guards: refreshed_oauth, refreshed_api_key, rotated_org, switched_project, cleared_token_cache
  • 6 format/payload guards: truncated_context, dropped_images, compressed_payload, switched_to_text_only, split_long_message, reduced_tool_count
  • 2 transport/rate-limit guards: switched_endpoint, rate_limit_backoff_applied
  • 2 restart signals: needs_fresh_turn, needs_model_fallback

Each guard is set to True the first time its recovery action is taken. If the same error recurs, the classifier sees the guard already set and escalates to a harder failure (typically model fallback or terminal error).

Exponential backoff with jitter

agent/retry_utils.py:19 implements jittered exponential backoff:

# hermes-agent: agent/retry_utils.py:19
base_delay = 5         # seconds
max_delay = 120        # 2 minutes cap
jitter_ratio = 0.5     # ±50% randomization

# Formula: min(base * 2^attempt * (1 ± jitter), max_delay)

Fallback chain

Hermes-agent maintains an ordered fallback chain on the agent instance: agent._fallback_chain — a list of (provider, model) pairs. When the classifier returns a hard failure reason (model_not_found, billing, content_filter), or when all one-shot guards for a soft failure are exhausted, the agent activates the next entry in the chain and resets retry_count to 0. This means the full recovery budget resets on each fallback step — a fresh model gets a fresh chance.

Codex: SQLite corruption recovery

Codex stands apart with a recovery mechanism unique to its architecture: the agent's runtime state is stored in a SQLite database, and SQLite corruption is a real failure mode in long-running agent sessions.

Backup-and-fresh-start

codex-rs/state/src/runtime/recovery.rs:71 implements a clean recovery strategy:

// codex-rs: codex-rs/state/src/runtime/recovery.rs:71
fn backup_runtime_db_for_fresh_start(corrupt_path: &Path) -> Result {
    // 1. Create backup dir: db-backups/sqlite-{timestamp}-{seq}/
    // 2. Move corrupt DB file into the backup directory
    // 3. Return the backup path (so the caller can inspect it)
    // 4. The caller creates a fresh DB at the original path
    // 5. Previous state is lost, but the corrupt DB is preserved for forensics
}

The detection function, is_sqlite_corruption_error(), walks the error chain (Rust's Error::source()) looking for SQLite error codes 11 (SQLITE_CORRUPT — database disk image is malformed) and 26 (SQLITE_NOTADB — file is not a database). This is a one-shot recovery with no retry — if the fresh DB also corrupts, the agent fails hard.

Nanobot: Provider retry, length recovery, and checkpoint restore

Nanobot implements recovery at three distinct layers:

Provider retry

nanobot/providers/base.py:817 — _run_with_retry() has two modes:

  • Standard mode: 3 attempts with delays of 1s, 2s, 4s. For normal API calls.
  • Persistent mode: Unlimited retries, capped at 60s delay. Used for long-running agent sessions where the user expects the agent to keep trying until the provider recovers.

Length recovery

nanobot/agent/runner.py:528 — when the LLM response has finish_reason == "length" (the model hit its output token limit mid-response):

# nanobot/agent/runner.py:528
# 1. build_length_recovery_message() constructs:
#    "Your previous response was cut off due to length limits.
#     Please continue from where you left off."
# 2. The message is injected as a user turn
# 3. Max 3 attempts — then the loop terminates

Empty response retry

Empty LLM responses (content-less with no tool calls) are retried up to 2 times. The retry message explicitly asks the model to produce output.

Stream stall recovery

If the streaming connection stalls, the on_stream_recover callback starts a new stream segment from the last received token. The model resumes generating where it left off, rather than the entire turn being discarded.

Checkpoint restore

nanobot/agent/loop.py:1725 — after a crash or restart, the in-flight turn checkpoint is loaded. Any tool calls that were in progress at the time of the crash get synthetic error results:

# nanobot/agent/loop.py:1725
# Recovered tool calls → synthetic error:
# {"error": "Tool call was interrupted by agent restart. The result is unavailable."}
# The LLM sees these as error tool results and can decide whether to retry or continue.

Opencode: Effect-TS typed errors and step limits

Opencode uses Effect-TS to encode errors in the type system:

// opencode: packages/core/src/session/runner/index.ts:20
type RunError =
  | LLMError               // Provider-level failures
  | Model.Error            // Model-specific errors
  | MessageDecodeError     // Failed to parse LLM output
  | ContextSnapshotDecodeError  // Corrupted context snapshot
  | StepLimitExceededError // Exceeded MAX_STEPS

Step limit

MAX_STEPS = 25. When the turn count reaches 25, a StepLimitExceededError is raised. This is a hard terminal — no retry, no escalation. The caller must decide what to do.

Context overflow recovery

compactAfterOverflow() performs a single-shot compaction and retry. If the context still overflows after one compaction, the error propagates upward. No multi-stage escalation like Claude Code's drain-and-collapse pipeline.

Tool cleanup on error

failUnsettledTools() is called when a provider error occurs mid-turn. Any tool calls that were queued but not yet settled are marked as failed with an error message, ensuring the conversation state remains consistent — the model sees that its tool calls were not executed.

Pi-mono, OpenClaw: Minimal error handling

These two TypeScript codebases take the simplest possible approach:

  • No built-in retry. The loop stops when stopReason === "error" or "aborted".
  • Tool execution errors are caught in executeToolCalls() and returned as error tool results — the LLM sees a ToolResultMessage with isError: true.
  • Recovery is delegated to the caller via two config hooks:
    • config.transformContext — allows the caller to modify the context before each turn (e.g., compact, truncate)
    • config.prepareNextTurn — called before each turn, the caller can inject recovery messages or switch models

The philosophy: the framework should not make recovery decisions. The caller knows their use case and their error tolerance. Give them hooks, not policy.

Design heuristics

Four principles emerge from studying all 7 codebases:

1. Recovery paths should be VISIBLE. Flatten the loop to see all terminal conditions in one function. Claude Code's queryLoop() is the exemplar — 12 terminal reasons and 7 recovery transitions, all in one file, all with line numbers. When recovery logic is scattered across files (retry in one, compaction in another, fallback in a third), it becomes impossible to reason about the complete state machine.
2. One-shot guards prevent infinite recovery loops. Every recovery path needs a flag that prevents it from activating twice for the same error. Hermes-agent's 14 TurnRetryState boolean flags are the clearest implementation of this principle. Claude Code's max-tokens recovery (escalate once, then inject up to 3 times) is another. Without these guards, a persistent error will loop forever.
3. Tool errors should be LLM-readable. Format tool failures as messages the model can reason about. Every codebase does this, but nanobot's checkpoint restore takes it furthest: orphaned tool calls get synthetic error messages that explain why the result is unavailable. The model can then decide whether to retry the tool call or work around the missing data.
4. The more complex the recovery, the flatter the loop should be. Claude Code has the most recovery paths (7) and the flattest loop (all in queryLoop()). This is not coincidence. When recovery logic is deeply nested or spread across modules, adding a new recovery path risks breaking existing ones. A flat, visible loop makes every transition explicit and every interaction observable.
