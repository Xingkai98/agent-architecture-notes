Lesson 0001: The Core Agent Loop

Think, act, observe, repeat — the fundamental architecture of every AI agent.

June 2026 · 8 primary sources · Permalink

Open any agent codebase. The first question to ask is: where is the loop? Every agent — whether it runs in a terminal, a Slack channel, or a Docker container — is built around a single architectural center of gravity: the execution loop that cycles between calling the LLM, running tools, and feeding results back.

CoALA (Sumers et al., 2023) was the first paper to formalize this as a cognitive architecture: a generalized decision loop with modular memory stores and a structured action space. Read it at arXiv:2309.02427.

This lesson maps that loop across seven production agent codebases — four coding agents (claude-code, codex, opencode, pi-mono) and three general agents (hermes-agent, openclaw, nanobot). We'll see two fundamental architectures emerge, and understand the tradeoffs that push teams toward one or the other.

The universal skeleton

Strip away the CLI, the chat channels, the plugins and sandboxes. Every agent loop collapses to this:

┌─────────────────────────────────────────────────────┐ │ while (should_continue): │ │ │ │ // THINK phase │ │ response = await llm.stream({ │ │ system: system_prompt, │ │ messages: conversation_history, │ │ tools: available_tools, │ │ }) │ │ │ │ // ACT phase │ │ if response.has_tool_calls(): │ │ for each tool_call: │ │ result = execute(tool_call) │ │ conversation_history.append(result) │ │ │ │ // OBSERVE (results already in history) │ │ // Loop continues — next LLM call sees results │ │ │ │ if response.is_text_only(): │ │ return response.text // terminal state │ │ │ └─────────────────────────────────────────────────────┘

This is the ReAct pattern (Yao et al., 2022) — interleaved reasoning and action. The LLM reasons about what to do, the agent executes it, and the result is fed back for the next round of reasoning. Nothing more.

But look closer at seven real implementations, and this simple loop fractures into two fundamentally different architectures.

The two architectures

Architecture A: Nested Two-Loop

Used by: pi-mono, openclaw (and likely inherited in part by opencode).

An outer loop handles follow-up messages (user interrupts, queued commands that arrive after the agent would otherwise stop). An inner loop runs the actual think-act-observe turns:

// pi-mono: packages/agent/src/agent-loop.ts, runLoop() // openclaw: packages/agent-core/src/agent-loop.ts, runLoop() // // (These two are structurally identical — openclaw's is a fork.) while (true) { // OUTER: follow-up loop while (hasMoreToolCalls // INNER: tool turn loop || pendingMessages > 0) { message = await streamAssistant() // THINK toolCalls = executeTools(message) // ACT hasMoreToolCalls = !allTerminate // OBSERVE → decide } followUps = getFollowUpMessages() if (followUps.length === 0) break // terminal state pendingMessages = followUps }

Why this design? Follow-up messages are a common pattern: the user interrupts mid-run ("no, use a different approach"), or a sub-agent completes and its results need to be injected. By separating follow-up handling into its own outer loop, the inner loop can focus purely on the think-act-observe cycle without checking for external interruptions on every iteration.

pi-mono and openclaw share this architecture because openclaw's agent-core package is a direct descendant of pi's agent package. The code lineages matter.

Architecture B: Flat While-True

Used by: claude-code, hermes-agent. A single while (true) loop with all concerns — tool execution, follow-up injection, recovery — handled as branches within one iteration.

// claude-code: src/query.ts, queryLoop() // hermes-agent: agent/conversation_loop.py, run_conversation() while (true) { // Phase A: Context prep (compaction, budget checks) prepareContext(state) // Phase B: THINK — stream LLM response for await (event of streamModel(state)) { if (event.type === 'tool_use') startToolExecution(event) // may run during streaming } // Phase C: No tools? → check stop hooks, token budget, return if (!needsFollowUp) { runStopHooks() if (ok) return { reason: 'completed' } } // Phase D: ACT — execute remaining tools if (needsFollowUp) { await executeAllTools() // Phase E: OBSERVE — collect results, refresh, continue state = nextState(state, toolResults) continue // back to top of while(true) } }

Why this design? When recovery logic is complex — prompt-too-long, max_output_tokens, reactive compaction, model fallback, abort handling — a flat loop with explicit phases is easier to reason about than nested loops where different recovery paths exit at different levels. Claude Code's loop has twelve distinct terminal reasons (query.ts lines 1050–1720); each is a return { reason: '...' } at the right point in the flat sequence.

A third variant: Layered State Machine

Used by: nanobot. Instead of nested loops or a flat while-true, nanobot uses an explicit state machine with 7 states (RESTORE → COMPACT → COMMAND → BUILD → RUN → SAVE → RESPOND → DONE). Within the RUN state, a separate for iteration in range(max_iterations) loop handles the think-act cycle.

Used by: codex (Rust). Four distinct loop layers — submission_loop, task loop, turn loop, sampling loop — each with its own termination condition. The turn loop calls run_sampling_request() which internally streams the LLM response, drains in-flight tool futures via FuturesOrdered, and determines needs_follow_up.

Used by: opencode. Uses Effect-TS fibers rather than raw async/await. The run() function is an Effect.fn generator with a for (let step = 0; step < MAX_STEPS; step++) loop inside a while (openActivity) outer guard.

The fundamental tradeoff: nesting depth

Why does codex have 4 loop levels while claude-code has 1? This is not an accident — it reflects different priorities:

More nesting = more separation of concerns. Codex's four levels each handle one thing: the submission_loop manages the session lifecycle, the task loop manages task lifecycle (abort/restart), the turn loop manages the think-act cycle, and the sampling loop manages retries. Each level can be tested and reasoned about independently.

Less nesting = easier to see all terminal paths. Claude Code's single flat loop makes all twelve exit conditions visible in one function. When recovery logic is as complex as it is in Claude Code (prompt-too-long → collapse drain → reactive compact → escalate max tokens → inject recovery message → finally return error), flattening the loop makes it harder to miss a recovery path.

Design heuristic: If your recovery logic is simple (retry on error, abort on signal), nested loops work well. If your recovery logic involves cascading fallbacks across multiple dimensions (context size, token budget, model availability, stop hooks), flatten the loop so all paths are visible in one place.

Another key dimension: when do tools execute?

Two camps:

  1. Batch execution (pi-mono, openclaw, nanobot, hermes-agent): Wait for the LLM stream to complete, collect all tool calls, then execute them. Simple to implement, but tools that could have started running 5 seconds ago sit idle while the model finishes generating.
  2. Streaming execution (claude-code, codex): Execute tools as soon as the model emits them in the stream. Tools run concurrently with the rest of the model's response. Reduces latency but requires careful state management — what happens when a tool result arrives before the model has finished streaming?

Claude Code's StreamingToolExecutor starts tools immediately when content_block_start fires for a tool_use block, but tracks in-progress results so they can be yielded to the loop at the right point in the turn cycle.

This connects to a deeper design question about the tool execution model itself — covered in Lesson 0002: Tool System.
