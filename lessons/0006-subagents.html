Lesson 0006: Sub-agents

Spawn, isolate, delegate, collect — the four acts of every multi-agent system.

June 2026 · 9 primary source codebases · Permalink

Every sub-agent system answers the same four questions: how is the child spawned, what can it see, how much can it spend, and how do results flow back to the parent. The differences between codebases are all in where they draw the isolation boundary and how they enforce it.

In Lesson 0002, we saw that tools are the engine of the agent loop. This lesson zooms out one level: when one agent becomes many. We'll trace sub-agent spawning, context isolation, resource budgeting, and result collection across all nine codebases, then extract the design heuristics that hold regardless of framework.

The spawn spectrum

Sub-agents exist on a continuum of isolation. The further right you go, the stronger the guarantee — and the higher the overhead:

In-process Container
In-process ─── Thread ─── Child process ─── Container │ │ │ │ │ │ │ └─ Full OS isolation │ │ └─ Process boundary, IPC │ └─ Shared memory, GIL concerns └─ Same heap, context object cloning

No codebase in our set uses containers for ad-hoc sub-agents (though several use them for the host sandbox). The real battle is between in-process cloning and child-process spawning — and the choice shapes everything downstream.

Claude Code — in-process with context isolation

Claude Code's sub-agent system is the most sophisticated in-process design in our set. It spawns children within the same Node.js process but gives each an isolated ToolUseContext so they cannot accidentally mutate the parent's state.

Spawn: runAgent()

At src/tools/AgentTool/runAgent.ts:248, runAgent() creates an isolated sub-agent by calling createSubagentContext(). The child gets its own conversation history, its own tool set, and its own abort signal — but it runs in the same event loop.

// claude-code: src/tools/AgentTool/runAgent.ts:248
// runAgent() creates an isolated ToolUseContext for the child,
// spawns the sub-agent conversation, and collects the result.

Context isolation: createSubagentContext()

At src/utils/forkedAgent.ts:345, createSubagentContext() clones three critical pieces of state:

  • readFileState — cloned so the child tracks its own file reads independently; the parent's readFileState is not polluted by child reads.
  • AbortController — linked as a child controller, so cancelling the parent cancels all descendants.
  • setAppState — replaced with a no-op for async sub-agents, since they cannot safely update shared UI state.

Three sub-agent modes

Claude Code supports three distinct sub-agent patterns:

  1. Synchronous (blocking): The parent pauses until the child completes. Used when the parent needs the child's answer to continue. This is the default AgentTool mode.
  2. Asynchronous (fire-and-forget): The parent dispatches the child and continues immediately. The child's result is delivered later via a callback or message bus. Used for parallel exploration tasks.
  3. Fork: At src/tools/AgentTool/forkSubagent.ts:1, the fork subagent inherits the parent's conversation context and uses a byte-identical API prefix. This allows the LLM provider to share prompt cache entries between parent and child — a significant cost optimization for deeply nested delegation.

Swarm mode

At src/utils/swarm/backends/InProcessBackend.ts, TeammateExecutor implements multi-agent orchestration using file-based mailboxes. Each agent writes to and reads from a designated file, creating an asynchronous message-passing system entirely within one process. The file system acts as a durable, inspectable message bus.

Hermes-agent — ThreadPoolExecutor with iteration budgets

Hermes-agent takes the thread-level approach: each sub-agent runs in a thread from a shared pool, with strict resource limits enforced by the framework.

Iteration budgets

At agent/iteration_budget.py, every sub-agent receives its own IterationBudget (default 50 iterations). This is the key insight: budgets must be separate. A sub-agent that loops or explores excessively must not exhaust the parent's token allowance. When the budget is spent, the child is interrupted and its partial result is returned to the parent.

Child execution

_run_single_child() creates a child AIAgent instance with:

  • A restricted toolset — only the tools the parent explicitly granted
  • An isolated task_id — the child's conversation is independently tracked
  • A focused system prompt — rewritten for the sub-task, not the parent's overall objective

Blocked tools for children

By default, children cannot use: delegate_task (prevent runaway recursion), clarify (children should resolve ambiguity independently), memory (memory is owned by the parent), send_message (only the parent communicates with the user), and execute_code (sandboxing concern).

Concurrency and nested delegation

max_concurrent_children defaults to 3. Nested delegation is controlled by role="orchestrator" combined with delegation.max_spawn_depth — an orchestrator child can spawn its own workers, but the depth is capped. The subagent registry is thread-safe and supports pause(), resume(), interrupt(), and list() operations for full lifecycle management.

Pi-mono — child process model

Pi-mono takes the strongest isolation approach short of containers: each sub-agent is a separate OS process.

Subagent extension

At packages/coding-agent/examples/extensions/subagent/index.ts, the complete subagent extension is implemented as a standalone module. It demonstrates that sub-agents in pi-mono are extensions, not built into the framework core — the framework provides the spawn primitive (child_process), and the extension defines the delegation protocol.

Three execution modes

  1. Single: Spawn one child, wait for completion, return the result.
  2. Parallel: Spawn up to 8 children with a concurrency limit of 4. Results are collected as they arrive. Failed children don't block siblings.
  3. Chain: Spawn children sequentially, each receiving the previous child's output via a {previous} placeholder in its prompt. Enables pipeline-style decomposition: "Find relevant files" → "Read those files" → "Summarize findings."

Process-level isolation

runSingleAgent() uses child_process.spawn() with --mode json and --no-session flags. The child writes JSON lines to stdout; the parent reads them as a stream. Per-task output is capped at 50KB to prevent any single child from flooding the parent with data.

Codex — multi-agent v2

Codex's multi-agent system is the most feature-complete in our set, with dedicated tools for every lifecycle operation.

Agent lifecycle tools

At codex-rs/core/src/tools/handlers/multi_agents_v2.rs, five tools form the complete lifecycle:

ToolPurpose
spawn_agentCreate a new sub-agent with a task description
wait_for_agentBlock until a specific child completes, collect its result
send_messageSend a mid-execution message to a running child
interrupt_agentStop a running child and collect its partial result
list_agentsEnumerate all active children with their statuses

Spawn options

At codex-rs/core/src/agent/control/spawn.rs:86, spawn_agent() accepts SpawnAgentOptions:

  • fork_context — whether the child inherits the parent's conversation
  • fork_context_last_n_turns — how many recent turns to include (sliding window)
  • max_threads — thread budget for the child's own parallel operations

Encrypted inter-agent communication

Codex uniquely encrypts all inter-agent messages via InterAgentCommunication::new_encrypted(). This is not a performance optimization — it is a security property: even within the same process, agents cannot inspect each other's messages without explicit permission. This matters when agents handle credentials or sensitive data.

Nanobot — asyncio.Tasks

Nanobot uses Python's native asyncio for lightweight, cooperative sub-agents within a single event loop.

Spawn and manage

At nanobot/agent/subagent.py:74, SubagentManager.spawn() creates an asyncio.Task — the lightest possible isolation unit. The task runs concurrently with the parent; the manager tracks its lifecycle.

# nanobot: nanobot/agent/subagent.py:74
class SubagentManager:
    async def spawn(self, goal: str, tools: list[str]) -> str:
        task = asyncio.create_task(self._run_child(goal, tools))
        self._tasks.append(task)
        return task_id

Result delivery

_announce_result() publishes the child's result via the message bus as a subagent_result injected event. The parent's context manager picks this up and injects it into the parent's conversation as if it were a tool result — the parent never sees the sub-agent as anything other than an async function call.

Status tracking

The SubagentStatus dataclass tracks phase transitions:

# nanobot: subagent status phases
initializing → awaiting_tools → tools_completed → final_response → done
                                                              ↘ error

Concurrency is controlled by the max_concurrent_subagents config value.

Opencode — child sessions

Opencode models sub-agents as child sessions, reusing the same session machinery that powers the top-level agent.

TaskTool

At packages/opencode/src/tool/task.ts, the TaskTool creates a child session via sessions.create(). The child session is a fully independent agent conversation with its own context window, tool set, and abort controller — but it shares the same process and the same LLM connection.

Permission derivation

At packages/opencode/src/agent/subagent-permissions.ts, the child's permission ruleset is derived from the parent's. This is not a simple copy: the derivation function strips permissions that are meaningless for sub-agents (e.g., allow_agent_spawn if nesting is disabled) and adds restrictions (e.g., read-only file access for worker children).

Foreground vs. background

Opencode supports two execution modes directly analogous to Claude Code's sync/async split:

  • Foreground (blocking): The parent waits for the child session to complete before continuing.
  • Background (fire-and-forget): The child session runs independently; the parent can poll or be notified on completion.

OpenClaw — ACP-based delegation

OpenClaw uses the Agent Communication Protocol (ACP) as its delegation transport, making sub-agents addressable across process and network boundaries.

Spawn parameters

At src/agents/acp-spawn.ts, SpawnAcpParams defines the spawn contract:

// openclaw: src/agents/acp-spawn.ts
interface SpawnAcpParams {
  task: string;              // the sub-task description
  agentId: string;           // which agent type to spawn
  resumeSessionId?: string;  // resume an existing session
  model?: string;            // model override for the child
  thinking?: boolean;        // enable extended thinking
  sandbox?: SandboxSpec;     // sandbox configuration
  streamTo?: string;         // channel to stream results to
}

Multi-agent routing

OpenClaw's ACP layer provides 24 channels for inter-agent communication. Each agent subscribes to channels it cares about; the ACP router delivers messages by channel subscription, not by agent identity. This decoupling means agents can be added, removed, or replaced without reconfiguring the routing table.

DeepAgents — sync sub-agents with state sharing

DeepAgents takes a pragmatic approach: fast, synchronous, and state-efficient. Sub-agents run in-process and share the parent's LangGraph state — only messages, todos, and structured_response are excluded.

SubAgentMiddleware

At middleware/subagents.py:734, SubAgentMiddleware exposes a single task tool to the parent agent. The tool is built by _build_task_tool() at line 528 and accepts a sub-agent type name, a task description, and optional configuration overrides.

Sub-agent specification and compilation

A sub-agent is defined by a SubAgent TypedDict (line 36–161) and compiled into a CompiledSubAgent (line 164–241). create_sub_agent() (line 459) calls create_agent() with the sub-agent's specified model, tools, and middleware. The system prompt is injected declaratively through the spec's prompt field — there is no runtime rewriting.

State sharing: parent state by default

The task() function (line 668) calls subagent.invoke(subagent_state, subagent_config) synchronously (line 693). The sub-agent runs on the parent's full state except for _EXCLUDED_STATE_KEYS (line 249): messages, todos, and structured_response. This means tool results, file reads, and other state channels are shared automatically — efficient, but tightly coupled. A sub-agent can pollute the parent's file-read cache or accumulate unbound tool outputs in shared state channels.

GENERAL_PURPOSE_SUBAGENT and system prompt injection

A GENERAL_PURPOSE_SUBAGENT is available by default (line 427). wrap_model_call() (line 848) injects the available sub-agent types and their descriptions into the system prompt, so the parent model always knows which sub-agents it can delegate to.

Concurrency model

DeepAgents sync sub-agents are inherently sequential — the parent blocks on subagent.invoke() until the child returns. There is no explicit concurrency limit because there is no concurrent execution path for sync sub-agents. The implicit limit is 1 at a time.

DeerFlow — async sub-agents with dual thread pools

DeerFlow takes the opposite approach from DeepAgents: full isolation, async-by-default, with a dedicated lifecycle manager backed by separate scheduler and execution thread pools.

SubAgentExecutor: dual thread pools

At subagents/executor.py:276, SubAgentExecutor manages the full sub-agent lifecycle. It maintains two thread pools: a _scheduler_pool (3 workers) for dispatching and status management, and an _execution_pool (3 workers) for running sub-agent agent loops. This separation prevents scheduling overhead from competing with execution throughput.

Execution paths

  • execute() (line 714): Synchronous entry point. Detects if there is a running event loop and uses an isolated persistent loop if needed, preventing event loop conflicts in async environments like FastAPI.
  • execute_async() (line 758): Fire-and-forget in the background via the scheduler thread pool. Returns immediately; the parent polls for completion.
  • _aexecute() (line 483): Core async implementation. Builds initial state, creates the agent, streams via agent.astream(), and collects AIMessages.

SubAgentConfig and tool control

At subagents/config.py:11, SubAgentConfig defines: name, description, system_prompt, tool allow/deny lists, skills, model (inherit or explicit), max_turns (default 50), and timeout_seconds (default 900 = 15 minutes). Built-in sub-agents (at builtins/__init__.py:12) include general-purpose (all tools except task) and bash (command specialist).

Isolation and polling

Sub-agent graphs are compiled with checkpointer=False — they are one-shot, never resumed. No state is shared with the parent. The parent polls at 5-second intervals for completion status. This is safer than sharing state (no risk of cache pollution or side effects) but adds latency: worst case, the parent waits 5 seconds after the sub-agent finishes before discovering the result.

Concurrency enforcement: SubAgentLimitMiddleware

SubAgentLimitMiddleware caps concurrent sub-agents at MAX_CONCURRENT_SUBAGENTS (default 3) by truncating excess task tool calls in after_model. This is an aggressive enforcement strategy: if the model requests 5 sub-agents and the limit is 3, the last 2 tool calls are silently dropped. The model never knows those sub-agents failed to spawn — it must infer from missing results.

DeepAgents vs DeerFlow: state-sharing vs isolation

"DeepAgents trusts the sub-agent; DeerFlow distrusts it."
This is the defining tradeoff between these two designs. DeepAgents sync sub-agents share the parent's state graph (except messages, todos, structured_response). Tool results, file reads, and other channels flow automatically between parent and child — no serialization, no polling. But a misbehaving sub-agent can corrupt shared state. DeerFlow compiles every sub-agent graph with checkpointer=False and shares nothing. The sub-agent is a black box: input goes in, result comes out after polling. Safer, but you pay for every byte of context that must be explicitly passed in the task description.

The concurrency models reflect the same philosophical split. DeepAgents sync sub-agents are sequential — the parent blocks, so concurrency is not even a question. DeerFlow embraces parallelism but defends against it with SubAgentLimitMiddleware at 3 concurrent sub-agents, enforced by truncating tool calls in after_model. DeepAgents async sub-agents sidestep the local concurrency question entirely by delegating to remote LangGraph deployments — the only codebase in our set to do so.

Design heuristics

"Isolation is the defining dimension — what the sub-agent can SEE determines what it can DO."
Every design decision in a sub-agent system flows from the isolation boundary. In-process cloning (claude-code, codex) is fast and cache-friendly but requires disciplined context separation. Process spawning (pi-mono) is safe by construction but pays IPC overhead on every delegation. Thread pools (hermes-agent) sit in the middle — safe-enough with Python's GIL, but the isolation is conventional, not enforced by the OS.
"Budgets must be separate: a sub-agent shouldn't exhaust the parent's token allowance."
Hermes-agent's per-child IterationBudget is the cleanest expression of this principle. Nanobot achieves it implicitly through asyncio cooperative scheduling. Claude Code relies on the parent to manage budgets (via AbortController linkage). The key failure mode is the same across all codebases: an unconstrained sub-agent burns tokens exploring dead ends, and the parent has no recourse except timeout-killing the child.
"The fork pattern (claude-code, codex) trades isolation for cache efficiency."
When a child inherits the parent's conversation prefix (fork_context), the LLM provider can reuse cached key-value pairs from the parent's inference. This saves substantial API cost — but it means the child can "see" the parent's conversation. If the parent was discussing sensitive data, the child inherits that data in its context. Forking is an optimization, not a default — use it when cache savings outweigh isolation concerns.
