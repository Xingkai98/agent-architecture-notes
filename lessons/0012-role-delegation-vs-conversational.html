Lesson 0012: Role Delegation vs Conversational Orchestration

Two fundamentally different answers to the same question: how should multiple agents coordinate?

Course 3: Multi-Agent Orchestration · June 2026 · Permalink

Courses 1 and 2 were about single agents — one LLM, one loop, one tool set. This lesson asks: what happens when you have multiple agents? Two frameworks — CrewAI and AutoGen (with its successor MAF) — give radically different answers. CrewAI says: "a manager assigns work." AutoGen says: "agents talk to each other." Both work. The difference is in who controls the workflow.

Multi-agent systems solve three problems that single agents struggle with: context window saturation (each agent has its own window), tool overload (each agent has a focused tool set), and single-point-of-failure (one agent's error doesn't kill the whole system).

Paradigm A: Role Delegation (CrewAI)

CrewAI models multi-agent coordination as a management hierarchy. A Crew owns a set of Task objects and a roster of Agent objects. The Crew assigns tasks to agents based on their roles:

Crew
Owns tasks + agents
→
Task₁: Research
Assigned to Agent A (role: researcher)
→
Task₂: Write
Assigned to Agent B (role: writer), context from Task₁
→
Task₃: Review
Assigned to Agent C (role: reviewer)

The Crew at crew.py:159 is the top-level orchestrator. It runs in one of two processes (process.py): sequential (tasks execute in order, each agent completes its task before the next begins) or hierarchical (a manager agent — either auto-created from manager_llm or explicitly provided as manager_agent — delegates tasks dynamically).

Tasks at task.py:114 are the unit of work. Each task has a description, an expected_output, an assigned agent, optional context from previous tasks (enabling task chaining), and optional async_execution for background tasks.

Agents at agent/core.py:171 are defined by role, goal, and backstory — a narrative-driven identity model. Each agent has its own internal ReAct loop (the CrewAgentExecutor at crew_agent_executor.py), but the architectural focus is on the Crew/Task layer, not the individual agent loop. Agents can delegate to each other via AgentTools (DelegateWorkTool, AskQuestionTool) at tools/agent_tools/agent_tools.py.

Paradigm B: Conversational Orchestration (AutoGen → MAF)

AutoGen takes the opposite approach: no central manager. Agents communicate through an event-driven runtime, and a GroupChat manager selects who speaks next. The workflow emerges from conversation, not from assignment:

Runtime
asyncio.Queue event loop
→
GroupChat Manager
select_speaker() — RoundRobin / LLM Selector / Swarm
→
Agent speaks
publishes response to group topic
→
Manager selects next
Based on conversation thread

The Runtime: SingleThreadedAgentRuntime at autogen_core/_single_threaded_agent_runtime.py:149 is an actor model with an asyncio.Queue event loop. Agents communicate via two primitives: send_message() (direct RPC to a specific AgentId) and publish_message() (pub-sub broadcast to a TopicId). Agents never call each other directly — all communication flows through the runtime.

GroupChat: BaseGroupChat at _base_group_chat.py:40 creates an embedded runtime and wraps each participant in a ChatAgentContainer. The BaseGroupChatManager at _base_group_chat_manager.py:25 orchestrates via speaker selection: select_speaker(thread) at line 306. Three strategies: RoundRobinGroupChat (cycles through participants), SelectorGroupChat (LLM chooses the next speaker based on conversation context), and SwarmGroupChat (handoff-based agent routing).

AutoGen → MAF: the evolution story

In October 2025, Microsoft announced AutoGen was entering maintenance mode. Its successor, Microsoft Agent Framework (MAF), merges AutoGen's conversational patterns with Semantic Kernel's enterprise foundation. This is not a rewrite — MAF is a new codebase that inherits architectural ideas from both parents:

Inherited fromWhat
AutoGenConversational multi-agent patterns, GroupChat, event-driven runtime, pub-sub messaging
Semantic KernelEnterprise middleware (filtering, logging, auth), type-safe .NET + Python parity, OpenTelemetry
New in MAFDAG-based Workflow/WorkflowExecutor, EdgeRunner (sequential, fan-out/in, switch-case), Skills system, A2A interoperability

The key architectural addition in MAF is Workflows: deterministic DAG-based execution alongside conversational GroupChat. Where AutoGen only had "agents talking," MAF also has "graphs executing" — a convergence of the conversational and workflow paradigms into a single framework.

When to use which

Role delegation (CrewAI) when the workflow is known in advance — report generation, content pipelines, multi-step analysis where each step has a clear owner. The top-down model gives you predictable, auditable execution.

Conversational (AutoGen/MAF) when the workflow emerges from interaction — collaborative problem-solving, negotiation, debate between perspectives. The bottom-up model handles unexpected directions better because no manager pre-commits to a task sequence.

MAF specifically when you need both: deterministic DAGs for the known parts of your workflow, conversational GroupChat for the parts that require agent discretion. This hybrid is the most architecturally interesting — and the most complex to reason about.

Design heuristic: Top-down is cheaper to build and easier to debug. Bottom-up is more flexible but harder to predict. Choose top-down unless your use case genuinely requires emergent coordination.
