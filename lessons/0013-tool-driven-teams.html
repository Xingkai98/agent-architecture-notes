Lesson 0013: Tool-Driven Teams — AgentScope

When team management is just another tool call — AgentScope's metacognitive approach to multi-agent coordination.

Course 3: Multi-Agent Orchestration · June 2026 · Permalink

Lesson 0012 covered two approaches to multi-agent coordination: CrewAI's top-down role delegation and AutoGen/MAF's bottom-up conversation. AgentScope (Alibaba, ~23k stars) introduces a third paradigm: team management as agent tools. The leader agent creates, communicates with, and destroys team members using the same tool-calling interface it uses for file operations and web searches. This is a metacognitive approach — the agent reasons about team structure the same way it reasons about any other task.

AgentScope is also unique among the three multi-agent frameworks in being a full single-agent harness — its Agent class has a complete ReAct loop with middleware, Toolkit/ToolGroup, and Workspace isolation. It can function as a standalone agent or recursively spawn itself into a multi-agent system.

Teams as tools: the core insight

In CrewAI, team management is a separate orchestration layer (the Crew class). In AutoGen, it's a separate runtime (the GroupChat). In AgentScope, it's just tools:

TeamCreate
Establish a team with caller as leader
→
AgentCreate
Spawn worker with session, toolkit, workspace
→
TeamSay
Send messages via MessageBus
→
TeamDelete
Dissolve the team

TeamCreate at _tools/_team_create.py:30 creates a persistent team record. The calling agent becomes the leader. Workers are spawned by subsequent AgentCreate calls.

AgentCreate at _tools/_agent_create.py:136 spawns a worker agent with its own session, toolkit, permission context, workspace, and SubAgentTemplate (a reusable blueprint at _types.py:29 with type, description, system_prompt_template, context_config, react_config). The leader assigns work through the prompt parameter — not through a task list or conversation thread, but as a parameter to the tool call that creates the worker.

TeamSay at _tools/_team_say.py:74 sends a HintBlock to a specific member or broadcasts to all members. Messages flow through the MessageBus, not direct agent-to-agent calls.

This inverts the control model: instead of a framework managing agents, the agent manages its own team through tools. The leader can decide at runtime how many workers to spawn, what roles they should have, and when to communicate with them — the same kind of decision-making it applies to reading files or searching the web.

MessageBus: the communication fabric

The MessageBus at message_bus/_base.py:49 provides three transport modes:

ModePatternUse case
Drain queueSingle-consumer, ack-on-readPer-session inboxes for agent messages
Replay logMulti-consumer, cursor-trackedSSE event replay for frontend streaming
Transient broadcastFire-and-forget pub/subWake-up signals, status notifications

The InboxMiddleware at app/middleware/_inbox_middleware.py:24 drains the session's inbox before each reasoning step, injecting HintBlock objects (team messages, tool offload results) into the agent's context. This means agent-to-agent communication is asynchronous — the leader sends a message, and the worker picks it up on its next reasoning cycle.

The AgentScope hybrid: single-agent harness meets multi-agent platform

Unlike CrewAI and AutoGen which are purely multi-agent frameworks, AgentScope is a complete single-agent harness that can scale to multi-agent:

  • Agent class at _agent.py:94 — full ReAct loop (_reply_impl() at line 542), reasoning phase (_reasoning_impl() at line 755), tool batching (_batch_tool_calls() at line 1102)
  • Toolkit/ToolGroup at tool/_toolkit.py:66 — group-based tool activation/deactivation, MCP integration, skill loading
  • Workspace at workspace/_base.py:36 — Local, Docker, and E2B sandbox isolation with lifecycle management
  • ChatService at app/_service/_chat.py:46 — the harness entry point that assembles agent, toolkit, middleware, and workspace

When a leader calls AgentCreate, it spawns another instance of this entire harness — new session, new workspace, new toolkit — running independently. The leader communicates through the MessageBus, not through shared memory. Each worker is a full agent, not a lightweight stub.

When to use AgentScope's approach

The tool-driven model is ideal when the agent should decide its own team structure. If you don't know in advance how many workers you'll need or what roles they should have — let the leader agent figure it out at runtime. This is metacognition: the agent reasons about how to organize work, not just how to do it.

The cost is opacity: tool-driven team creation is harder to debug than a predefined Crew or GroupChat because the team structure is not visible in the code — it emerges from the leader's tool calls.

Design heuristic: When team structure is known, use CrewAI (simpler). When it emerges from conversation, use AutoGen/MAF. When the agent should figure out its own team, use AgentScope. The metacognitive model is the most powerful — and the hardest to predict.
