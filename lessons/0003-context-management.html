Lesson 0003: Context Management

How AI agents manage their context window — compaction, token budgets, and the tree model.

June 2026 · 9 primary source codebases · Permalink

The context window is every agent's hardest resource constraint. Too little context and the agent forgets what it was doing. Too much and the LLM gets slow, expensive, and confused. Every codebase solves this differently, but they all face the same problem: how to keep what matters and drop what doesn't.

In Lesson 0002, we covered tool definition through normalization. This lesson zooms into the opposite direction: how agents squeeze tool results and conversation history back down to fit the context window.

The four fundamental strategies

Across the nine codebases, four distinct compaction strategies emerge:

Proactive (before the call) Reactive (after the error)
  1. Proactive compaction (claude-code, hermes-agent): Check token count before every API call. If over threshold, compact first. The safest approach — the LLM never sees an overfull context.
  2. Reactive compaction (pi-mono, openclaw, hermes-agent): Wait for the provider to return a 413 prompt-too-long error, then compact and retry. Simpler to implement but wastes one API call.
  3. Per-turn consolidation (nanobot): Run a lightweight consolidation check every turn; if needed, run up to 5 rounds of summarization before proceeding.
  4. Delegate-to-provider (opencode, codex): Leave compaction to an external package (@opencode-ai/core) or configurable shell hooks. No built-in LLM summarizer.

The most sophisticated: claude-code's 7-layer approach

Claude Code has the most elaborate context management of all nine codebases. Seven distinct layers work together:

Layer 1: Token budget (gatekeeper)

Before every API call, src/query/tokenBudget.ts checks whether the conversation fits. The threshold is 90% of the context window (COMPLETION_THRESHOLD = 0.9). It also detects diminishing returns: if 3 or more continuations each add less than 500 tokens of new content, the system forces a compaction — the conversation is spinning without progress.

The diminishing-returns heuristic is subtle but important. Without it, an agent that keeps making tiny edits to the same file would never trigger compaction by token count alone — each turn adds little. But across 10+ turns, the context fills with near-duplicate tool results. The heuristic catches this pattern: "you're not adding meaningful new information, so compact now."

// src/query/tokenBudget.ts — threshold and diminishing-returns check
const COMPLETION_THRESHOLD = 0.9;

function isDiminishingReturns(continuations: Continuation[]): boolean {
  return continuations.length >= 3 &&
    continuations.slice(-3).every(c => c.tokenDelta < 500);
}

Layer 2: Context window resolution

src/utils/context.ts — getContextWindowForModel() resolves the active model's context window. It checks for the [1m] suffix (denoting million-token models), consults model capabilities, and defaults to 200K tokens. This is the foundation that all other layers reference.

Layer 3: Token counting

src/utils/tokens.ts — tokenCountWithEstimation() walks backward from the last known API usage point. Everything after that is estimated rather than precisely counted. This gives a fast, reasonably accurate count without re-counting everything from scratch.

Layer 4: Auto-compact (proactive)

src/query.ts:454 — before every API call, the system checks the token threshold. If exceeded, it calls buildPostCompactMessages() which runs summarization and produces a compacted conversation. This is the primary compaction path.

Layer 5: Reactive compact

src/services/compact/reactiveCompact.ts — if auto-compact somehow misses the trigger (race condition, unexpected token surge), the reactive layer catches 413 prompt-too-long errors. It runs a single-shot compaction and retries. There's no retry loop — if reactive compaction fails to reduce enough, the error propagates.

Layer 6: Micro-compact

src/services/compact/microCompact.ts — before the main compaction runs, micro-compact collapses identical consecutive tool results. If a tool was called 5 times with the same result, only 1 entry is kept. This often reduces token count enough to avoid full summarization.

Layer 7: Context collapse, session memory, and snip

Three specialized compaction types:

  • Context collapse (src/services/contextCollapse/): produces a projected read-time view using a commit-log model — each collapse is a "commit" of the state, and the LLM sees a summary rather than raw history.
  • Session memory compact (src/services/compact/sessionMemoryCompact.ts): experimental, uses session memory data for a higher-quality summary.
  • Snip compact (src/query.ts:401): removes hidden or intentionally-snipped message content before the conversation is sent.

System prompt composition

src/utils/systemPrompt.ts — buildEffectiveSystemPrompt() merges prompts from 5 sources in priority order: override > coordinator > agent > custom > default. Each layer can add, override, or remove sections. This isn't compaction per se, but it controls the fixed token overhead at the top of every conversation.

Why this matters: the system prompt is fixed overhead that sits at the top of every API call. A 10K token system prompt on a 200K window with 90% threshold leaves only 170K for actual conversation — the system prompt consumes 5% of the available budget permanently. Layered composition with overrides lets different agent modes (coordinator, sub-agent, standalone) tune this overhead.

The session-tree model (pi-mono, openclaw)

pi-mono and openclaw share a fundamentally different data model: history is a tree of typed entries, not a flat array.

// Simplified tree structure
type SessionEntry =
  | { type: 'message', role, content }
  | { type: 'compaction', summary, entriesCompacted }
  | { type: 'branch_summary', branchId, summary }

// buildSessionContext() walks the tree → flat message list
// beforeBranch → ... → compaction → afterCompaction → ... → current

Key properties of this model:

  • Compaction is recorded, not destructive: compacted entries remain in the tree. They're just not included in the active context. You can "uncompact" if needed.
  • CompactionSettings: { enabled: true, reserveTokens: 16384, keepRecentTokens: 20000 } — always preserves room for the LLM response and the most recent turns.
  • Split-turn compaction: when the token cut point falls in the middle of a tool-call turn (some tools returned, some didn't), both the prefix and suffix of that turn are summarized separately. The model never sees a half-finished tool cycle.
  • File operation tracking: file operations (reads, writes, edits) are tracked across sessions and appended to compaction entries, giving the summary awareness of which files were touched.
  • Iterative summary updates: UPDATE_SUMMARIZATION_PROMPT merges new messages into an existing summary rather than re-summarizing from scratch. This is cheaper and preserves continuity.

The 3-layer memory context (hermes-agent)

Hermes-agent combines three sources into a single context:

┌─────────────┐  ┌──────────┐  ┌──────────────┐
│  MEMORY.md   │  │ USER.md  │  │  SQLite FTS5 │
│  (project)   │  │  (user)  │  │  (search)    │
└──────┬───────┘  └────┬─────┘  └──────┬───────┘
       └───────────────┼───────────────┘
                       ▼
              ContextCompressor

The ContextCompressor at agent/context_compressor.py runs a 5-phase algorithm:

┌─────────┐ ┌─────────────┐ ┌──────────────┐ ┌─────────────┐ ┌──────────────┐ │ PHASE 1 │ → │ PHASE 2 │ → │ PHASE 3 │ → │ PHASE 4 │ → │ PHASE 5 │ │ Prune │ │ Boundaries │ │ Prior Summary │ │ LLM Summary │ │ Assemble │ │ │ │ │ │ │ │ │ │ │ │ Trim │ │ Which msgs │ │ Update vs. │ │ Generate │ │ prefix + │ │ verbose │ │ stay raw │ │ re-create │ │ concise │ │ summary + │ │ tool out │ │ vs. summary │ │ summary │ │ summary │ │ recent msgs │ └─────────┘ └─────────────┘ └──────────────┘ └─────────────┘ └──────────────┘
  1. Prune tool results: trim verbose tool output before anything else. This is the cheapest reduction and often sufficient on its own.
  2. Determine boundaries: find the cut point — which messages stay as raw text and which get summarized.
  3. Search for prior summary: if a summary already exists, retrieve it so the compressor can update rather than re-create.
  4. Generate LLM summary: call the LLM to produce a concise summary of the pruned entries.
  5. Assemble: combine system prompt + summary (reference-only prefix) + recent messages into the final context.

Key heuristics in hermes-agent's approach:

  • Anti-thrashing: should_compress() skips compaction if the last 2 compressions each saved less than 10% of tokens. Compaction has a cost — if it's not paying off, stop doing it.
  • Reference-only prefix: the summary is prefixed with explicit instructions that it is background information, not active instructions. This prevents the model from treating stale summaries as commands.
  • Fallback deterministic summary: if the LLM summarization call fails, the compressor falls back to a rules-based summary (extract first sentence of each message, cap at 8K chars).
  • Preflight compression: runs before the tool loop starts, not during. Up to 3 passes. This means the agent never starts a turn with an overfull context.

File reference system with hard limits (hermes-agent)

Hermes-agent's @reference system supports @file:, @folder:, @git:, and @url: directives. References expand to their content, but subject to a 50% context window hard limit — no reference can consume more than half the available context. This prevents a single large file from crowding out conversation history.

The 50% limit is not arbitrary. It guarantees that conversation history always has at least as much room as injected references, which means the model can always "think about" the injected content with equal-weight history context. A 90% limit would let references dominate; a 20% limit would make references too small to be useful for large files. 50% is the balanced split.

The @folder: directive is the most dangerous of the four — recursively expanding a directory tree can pull in thousands of files. Hermes-agent handles this by expanding entries with file-size estimates first, sorting by relevance (using the FTS5 index), and stopping when the 50% budget is exhausted.

The explicit state machine (nanobot)

Nanobot makes context management a first-class state in its turn loop:

// nanobot/agent/loop.py — TurnState enum
RESTORE → COMPACT → COMMAND → BUILD → RUN → SAVE → RESPOND → DONE

Each component has a clear role:

  • AutoCompact (nanobot/agent/autocompact.py): idle-session TTL-based archiving. If a session has been idle, it keeps the last 8 messages and summarizes the rest. This is orthogonal to per-turn consolidation — it handles the case where the user walks away and comes back hours later.
  • Consolidator (nanobot/agent/memory.py): maybe_consolidate_by_tokens() runs per-turn. If the context exceeds the budget, it runs up to 5 consolidation rounds, each round calling the LLM to compress the oldest uncompressed messages. The multi-round design handles the case where a single summarization pass doesn't reduce enough — it keeps going until the context fits.
  • ContextBuilder (nanobot/agent/context.py): assembles the final context in this order: identity → bootstrap files → tool contract → memory → skills → history (capped at 120 messages) → session summary. The order matters: identity and tool contract are fixed overhead; history is the most volatile and the first thing to get capped or summarized.

Hook-driven and delegate-based approaches

Two codebases take a fundamentally different approach — they don't implement compaction themselves:

codex (hook-driven): Compaction is driven by configurable hooks in codex-rs/hooks/src/events/compact.rs. The PreCompactRequest and PostCompactRequest structs carry context about what to compact and what was compacted. The actual summarization is done by user-configured shell commands — codex has no built-in LLM summarizer. Context fragments are delimited by typed markers so the hook scripts can identify sections.

This design is philosophically consistent with codex's approach to sandboxing (Lesson 0002): the framework provides the event and the contract, but the policy is entirely user-defined. A PostCompactRequest carries the compacted result and the token counts before/after, letting the hook decide whether to accept or redo the compaction.

opencode (delegate): Compaction is delegated entirely to the external @opencode-ai/core package and to the provider level. OpenCode itself doesn't run summaries — it relies on the provider's native context handling. This keeps the OpenCode codebase small, but means compaction behavior is opaque and non-portable across providers.

The tradeoff is clear: codex and opencode have zero lines of summarization code to maintain, debug, or security-review. But they also have zero control over summarization quality. If the hook script or provider produces a bad summary, there's no fallback and no recourse.

Middleware-driven context management (deepagents, deer-flow)

DeepAgents and DeerFlow introduce a fundamentally different pattern: context management as explicit named middleware rather than implicit logic inside the agent loop. This is the same "explicit middleware vs implicit coupling" pattern seen in Lessons 0001 and 0002 — but applied to context management. In the existing seven codebases, compaction, token counting, and content trimming are functions called inside the agent loop. In DeepAgents and DeerFlow, they are standalone middleware classes with clear interfaces and configurable triggers.

DeepAgents: SummarizationMiddleware and FilesystemMiddleware

DeepAgents encodes context management in two dedicated middleware classes:

SummarizationMiddleware (middleware/summarization.py:1003) wraps model_call and performs LLM-driven summarization when a token threshold is exceeded. The default trigger is ("fraction", 0.85) when the model has a known max_input_tokens, otherwise ("tokens", 170000). This gives DeepAgents the same proactive posture as claude-code and hermes-agent — check before calling, not after error. If the LLM summarization call itself fails with ContextOverflowError, it falls back to emergency summarization (truncating old tool arguments).

FilesystemMiddleware (middleware/filesystem.py:692-749) automatically offloads large tool results to the filesystem backend, replacing them with previews in the context. The token limits are 20,000 for tool results and 50,000 for human-injected messages. This is an automatic content gating mechanism — the LLM sees previews, not raw dumps, and the full content is retrieved on demand. It is functionally equivalent to hermes-agent's tool output pruning (phase 1) and claude-code's micro-compact, but implemented as a standalone middleware layer that can be toggled on or off independently.

DeepAgents also uses a DeltaChannel on messages (graph.py:67) that reduces checkpoint growth from O(N²) to O(N) by only storing deltas, with a full snapshot every 50 steps. This is a structural optimization — it doesn't change what the LLM sees, but dramatically reduces the storage cost of maintaining conversation history in stateful graphs.

The system prompt assembly order in graph.py:70-143 is: USER → (BASE or CUSTOM) → SUFFIX, joined by blank lines. This is a simpler composition model than claude-code's 5-priority merge, but achieves the same goal: fixed overhead at the top of every conversation that can be tuned per agent mode.

DeerFlow: 19-middleware pipeline with deferred tool filtering

DeerFlow pushes the middleware pattern even further with a 19-middleware chain:

ThreadDataMiddleware → UploadsMiddleware → SandboxMiddleware →
DanglingToolCallMiddleware → SummarizationMiddleware → TodoListMiddleware →
... → ClarificationMiddleware (19 total)

Each middleware addresses a specific cross-cutting concern. For context management, three are particularly relevant:

  • DynamicContextMiddleware injects the current date and memory as a <system-reminder> into the first HumanMessage. This is a lightweight alternative to hermes-agent's three-source context assembly (MEMORY.md + USER.md + FTS5) — it injects dynamic state without reserving permanent system prompt space.
  • SummarizationMiddleware is configurable via config.yaml: triggers can be token-based, message-count-based, or fraction-of-max-input. Keeps recent messages while summarizing older ones. ContextConfig at agent/_config.py:56 exposes trigger_ratio (default 0.8), reserve_ratio (default 0.1), compression_prompt, and summary_template — all user-configurable without modifying the source.
  • DeferredToolFilterMiddleware hides MCP tool schemas from the LLM until a tool_search promotion action. This is a pre-emptive context-saving strategy: instead of sending all available tool schemas in every system prompt (which can consume thousands of tokens per turn), only promoted tools are visible. The rest are deferred until needed.

The deferred tool filtering is DeerFlow's most distinctive contribution to context management. In codebases like claude-code or hermes-agent, all tool schemas are always present in the system prompt. For an agent with 50+ MCP tools, that is 5,000–15,000 tokens of fixed overhead per turn. DeerFlow's approach collapses that to only the tools the agent has explicitly requested — the functional equivalent of lazy-loading for tool definitions.

Explicit vs implicit: the middleware pattern across lessons

This is the third time the explicit-middleware pattern appears in our codebase survey:

  • Lesson 0001 (Agent Loop): DeepAgents and DeerFlow externalize the loop as middleware, while the other 7 codebases embed loop logic directly in the agent.
  • Lesson 0002 (Tool System): DeepAgents normalizes tools through middleware (ToolNormalizationMiddleware), while other codebases handle normalization ad-hoc inside tool registration.
  • Lesson 0003 (Context Management): DeepAgents and DeerFlow make compaction, content gating, and tool visibility into named middleware classes, while the other 7 codebases handle these concerns implicitly inside the loop.

The pattern is consistent: middleware makes context management composable (you can add/remove/reorder strategies), testable (each middleware is an isolated unit), and observable (the middleware name tells you exactly which concern is being applied). The cost is conceptual overhead — a 19-middleware chain is harder to reason about end-to-end than a single buildPostCompactMessages() function. But for complex agents that need to mix and match context strategies, the middleware approach scales more gracefully than ad-hoc conditionals inside a loop.

Why the tree model matters

The difference between a flat array and a tree is not just a data structure choice — it determines what operations are possible:

OperationFlat array (claude-code, hermes-agent, nanobot, deepagents, deer-flow, codex, opencode)Tree (pi-mono, openclaw)
Undo compactionMust keep old array copy or re-requestChange which nodes are "active" — no data loss
Branch from mid-conversationCopy the array up to that pointAdd a new branch node — O(1)
Merge branchesManual reconciliation of two arraysWalk both subtrees, produce merged summary node
Re-summarize with different granularityCan't — the original messages are goneDelete old compaction node, create new one from preserved children

pi-mono and openclaw chose the tree model primarily to support branching — the same conversation can fork into multiple directions. But the secondary benefit (undoable, redoable compaction) turned out to be just as valuable.

Design heuristics

Patterns that hold across the codebases:

If you control the context window size, proactive is better than reactive. Claude Code and hermes-agent both check tokens before calling the LLM. This avoids wasted API calls and gives the user a smoother experience. Reactive only makes sense when the provider controls the window and you can't predict where the limit is.
If you have multiple context sources, the tree model is cleaner than flat arrays. pi-mono's session tree stores messages, compactions, and branch summaries as typed nodes. This makes it trivial to reconstruct context for any point in the session, undo compactions, or fork branches. Flat arrays force you to mutate or duplicate history.
User files (MEMORY.md, USER.md) must be "reference-only" in summaries. Hermes-agent's reference-only prefix is a critical safety measure: compacted MEMORY.md content should inform the model but never override the user's current instructions. Blending memory into the system prompt without this distinction causes the agent to follow stale directives.
Tool output pruning before LLM summarization significantly reduces compaction costs. Every codebase that does compaction prunes tool outputs first. Hermes-agent's phase 1 (prune) and claude-code's micro-compact both do this. LLM summarization is expensive — trimming verbose tool output first pays for itself immediately.

What happens when compaction fails?

Every compaction system can fail — the LLM summarizer returns garbage, the token count doesn't drop enough, or a bug corrupts the summary. How each codebase handles this:

  • claude-code: if reactive compaction fails to reduce context below the limit, the error propagates to the user. The single-shot design means there's no infinite retry loop.
  • hermes-agent: the deterministic fallback summary (8K chars, rule-based extraction) guarantees some summary is produced even if the LLM is down. The anti-thrashing check also prevents the compressor from burning tokens on ineffective compactions.
  • pi-mono / openclaw: if the LLM summarization fails, the compaction entry is not created and the original messages stay in the tree. The agent proceeds with an overfull context — which may trigger a provider 413 error on the next call, but at least the data is preserved.
  • nanobot: the multi-round consolidation (up to 5 rounds) provides a natural backstop. If after 5 rounds the context still doesn't fit, the oldest messages are truncated (not summarized) as a last resort.
  • codex: hook scripts control failure handling entirely. The PostCompactRequest carries before/after token counts — a well-written hook can detect insufficient reduction and retry with different parameters.
  • opencode: failure handling is entirely the provider's responsibility. If the provider's compaction fails, the agent simply stops working.
  • deepagents: if the LLM summarization raises ContextOverflowError, the SummarizationMiddleware falls back to emergency summarization — truncating old tool arguments rather than attempting LLM summarization. This is a structural fallback, not a retry.
  • deer-flow: the configurable reserve_ratio (default 0.1) ensures space is always reserved for the LLM response, so compaction failure manifests as a truncated summary rather than a crashed turn. The 19-middleware chain means a failure in one middleware (e.g., SummarizationMiddleware) does not block other middleware (e.g., TodoListMiddleware) from running.
