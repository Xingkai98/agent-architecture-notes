Lesson 0004: Permission Model

From "trust the LLM completely" to "trust nothing" — how seven agent codebases decide what a tool can and cannot do.

June 2026 · 7 primary source codebases · Permalink

Permission models in AI agents answer two questions: can the LLM ask for this tool? and can the harness safely execute it? These are different questions, and the codebases answer them at different points in the lifecycle. The difference between a hook, an ACL, and a sandbox is when the check happens and who enforces it.

In Lesson 0002, we saw that tool dispatch is a central stage in every agent loop. This lesson zooms into the permission checks that gate that dispatch — how each codebase prevents the LLM from doing things it should not.

The permission spectrum

Every codebase sits somewhere on this continuum:

No permissions Full sandboxing
  1. No permissions: nanobot, hermes-agent — tools run unconditionally. Any gating (check_fn, SSRF detection) is an availability or safety net, not a security boundary.
  2. Hook-based delegation: pi-mono, openclaw — the framework provides a beforeToolCall hook; the caller implements whatever permission logic they want. The framework itself is neutral.
  3. ACL (Access Control List): claude-code, opencode — each tool has a permission level (allow, ask, deny). Rules are evaluated at registration time (what the LLM can see) and execution time (what actually runs).
  4. Full sandboxing: codex — the OS kernel enforces boundaries. Tools run inside Landlock, Windows sandbox, or macOS sandbox. The sandbox can escalate on retry.

Claude Code — Permission hooks + ACL

Claude Code has the most fully-realized ACL system among the seven codebases. Permission checking is a first-class lifecycle method on every tool.

Per-tool permission check

Every tool in src/Tool.ts has a checkPermissions() method. The buildTool() factory provides a safe default — if the tool author does not override it, the tool defaults to asking the user. This means new tools are safe by default.

// claude-code: src/Tool.ts — each tool carries its own permission check
buildTool({
  name: ToolNames.BASH,
  // ... schema, call handler ...
  // buildTool provides defaults for:
  // checkPermissions(input, context) — default: ask user
  // isConcurrencySafe, isReadOnly, isEnabled,
  // validateInput, renderToolUseMessage, renderToolResultMessage
})

Permission filtering pipeline

The tool list that reaches the LLM is not the full set of registered tools. Three filters are applied in sequence:

// claude-code: src/tools.ts:271 — getTools() filters by context and mode
// Returns only tools allowed in the current permission context.
// Tools can be excluded by: permission mode, feature flags, deny rules.

// claude-code: src/tools.ts:263 — filterToolsByDenyRules()
// Applies user-configured deny rules before the LLM ever sees the tool list.
// Deny rules can target specific tools by name.

// claude-code: src/tools.ts:345 — assembleToolPool()
// Merges built-in tools + MCP tools with deduplication.
// MCP tools inherit the permission context of the session.

Permission levels and persistence

Each tool can be set to one of three levels: allow (run without asking), ask (prompt the user each time), or deny (block outright). The --dangerously-bypass-permissions flag overrides all checks — an explicit escape hatch for automated environments.

Permission decisions are persisted at three scopes: session-scoped (lasts for the current invocation), project-scoped (stored in .claude/settings.json), and user-scoped (stored in ~/.claude/settings.json). This gives the user granular control over how long a permission grant lasts.

Codex — Multi-layered sandboxing

Codex is the only codebase with a serious sandboxing story. Where Claude Code asks "should this tool run?", Codex asks "can the OS enforce that this tool cannot escape?"

The orchestrator pipeline

The ToolOrchestrator in codex-rs/core/src/tools/orchestrator.rs runs every tool through a four-stage pipeline:

┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ APPROVAL │ → │ SANDBOX │ → │ EXECUTE │ → │ RETRY │ │ │ │ SELECTION │ │ │ │ (escalated) │ │ User prompt │ │ Landlock / │ │ Run tool │ │ If sandbox │ │ or auto- │ │ Win / Mac │ │ inside │ │ blocked, │ │ approve │ │ sandbox pick │ │ sandbox │ │ retry higher │ └──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘

Approval: Before execution, the orchestrator checks whether the tool needs user approval. Network access has its own approval system with two modes: immediate (ask now) and deferred (allow the tool to run but gate network use until approved).

Sandbox selection: The orchestrator picks the appropriate sandbox for the platform: Landlock on Linux (kernel-level filesystem restrictions), Windows sandbox, or macOS sandbox. Each sandbox has a restricted and an elevated variant.

Retry with escalation: This is the unique feature. If a tool fails because the restricted sandbox blocked an operation (e.g., a file write outside the workspace), the orchestrator can retry with the elevated sandbox — the user does not need to re-prompt. The sandbox becomes a dynamic property rather than a static wall.

Parallel execution safety

codex-rs/core/src/tools/parallel.rs defines ToolCallRuntime, which manages concurrent tool execution. Each tool's sandbox is independent, so parallel tools cannot interfere with each other — a property that hook-based and ACL-based systems cannot guarantee.

Opencode — ACL-based

Opencode uses Effect-TS to encode permission rules into the type system. Permission checking happens at two points: materialization time (before tool definitions are sent to the LLM) and execution time (when the tool actually runs).

Sub-agent permission derivation

// opencode: packages/opencode/src/agent/subagent-permissions.ts
// deriveSubagentSessionPermission():
//   parent deny rules  +  subagent allow rules
//   = effective subagent permission set
//
// Sub-agents can only use tools that:
//   1. The parent has NOT explicitly denied, AND
//   2. The subagent's own allow rules permit

Tool execution wrapping

packages/core/src/tool/tool.ts provides withPermission(), which wraps tool execution. The wrapper checks permission rules before the tool's execute() is called. If denied, the tool never runs — the LLM receives an error result.

packages/core/src/tool/registry.ts applies permission rules at materialization time — before the tool list is sent to the LLM. If a tool is denied, the LLM cannot even see it in the function list. This is a critical design choice: if the LLM cannot ask for it, it cannot social-engineer the user into approving it.

Default-deny for dangerous tools

Two tools are default-deny in sub-agents: todowrite and task. The Wildcard.match utility is used for deny-rule matching on tool names, supporting glob patterns so a single rule can cover families of tools (e.g., mcp__* to deny all MCP tools).

Pi-mono, OpenClaw — Hook-based delegation

Neither pi-mono nor openclaw ships with a built-in permission system. Instead, the framework exposes a beforeToolCall hook in agent-loop.ts. The caller — the application that embeds the agent framework — implements permission checking by providing a hook function.

// pi-mono / openclaw: packages/agent/src/agent-loop.ts
// The framework calls beforeToolCall(toolCall, context) before dispatch.
// If the hook throws, the tool is skipped and the error is wrapped
// as a ToolResultMessage with isError: true.
//
// The framework itself has NO opinion about what is allowed.
// All permission logic lives in the consumer application.

When a tool throws (whether from a permission hook or from the tool itself), the agent loop catches the exception and wraps it as an error result message. The LLM sees the error and can decide how to proceed. This means the permission model degrades gracefully — a denied tool is just another error the LLM can reason about.

Hermes-agent, Nanobot — Minimal / Nothing

These two Python codebases have the thinnest permission layers. They are included on the spectrum to show what "no permission model" looks like in practice.

Hermes-agent: check_fn as availability gate

Each ToolEntry in hermes-agent's registry can have a check_fn — but this is an availability gate, not a security boundary. It answers "is this tool's dependency available?" (e.g., is Docker running?), not "is the LLM allowed to use this tool?" The max_result_size_chars per-tool setting is a resource limit, not a permission check.

# hermes-agent: tools/registry.py:77
class ToolEntry:
  name: str
  check_fn: Callable | None  # "is Docker installed?" — not "may the LLM use this?"
  max_result_size_chars: int # resource cap, not security

Nanobot: SSRF detection and workspace boundaries

Nanobot's runner.py has two safety checks embedded in tool execution, not in a permission layer:

  • SSRF detection: Before a tool makes an HTTP request, nanobot checks if the target is a private IP. If so, the tool is blocked and the LLM receives an escalating hint.
  • Workspace boundary violation: File operations outside the workspace root are detected and classified with escalating hints — from a gentle "you are outside the workspace" to a hard block.

These are classified as escalating hints, not permission denials. The LLM is nudged, not stopped — a fundamentally different philosophy from ACL-based or sandbox-based approaches.

The fundamental tradeoff: safety vs. complexity

The permission model is where safety guarantees meet implementation complexity head-on:

More safety guarantees = more code, more failure modes, more user friction. Codex's sandbox escalation means a tool can retry automatically, but it requires platform-specific sandbox implementations (three different OS sandboxes, each with its own API and failure modes). Claude Code's three-scope permission persistence gives users fine-grained control, but it also means permission state can be confusing — "why is this tool asking me again? did I allow it for the session or the project?"

Less safety infrastructure = simpler code, but the user (or the embedding application) shoulders all the risk. Pi-mono's hook-based approach has zero permission code in the framework. That is elegant — until an LLM runs rm -rf / because nobody wired up the hook.

Design heuristic: Permission at registration/materialization time keeps the LLM from asking for things it cannot do. Permission at execution time keeps the harness safe from the LLM doing things it should not. The strongest systems (claude-code, opencode, codex) do both. The weakest (hermes-agent, nanobot) do neither — they gate on availability or nudge with hints, trusting the LLM to behave. Choose your point on the spectrum based on your threat model: if the LLM has access to a shell, execution-time enforcement is not optional.
