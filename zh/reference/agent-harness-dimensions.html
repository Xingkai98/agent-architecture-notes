Agent Harness 架构维度

定义 Agent 系统架构的设计维度分类体系。

参考文档，2026-06-13。基于对 7 个生产级代码库的分析。

1. 核心循环（Core Loop）

最基础的执行循环：Agent 如何接收输入、调用 LLM、执行工具（Tool）并决定继续或停止。每个 Agent 都具备此机制。

关键设计决策

  • 循环嵌套深度：从单层（claude-code）到四层（codex）。层级越多，关注点分离越清晰，但需要追踪的控制流也更复杂。
  • 状态表示：通过迭代传递的可变状态 vs. 事件流 vs. 原地修改消息列表。
  • 工具执行时机：在流式输出完成后执行（批量模式）vs. 在流式输出过程中执行（流式执行器）。
  • 轮次门控（Turn Gating）：什么构成一个"轮次"？有些 Agent 区分"轮次"（一次 LLM 调用 + 工具结果）和"迭代"（单次 LLM 调用）。

代码库链接

  • pi-mono: packages/agent/src/agent-loop.ts — runLoop()
  • nanobot: nanobot/agent/loop.py — AgentLoop.run()
  • codex: codex-rs/core/src/session/turn.rs — run_turn()
  • opencode: packages/core/src/session/runner/llm.ts — run()
  • openclaw: packages/agent-core/src/agent-loop.ts — runLoop()
  • claude-code: src/query.ts — queryLoop()
  • hermes-agent: agent/conversation_loop.py — run_conversation()

2. 工具系统（Tool System）

工具如何被定义（Schema）、发现（注册表/Registry）、调用（调度/Dispatch），以及其结果如何被规范化处理后反馈给 LLM。

关键设计决策

  • 并发模型：仅顺序执行 vs. 批量并行（Promise.all / asyncio.gather / FuturesOrdered）vs. 流式执行器（工具在 LLM 流式输出期间运行）。
  • 工具注册：硬编码枚举 vs. 基于插件/MCP 的发现机制 vs. 文件系统扫描。
  • 结果规范化：大小截断、大结果的文件卸载、用于短路循环的"终止"标志。
  • 沙箱（Sandboxing）：工具在哪个层级实现沙箱隔离？工具级别（bash 沙箱）、进程级别，还是容器级别？

3. 上下文管理（Context Management）

对话上下文（发送给 LLM 的消息）如何被组合、管理和压缩以保持在模型的上下文窗口（Context Window）之内。

关键设计决策

  • 压缩（Compaction）策略：主动式（在溢出前压缩）vs. 被动式（收到 413 错误后压缩）。摘要 vs. 截断 vs. 微压缩。
  • 系统提示词（System Prompt）组合：静态基础部分 + 动态注入（记忆、技能、运行时上下文）vs. 单一模板。
  • 消息预算：每条消息的大小上限、工具结果截断、从头部裁剪对话历史。
  • 缓存控制：提示词的哪些部分被缓存？因提供商而异（Anthropic 断点标记 vs. 自动缓存）。

4. 权限与安全模型（Permission & Safety Model）

Agent 如何决定哪些操作是允许的，以及使用什么机制来执行这些决策。

关键设计决策

  • 审批（Approval）模式：始终询问（交互式）vs. 自动批准（沙箱）vs. 基于 ACL（按工具/路径授权）。
  • 沙箱：无沙箱（信任模型）vs. 目录限制 vs. 禁用网络 vs. 完全容器隔离。
  • 权限持久化：会话级别 vs. 跨会话持久化 vs. 层级式（项目 -> 用户 -> 全局）。
  • 护栏（Guardrails）：工具执行前钩子、工具执行后验证、内容过滤器、工作区边界执行。

5. 记忆架构（Memory Architecture）

Agent 如何跨会话持久化状态——用户偏好、项目约定、学习到的技能、对话历史。

关键设计决策

  • 记忆层级（Memory Tiers）：工作记忆（上下文内）vs. 短期记忆（会话历史）vs. 长期记忆（持久化文件/数据库）。
  • 存储格式：Markdown 文件（MEMORY.md、USER.md）vs. SQLite/向量数据库 vs. JSON Blob。
  • 整合（Consolidation）：何时以及如何对记忆进行回顾、摘要和修剪？自动（后台）vs. 手动。
  • 技能创建：Agent 能否从经验中创建新的可复用技能？（hermes-agent 的标志性特性）。

6. 子 Agent 与委托（Sub-agent & Delegation）

如何将工作拆分到多个 Agent 实例中——派生、通信、结果收集。

关键设计决策

  • 派生（Spawn）模型：同一进程内（函数调用）vs. 子进程 vs. 远程/容器。
  • 隔离（Isolation）：共享上下文（消息对父 Agent 可见）vs. 沙箱化（仅返回最终结果）。
  • 并发：顺序委托 vs. 并行子 Agent 扇出（Fan-out）。
  • 预算（Budget）：子 Agent 是否有自己的 Token/轮次预算？（hermes-agent：是，每个子 Agent 有独立的 IterationBudget）。

7. UI / 传输层（UI / Transport Layer）

用户如何与 Agent 交互——终端 UI、Web、消息渠道、API。

关键设计决策

  • 界面范式：REPL（交互式终端）vs. 一次性调用（CLI 参数）vs. 守护进程（后台）vs. 聊天（消息收发）。
  • 渲染：TUI 库（Ink、Bubble Tea、Textual、自定义）vs. 纯标准输出 vs. Web UI vs. 移动端。
  • 多渠道：单一界面 vs. 统一网关（openclaw：25+ 渠道；hermes-agent：多平台）。
  • 客户端/服务器分离：单体进程 vs. 独立服务器 + 客户端（opencode：Go TUI + TypeScript 服务器）。

8. 错误恢复与韧性（Error Recovery & Resilience）

Agent 如何处理故障——API 错误、上下文溢出、格式错误的响应、工具执行失败。

关键设计决策

  • LLM 错误恢复：带退避的重试 vs. 模型回退（Fallback）vs. 提供商回退。
  • 上下文溢出：自动压缩 vs. 被动压缩（收到 413 后）vs. 折叠排空（移除最早的轮次）。
  • 格式错误的工具调用：验证并重试（hermes-agent：对无效 JSON、无效工具名进行 3 次尝试）vs. 静默注入错误结果。
  • 输出截断：达到最大 Token 数后的恢复（注入继续消息）vs. 提升 Token 限制 vs. 接受部分结果。
