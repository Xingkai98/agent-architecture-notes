第6课：子智能体与任务委派

spawn、isolate、delegate、collect —— 每个多 agent 系统的四个动作。

2026年6月 · 8 个主要源代码库 · 固定链接

每个 sub-agent 系统都回答同样的四个问题：如何 spawn 子级、能看到什么、能花多少资源、以及结果如何流回父级。各代码库之间的区别全部在于在何处划定隔离边界以及如何强制执行它。

在第2课中，我们看到工具是 agent 循环的引擎。本课将视角上移一层：当一个 agent 变成多个时。我们将追踪所有八个代码库中的 sub-agent spawn、上下文隔离、资源预算和结果收集，然后提取无论框架如何都成立的设计启发。

Spawn 频谱

Sub-agent 存在于一个隔离连续体上。越往右，保证越强——开销也越高：

进程内（In-process） 容器（Container）
进程内 ─── 线程 ─── 子进程 ─── 容器 │ │ │ │ │ │ │ └─ 完整操作系统隔离 │ │ └─ 进程边界，IPC 通信 │ └─ 共享内存，GIL 关注 └─ 同一堆，上下文对象克隆

我们的集合中没有代码库对临时 sub-agent 使用容器（尽管有几个将其用于宿主沙箱）。真正的较量在于进程内克隆与子进程 spawn 之间——这个选择塑造了下游的所有事情。

Claude Code —— 进程内上下文隔离

Claude Code 的 sub-agent 系统是我们集合中最复杂的进程内设计。它在同一 Node.js 进程内 spawn 子级，但给每个子级提供隔离的 ToolUseContext，因此它们不会意外修改父级状态。

Spawn：runAgent()

在 src/tools/AgentTool/runAgent.ts:248，runAgent() 通过调用 createSubagentContext() 创建隔离的 sub-agent。子级获得自己的对话历史、自己的工具集和自己的 abort signal——但它在同一事件循环中运行。

上下文隔离：createSubagentContext()

在 src/utils/forkedAgent.ts:345，createSubagentContext() 克隆三个关键状态：

  • readFileState —— 克隆，以便子级独立跟踪其自己的文件读取；父级的 readFileState 不会被子级读取污染。
  • AbortController —— 作为子控制器链接，因此取消父级会取消所有后代。
  • setAppState —— 对异步 sub-agent 替换为 no-op，因为它们无法安全更新共享的 UI 状态。

三种 sub-agent 模式

Claude Code 支持三种不同的 sub-agent 模式：

  1. Synchronous（同步/阻塞）： 父级暂停直到子级完成。当父级需要子级的答案才能继续时使用。这是默认的 AgentTool 模式。
  2. Asynchronous（异步/即发即忘）： 父级分派子级并立即继续。子级结果稍后通过回调或消息总线传递。用于并行探索任务。
  3. Fork： 在 src/tools/AgentTool/forkSubagent.ts:1，fork subagent 继承父级的对话上下文并使用字节相同的 API 前缀。这允许 LLM 提供商在父级和子级之间共享 prompt cache 条目——对于深度嵌套委派来说，这是显著的成本优化。

Swarm 模式

在 src/utils/swarm/backends/InProcessBackend.ts，TeammateExecutor 使用基于文件的 mailbox 实现多 agent 编排。每个 agent 写入并从指定文件读取，完全在一个进程内创建异步消息传递系统。文件系统充当持久化、可检查的消息总线。

Hermes-agent —— ThreadPoolExecutor 配合迭代预算

Hermes-agent 采用线程级方法：每个 sub-agent 在共享池的线程中运行，具有框架强制执行的严格资源限制。

迭代预算

在 agent/iteration_budget.py，每个 sub-agent 获得自己的 IterationBudget（默认 50 次迭代）。这是关键洞察：预算必须分开。一个循环或过度探索的 sub-agent 绝不能耗尽父级的 token 配额。当预算用完时，子级被中断，其部分结果返回给父级。

子级执行

_run_single_child() 创建一个子级 AIAgent 实例，具有：

  • 受限工具集 —— 仅父级明确授予的工具
  • 隔离的 task_id —— 子级对话被独立跟踪
  • 聚焦的 system prompt —— 为子任务重写，而非父级的总体目标

子级的禁用工具

默认情况下，子级不能使用：delegate_task（防止失控递归）、clarify（子级应独立解决歧义）、memory（记忆由父级拥有）、send_message（仅父级与用户通信）、execute_code（沙箱关注）。

并发与嵌套委派

max_concurrent_children 默认为 3。嵌套委派由 role="orchestrator" 结合 delegation.max_spawn_depth 控制——orchestrator 子级可以 spawn 自己的 worker，但深度有上限。subagent 注册表是线程安全的，支持 pause()、resume()、interrupt() 和 list() 操作，用于完整的生命周期管理。

Pi-mono —— 子进程模型

Pi-mono 采用了仅次于容器的最高隔离方法：每个 sub-agent 是一个独立的操作系统进程。

Subagent 扩展

在 packages/coding-agent/examples/extensions/subagent/index.ts，完整的 subagent 扩展被实现为一个独立模块。这表明 pi-mono 中的 sub-agent 是扩展，而非内置在框架核心中——框架提供 spawn 原语（child_process），扩展定义委派协议。

三种执行模式

  1. Single： Spawn 一个子级，等待完成，返回结果。
  2. Parallel： Spawn 最多 8 个子级，并发限制为 4。结果在到达时收集。失败的子级不会阻塞兄弟。
  3. Chain： 顺序 spawn 子级，每个接收前一个子级的输出，通过其 prompt 中的 {previous} 占位符。支持管道式分解："查找相关文件" → "读取那些文件" → "总结发现。"

进程级隔离

runSingleAgent() 使用 child_process.spawn() 配合 --mode json 和 --no-session 标志。子级将 JSON 行写入 stdout；父级以流方式读取。每个任务的输出上限为 50KB，防止任何单个子级向父级泛滥数据。

Codex —— 多 agent v2

Codex 的多 agent 系统是我们集合中功能最完整的，为每个生命周期操作提供了专用工具。

Agent 生命周期工具

在 codex-rs/core/src/tools/handlers/multi_agents_v2.rs，五个工具构成完整生命周期：

工具用途
spawn_agent创建带有任务描述的新 sub-agent
wait_for_agent阻塞直到特定子级完成，收集其结果
send_message向运行中的子级发送中间执行消息
interrupt_agent停止运行中的子级并收集其部分结果
list_agents枚举所有活跃子级及其状态

Spawn 选项

在 codex-rs/core/src/agent/control/spawn.rs:86，spawn_agent() 接受 SpawnAgentOptions：

  • fork_context —— 子级是否继承父级的对话
  • fork_context_last_n_turns —— 包含最近多少轮（滑动窗口）
  • max_threads —— 子级自身并行操作的线程预算

加密的 agent 间通信

Codex 独特地通过 InterAgentCommunication::new_encrypted() 加密所有 agent 间消息。这不是性能优化——它是一种安全属性：即使在同一进程内，agent 也无法在没有明确许可的情况下检查彼此的消息。这在 agent 处理凭证或敏感数据时很重要。

Nanobot —— asyncio.Tasks

Nanobot 使用 Python 原生的 asyncio 在单个事件循环中实现轻量级协作 sub-agent。

Spawn 和管理

在 nanobot/agent/subagent.py:74，SubagentManager.spawn() 创建一个 asyncio.Task——可能的最轻隔离单元。任务与父级并发运行；管理器跟踪其生命周期。

# nanobot: nanobot/agent/subagent.py:74
class SubagentManager:
    async def spawn(self, goal: str, tools: list[str]) -> str:
        task = asyncio.create_task(self._run_child(goal, tools))
        self._tasks.append(task)
        return task_id

结果投递

_announce_result() 通过消息总线将子级结果作为 subagent_result 注入事件发布。父级的上下文管理器接收此事件，并将其作为工具结果注入到父级的对话中——父级永远只将 sub-agent 视为一个异步函数调用。

状态跟踪

SubagentStatus dataclass 跟踪阶段转换：

# nanobot: subagent 状态阶段
initializing → awaiting_tools → tools_completed → final_response → done
                                                              ↘ error

并发由 max_concurrent_subagents 配置值控制。

Opencode —— 子会话

Opencode 将 sub-agent 建模为子会话，重用与顶级 agent 相同的会话机制。

TaskTool

在 packages/opencode/src/tool/task.ts，TaskTool 通过 sessions.create() 创建子会话。子会话是一个完全独立的 agent 对话，拥有自己的上下文窗口、工具集和 abort controller——但它共享相同的进程和相同的 LLM 连接。

权限派生

在 packages/opencode/src/agent/subagent-permissions.ts，子级的权限规则集是从父级派生的。这不是简单的复制：派生函数会剥离对 sub-agent 无意义的权限（例如，如果禁用嵌套则移除 allow_agent_spawn），并添加限制（例如，worker 子级仅允许只读文件访问）。

前台 vs. 后台

Opencode 支持两种执行模式，直接类比于 Claude Code 的 sync/async 划分：

  • Foreground（前台/阻塞）： 父级等待子会话完成后才继续。
  • Background（后台/即发即忘）： 子会话独立运行；父级可以轮询或在完成时接收通知。

OpenClaw —— 基于 ACP 的委派

OpenClaw 使用 Agent Communication Protocol (ACP) 作为其委派传输层，使 sub-agent 能够跨进程和网络边界寻址。

Spawn 参数

在 src/agents/acp-spawn.ts，SpawnAcpParams 定义了 spawn 合约：

// openclaw: src/agents/acp-spawn.ts
interface SpawnAcpParams {
  task: string;              // 子任务描述
  agentId: string;           // 要 spawn 的 agent 类型
  resumeSessionId?: string;  // 恢复现有会话
  model?: string;            // 子级的 model 覆盖
  thinking?: boolean;        // 启用 extended thinking
  sandbox?: SandboxSpec;     // 沙箱配置
  streamTo?: string;         // 将结果流式传输到的频道
}

多 agent 路由

OpenClaw 的 ACP 层为 agent 间通信提供了 24 个 channel。每个 agent 订阅它关心的 channel；ACP 路由器按 channel 订阅投递消息，而不是按 agent 身份。这种解耦意味着 agent 可以在不重新配置路由表的情况下添加、删除或替换。

设计启发

"隔离是定义性维度——sub-agent 能看到什么决定了它能做什么。"
Sub-agent 系统中的每个设计决策都源于隔离边界。进程内克隆（claude-code、codex）速度快且对缓存友好，但需要严格的上下文分离。进程 spawn（pi-mono）在构造上是安全的，但每次委派都要付出 IPC 开销。线程池（hermes-agent）处于中间位置——在 Python GIL 下足够安全，但隔离是约定性的，而非由操作系统强制执行的。
"预算必须分开：sub-agent 不应该耗尽父级的 token 配额。"
Hermes-agent 的每子级 IterationBudget 是对这一原则最清晰的表达。Nanobot 通过 asyncio 协作调度隐式实现了它。Claude Code 依赖父级管理预算（通过 AbortController 链接）。所有代码库中关键的失败模式是相同的：一个不受约束的 sub-agent 在探索死胡同时燃烧 token，而父级除了超时杀死子级之外别无他法。
"Fork 模式（claude-code、codex）用隔离换取缓存效率。"
当子级继承父级的对话前缀（fork_context）时，LLM 提供商可以重用父级推理中已缓存的键值对。这节省了大量的 API 成本——但这意味着子级可以"看到"父级的对话。如果父级正在讨论敏感数据，子级会继承其上下文中的该数据。Fork 是一种优化，而非默认——在缓存节省超过隔离关注时使用它。
