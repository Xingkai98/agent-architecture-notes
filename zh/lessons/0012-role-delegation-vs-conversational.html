第12课：角色委派 vs 会话式编排

对同一个问题——多个 agent 如何协调——的两种根本不同的答案。

课程三：多 Agent 编排 · 2026年6月 · 永久链接

课程一和二是关于单 agent 的。本课问：当你需要多个 agent 时怎么办？两个框架——CrewAI 和 AutoGen（及其后继 MAF）——给出了截然不同的答案。CrewAI 说："管理者分配工作。"AutoGen 说："agent 彼此对话。"两者都有效。区别在于谁控制工作流。

多 agent 系统解决单 agent 难以应对的三个问题：上下文窗口饱和、工具过载和单点故障。

范式 A：角色委派（CrewAI）

CrewAI 将多 agent 协调建模为管理层级。一个 Crew 拥有一组 Task 对象和一个 Agent 名单。Crew 根据角色将任务分配给 agent：

Crew
拥有任务 + agent
→
任务₁：调研
分配给 Agent A（角色：研究员）
→
任务₂：撰写
分配给 Agent B（角色：写作员），上下文来自任务₁
→
任务₃：审阅
分配给 Agent C（角色：审阅员）

Crew（crew.py:159）是顶层编排者。两种流程（process.py）：顺序（任务按顺序执行）或层级（管理者 agent 动态委派任务）。Task（task.py:114）有 description、expected_output、指定的 agent 和可选的 context。每个 agent 有自己的内部 ReAct 循环，但架构重点在 Crew/Task 层，而非单个 agent 循环。

范式 B：会话式编排（AutoGen → MAF）

AutoGen 采用相反的方法：没有中心管理者。agent 通过事件驱动运行时通信，GroupChat 管理器选择谁接着发言：

Runtime
asyncio.Queue 事件循环
→
GroupChat 管理器
select_speaker()——RoundRobin / LLM Selector / Swarm
→
Agent 发言
将响应发布到群组 topic
→
管理器选择下一位
基于对话线程

SingleThreadedAgentRuntime（_single_threaded_agent_runtime.py:149）是一个带有 asyncio.Queue 事件循环的 actor 模型。Agent 通过 send_message()（直接 RPC）和 publish_message()（发布-订阅广播）通信。BaseGroupChat（_base_group_chat.py:40）为每个参与者创建嵌入式运行时并包装在 ChatAgentContainer 中。三种发言者选择策略：RoundRobin、LLM Selector、Swarm。

AutoGen → MAF：演化故事

2025 年 10 月，微软宣布 AutoGen 进入维护模式。其后继 Microsoft Agent Framework (MAF) 融合了 AutoGen 的会话模式与 Semantic Kernel 的企业级基础，并新增了 DAG 工作流。MAF 不仅拥有"agent 对话"，还拥有"图执行"——在单一框架中实现会话和流程两种范式的融合。

维度CrewAIAutoGenMAF
编排模型自上而下（Crew 分配）自下而上（agent 对话）两者兼有（GroupChat + DAG 工作流）
Agent 身份角色 + 目标 + 背景故事Agent 类型 + 订阅Agent 类型 + 技能 + 工作流
通信方式AgentTools 委派send_message / publish_message两者 + A2A 协议
生产状态活跃（~51k stars）维护模式（~57k stars）活跃（后继产品）
设计启发：自上而下更便宜也更易调试。自下而上更灵活但更难预测。除非你的用例确实需要涌现式协调，否则选择自上而下。
