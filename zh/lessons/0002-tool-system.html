第2课：工具系统 — 定义、注册、调度、规范化

定义、注册、调度、规范化 — 每个 agent 工具必经的四个阶段。

2026年6月 · 9个主要源代码库 · 永久链接

每个 agent 中的每个工具都精确地经历四个阶段：必须被定义（让 LLM 知道其 schema）、注册（让 harness 能找到它）、调度（让它实际运行），以及规范化（让结果能反馈回对话中）。各代码库之间的差异全在于它们如何实现这四个阶段。

在第1课中，我们看到每个 agent 都围绕一个循环构建。本课深入到该循环的内部引擎：工具是如何工作的。我们将追踪全部九个代码库中的四个阶段，并识别出划分它们的四种基本设计哲学。

四个阶段

每次工具交互都遵循以下管道：

┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ 定义 │ → │ 注册 │ → │ 调度 │ → │ 规范化 │ │ DEFINE │ │ REGISTER │ │ DISPATCH │ │ NORMALIZE │ │ │ │ │ │ │ │ │ │ Schema、 │ │ 注册表、 │ │ 执行、 │ │ 结果 → │ │ 参数、 │ │ 发现、 │ │ 沙箱、 │ │ LLM 可读 │ │ 描述 │ │ 可用性 │ │ 权限 │ │ 消息 │ └──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘

四种设计哲学

纵观九个代码库，浮现出四种截然不同的哲学：

极致简约 全流程编排
  1. 极简主义： pi-mono、openclaw、nanobot。工具是带有 execute 函数的简单对象。没有中央注册表或发现机制 — 工具存放在数组或字典中。框架本身甚至不提供内置工具；工具由使用者提供。
  2. 类型安全 / 依赖注入： opencode（Effect-TS）。工具使用运行时校验的 schema 定义，通过依赖注入注册到作用域容器中，调度时自动清理。类型系统保证正确性。
  3. 全流程编排： claude-code、codex、hermes-agent。工具经过多阶段管道：权限检查、沙箱选择、执行、重试、结果规范化，以及（claude-code 的）UI 渲染。每个阶段都是独立的关注点，具有可插拔的实现。
  4. 中间件驱动： deepagents、deer-flow。工具不由开发者直接注册 — 它们由中间件贡献。中间件栈中的每个中间件（文件系统、子 Agent、Skills、MCP）暴露各自的工具。LLM 在任何时刻可用的工具是所有中间件贡献的工具加上用户提供的工具的并集。这翻转了所有权模型：工具生命周期由拦截器管理，而非中央注册表。（对照第1课中的架构 D — 中间件栈即是架构，工具所有权遵循相同模式。）

阶段 1：定义 — 工具如何描述自己

Schema 优先（pi-mono、openclaw）

工具实现 AgentTool<TParameters> 接口，该接口扩展了泛型 Tool，增加了 label、execute() 和 executionMode。schema 声明为 TypeScript 泛型 — 类型即是 schema。除了工具自身的 execute() 内部逻辑外，没有运行时 schema 校验。

// pi-mono: packages/agent/src/types.ts:365
export interface AgentTool<TParameters = any, TDetails = any>
  extends Omit<Tool<TParameters>, 'execute'> {
  label: string;
  execute(toolCall, context, options?): Promise<AgentToolResult<TDetails>>;
  executionMode: 'sequential' | 'parallel';
}

// openclaw: packages/agent-core/src/types.ts:438 — 结构相同
export interface AgentTool<TParameters = any, TDetails = any>
  extends Omit<Tool<TParameters>, 'execute'> {
  label: string;
  execute(toolCall, context, options?): Promise<AgentToolResult<TDetails>>;
  executionMode: 'sequential' | 'parallel';
}

Zod 类型对象（claude-code）

Claude Code 工具是实现 Tool<Input, Output> 接口的对象，使用 Zod schema 作为 inputSchema 和 outputSchema。buildTool() 工厂函数为 7 个生命周期方法填充安全的默认值。

// claude-code: src/Tool.ts:783
buildTool({
  name: ToolNames.BASH,
  description: 'Execute a bash command',
  inputSchema: BashInputSchema,  // Zod schema
  async *call(input, context) { /* ... */ },
  // buildTool 为以下提供默认值：
  // isConcurrencySafe、isReadOnly、isEnabled、
  // checkPermissions、validateInput、
  // renderToolUseMessage、renderToolResultMessage
})

JSON Schema 字典（hermes-agent、nanobot）

两个 Python 代码库都使用 JSON Schema 字典。Hermes-agent 的 ToolEntry 存储一个 schema 字典；nanobot 的 Tool ABC 有一个返回 JSON Schema 字典的 parameters 属性。区别在于：nanobot 使用 Python ABC 配合 @tool_parameters 装饰器进行声明式 schema 附加，而 hermes-agent 直接将 schema 字典传给 registry.register()。

# hermes-agent: tools/registry.py:77
class ToolEntry:
  name: str
  toolset: str
  schema: dict            # OpenAI 格式的 JSON Schema
  handler: Callable       # 同步或异步
  check_fn: Callable | None  # 可用性守门

# nanobot: nanobot/agent/tools/base.py:124
class Tool(ABC):
  name: str
  description: str
  parameters: dict        # JSON Schema
  async def execute(self, **kwargs): ...

  @tool_parameters(StringParam("query", "..."))
  async def search(self, query: str): ...

Rust trait 驱动（codex）

Codex 使用 Rust trait：CoreToolRuntime 用于工具处理器，ToolOutput 用于结果，ToolSpec 用于 schema 定义。类型系统在编译时强制保证正确性。

// codex: codex-rs/core/src/tools/registry.rs:47
pub trait CoreToolRuntime: Send + Sync {
  fn apply_patch(&self, input) -> ...;
  fn shell(&self, input) -> ...;
  fn exec(&self, input) -> ...;
  fn mcp_dynamic_tool(&self, input) -> ...;
  fn view_image(&self, input) -> ...;
  // ... 30+ 个方法，每个工具处理器一个
}

阶段 2：注册 — 工具如何被发现

注册策略与所选语言和框架高度相关：

无注册表（pi-mono、openclaw）： 工具以 AgentTool[] 数组的形式传递到 AgentContext 上。使用者完全拥有工具的生命周期。这是最简单的方法 — 也最灵活，因为没有框架需要对抗。

pi-mono 的方法："原语而非特性"。框架不提供任何内置工具。四个工具（read、write、edit、bash）定义在应用层，而非框架层。

扁平数组配合条件包含（claude-code）： src/tools.ts:193 中的 getAllBaseTools() 返回包含 30+ 个工具的扁平数组。工具基于 feature flag（REPLTool、WebBrowserTool）和权限上下文进行条件包含。MCP 工具通过 assembleToolPool()（第 345 行）合入，并进行去重。

单例注册表配合 AST 发现（hermes-agent）： tools/registry.py:151 中的 ToolRegistry 单例存储所有工具。discover_builtin_tools()（第 57 行）中的发现逻辑用 ast 模块扫描 tools/*.py — 它在导入模块之前先解析 Python 源码来查找 registry.register(...) 调用。这避免了导入依赖不可用（Docker、Playwright 等）的工具时产生的副作用。

包扫描（nanobot）： nanobot/agent/tools/loader.py:21 中的 ToolLoader 使用 pkgutil.iter_modules 发现 Tool 子类，并通过 entry_points(group="nanobot.tools") 支持插件发现。_plugin_discoverable 类属性可防止抽象基类被自动发现。

每轮构建（codex）： spec_plan.rs 中的 build_tool_router() 每轮重新构建 ToolRouter。它整合了 MCP 工具、动态工具（运行时发现）、扩展工具和标准工具。路由器将 ToolName 映射到 ToolRegistry 条目。

依赖注入 + 作用域（opencode）： registry.ts 中的 ToolRegistry.Service 使用 Effect 的 Layer/Scope 进行依赖注入。工具注册到作用域容器中，在作用域退出时自动清理。materialize() 方法为 LLM 快照工具定义。

中间件驱动工具（deepagents、deer-flow）

这两个代码库中出现了一种新范式：工具不由开发者声明并注册到中央列表中。相反，每个中间件贡献自己的工具，最终的工具列表是所有中间件贡献加上用户提供的工具的并集。

deepagents： graph.py:236 中的 create_deep_agent() 组装一个中间件栈，每个中间件暴露工具：TodoListMiddleware → write_todos，FilesystemMiddleware → ls/read_file/write_file/edit_file/glob/grep/execute，SubAgentMiddleware → task。middleware/_tool_exclusion.py 中的 _ToolExclusionMiddleware 可根据 HarnessProfile.excluded_tools 过滤工具。工具描述可以在不修改工具对象的情况下覆盖，通过 _tools.py:30-65 中的 _apply_tool_description_overrides() 实现。

deer-flow： tools/tools.py:44 中的 get_available_tools() 按严格的优先级顺序组装工具：配置定义的工具 → 内置工具（present_files、ask_clarification）→ 子 agent task 工具 → vision 工具 → MCP 工具（惰性初始化，用 mtime 做缓存）→ ACP 工具 → skill 管理。DeferredToolFilterMiddleware 对 LLM 隐藏 MCP 工具 schema，直到 tool_search 提升触发它们 — 这是一种独特的"惰性工具加载"模式，当 MCP 服务器暴露数百个工具时，可保持上下文窗口精简。

这种中间件驱动的方法翻转了控制权：工具可用性不再是配置好注册表再执行，而是成为中间件栈当前状态的函数。skill 激活可以添加工具，权限变更可以移除工具。工具系统是动态的，而非静态的。

阶段 3：调度 — 工具如何执行

执行时机

如第1课所述，分为两大阵营：

  • 批量（流结束后）： pi-mono、openclaw、nanobot、hermes-agent — 等待 LLM 流完成后，收集工具调用，一起执行
  • 流式（流进行中）： claude-code — StreamingToolExecutor 在 LLM 逐个输出时即执行工具
  • 混合： codex — FuturesOrdered 在流式传输过程中排空进行中的工具 future；opencode — 可配置

并发模型

每个代码库都有“哪些工具可以一起运行”的概念：

代码库并发模型实现
pi-mono顺序或并行（Promise.all）executeToolCallsSequential / executeToolCallsParallel
openclaw顺序或并行（Promise.all）与 pi-mono 相同模式，每个工具有 executionMode
claude-code并发安全工具并行，其他串行StreamingToolExecutor 按 isConcurrencySafe() 分区
hermes-agent顺序（同步上下文 + 异步桥接）_run_async() 使用每线程 event loop 处理协程
nanobot顺序，带 concurrency_safe 标志Tool.concurrency_safe / Tool.exclusive 属性
codex通过 Tokio + FuturesOrdered 并行tool_supports_parallel() 控制并发执行
opencode通过 Effect fiber 顺序 settle每次 settle() 调用是其自身 Effect
deepagents通过 ToolNode 顺序执行LangGraph 的 ToolNode 按顺序执行工具；中间件可通过 wrap_tool_call 包装
deer-flow顺序 + 并发批次_execute_concurrent_tool_calls() 用于安全工具；SubAgentLimitMiddleware 上限为 3

沙箱

只有两个代码库有真正的沙箱方案：

codex： ToolOrchestrator（orchestrator.rs）提供多层沙箱：Landlock（Linux）、Windows 沙箱、macOS 沙箱、提升/受限 token。当沙箱拒绝时，编排器可以以提升的沙箱级别重试 — 如果受限沙箱阻止了文件写入，工具可以用提升的沙箱重试。网络访问审批是一个一等概念，支持立即和延迟两种模式。

deer-flow： 第二个有真正沙箱方案的代码库。sandbox/sandbox_provider.py 中的 SandboxProvider 模式抽象了跨 Local、Docker 和 AioSandbox 后端的执行。工具看到的是虚拟路径（/mnt/user-data/workspace），这些路径被转换为宿主机路径。沙箱工具（bash、ls、read_file、write_file、glob、grep、str_replace）经过虚拟路径转换（validate_local_tool_path()）和路径遍历检测。写入大小上限 80KB。这是沙箱化作为一等架构维度，而非事后补救 — 并且它被集成到工具系统之中，而非包裹在外部。

所有其他代码库将沙箱化委托给外部机制：pi-mono/openclaw 使用 beforeToolCall 钩子，hermes-agent 使用基于环境的终端，claude-code 使用工具访问的权限检查（而非执行沙箱），deepagents 委托给后端协议（SandboxBackendProtocol）。

阶段 4：规范化 — 结果如何反馈

执行后，工具结果必须变成 LLM 可以消费的消息。这是分歧最大的环节：

大小截断（全部）： 每个代码库都有最大结果大小限制。Claude Code 的 applyToolResultBudget() 将大结果持久化到磁盘，替换为截断后的预览。Hermes-agent 的 max_result_size_chars 按工具截断。Nanobot 将 read_file 排除在卸载之外，以防止“持久化到磁盘 → 从磁盘读取”的循环。

错误规范化（全部）： 所有代码库都捕获工具异常并将其转换为 LLM 可以理解的错误消息。Hermes-agent 和 nanobot 返回 {"error": "..."} JSON 字符串。pi-mono/openclaw 创建带 isError: true 的 ToolResultMessage 对象。Codex 使用 FunctionCallError 变体。

进度流式传输（pi-mono、openclaw）： AgentToolResult 可以包含一个 progress 回调用于部分流式传输，允许 UI 在完成前显示增量的工具输出。OpenClaw 在 pi-mono 的基础上添加了这一特性。

逐工具渲染（claude-code）： 独特之处在于，Claude Code 工具有 7+ 个渲染方法（renderToolUseMessage、renderToolResultMessage、renderToolUseProgressMessage 等）— 工具结果不仅是给 LLM 的消息，还是 TUI 的 UI 产物。

核心权衡：复杂度预算

工具系统是各代码库分歧最大的地方。pi-mono 有零行工具注册表代码。Codex 有一整套 ToolOrchestrator，包含审批、沙箱选择和重试升级。为什么？

更多的工具基础设施 = 更多的安全保障，但需要维护更多代码。 Codex 的沙箱重试逻辑意味着由于受限沙箱的权限拒绝而失败的工具可以自动以提升的权限重试 — 用户无需重新运行。但这也意味着一个多阶段管道，仅工具编排就需要 10+ 个文件。

更少的工具基础设施 = 更大的灵活性，但消费者负担更重。 pi-mono 给你 AgentTool[] 然后说“开始吧”。你可以实现任何你想要的工具系统。但你必须自己实现 — 框架不会帮你做权限检查、沙箱、结果规范化或错误恢复。

设计启发： 如果你控制执行环境且信任你的工具，从极简开始 — 一个包含 execute 函数的对象数组。如果你需要沙箱隔离、权限守门和可观测性，编排式方法（codex、claude-code）是值得的 — 但要预见到工具系统将成为你代码库中最大的子系统。
