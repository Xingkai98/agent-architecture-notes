第11课：从 AgentExecutor 到中间件协议

LangChain 的 agent 系统如何从回调驱动的 while 循环演化为基于 LangGraph 的可组合中间件栈。

课程二：Agent 基础设施 · 2026年6月 · 永久链接

LangChain 仓库中包含两代 agent 系统。经典的 AgentExecutor（已弃用）是一个回调驱动的 while 循环——和课程一中 pi-mono 和 hermes-agent 的模式相同。V1 的 create_agent（当前）是一个编译到 LangGraph StateGraph 的中间件栈——和 deepagents 与 deer-flow 所扩展的模式相同。本课追踪这一演化并提炼架构教训：回调耦合，中间件组合。

两套系统共存于同一仓库这一事实很不寻常且很有价值——它是从命令式 agent 循环到声明式中间件栈的架构转型的化石记录。

AgentExecutor：while 循环时代

AgentExecutor 类位于 langchain_classic/agents/agent.py:1012。_call() 方法（第 1570 行）不断迭代，直到 AgentFinish 或 max_iterations（默认 15）。每次迭代调用 agent.plan() 获取下一个 AgentAction，然后 _perform_agent_action()（第 1380 行）通过 tool.run() 执行工具。

检测是基于回调的：CallbackManager 在固定点触发 on_agent_action、on_agent_finish、on_tool_start、on_tool_end。回调是观察者，而非参与者。如果想添加重试逻辑、上下文摘要或权限检查，必须修改 executor 本身或从外部包装它。

create_agent：图时代

V1 的 create_agent() 位于 factory.py:787，构建一个带有"model"和"tools"两个节点并由条件边连接的 StateGraph。递归限制为 9999（而 AgentExecutor 为 15）。真正的架构创新不是图本身，而是中间件 hook 如何成为图节点。

6 个 hook 的 AgentMiddleware 协议

AgentMiddleware 位于 middleware/types.py:383，定义了六个 hook 点：

Hook触发时机用途
before_agent循环开始前一次设置、加载状态
before_model每次 LLM 调用前修改请求、注入上下文
after_model每次 LLM 调用后检查响应、检测工具调用
after_agent循环结束后一次清理、保存状态
wrap_model_call包装 LLM 调用重试、回退、重写请求
wrap_tool_call包装工具执行重试、监控、修改输入/输出

与回调的关键区别：wrap hook 可以拦截。wrap_model_call 处理器接收 ModelRequest 和一个调用实际模型的 handler 回调。中间件可以在传递给 handler 之前修改请求、捕获异常并重试，甚至完全跳过 handler。这是"观察执行"和"参与执行"的区别。

在编译后的图中，before_agent、before_model、after_model 和 after_agent 成为独立的图节点。而 wrap_model_call 和 wrap_tool_call 通过 _chain_model_call_handlers()（factory.py:234）组合成嵌套的洋葱层——列表中的第一个是最外层。

LangChain V1 在 middleware/ 下提供了 15+ 个内置中间件：SummarizationMiddleware（自动 compact）、TodoListMiddleware（暴露 write_todos 工具）、HumanInTheLoopMiddleware、ModelRetryMiddleware、ModelFallbackMiddleware、ToolRetryMiddleware、ShellToolMiddleware（沙箱执行）、PIIMiddleware 等。每个中间件都是独立的——一个文件，100-300 行，实现一两个 hook。这种模块化正是 deepagents 和 deer-flow 继承和扩展的东西。
