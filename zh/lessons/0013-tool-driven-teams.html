第13课：工具驱动团队——AgentScope

当团队管理只是另一个工具调用——AgentScope 对多 agent 协调的元认知方法。

课程三：多 Agent 编排 · 2026年6月 · 永久链接

第12课讲解了多 agent 协调的两种方法。AgentScope（阿里巴巴，~23k stars）引入第三种范式：团队管理作为 agent 工具。主 agent 使用与文件操作和网络搜索相同的工具调用接口来创建、通信和销毁团队成员。这是一种元认知方法——agent 像推理其他任务一样推理团队结构。

AgentScope 在三个多 agent 框架中也是独一无二的——它同时是一个完整的单 agent harness，可以在独立 agent 和递归多 agent 系统之间无缝切换。

团队即工具：核心洞察

在 CrewAI 中，团队管理是一个独立的编排层（Crew 类）。在 AutoGen 中，它是一个独立的运行时（GroupChat）。在 AgentScope 中，它就是工具：

TeamCreate
创建团队，调用者成为 leader
→
AgentCreate
生成带有独立 session、toolkit、workspace 的 worker
→
TeamSay
通过 MessageBus 发送消息
→
TeamDelete
解散团队

TeamCreate（_tools/_team_create.py:30）创建持久化团队记录。AgentCreate（_tools/_agent_create.py:136）使用 SubAgentTemplate（_types.py:29）生成完整的 worker agent——新的 session、toolkit、permission context、workspace。leader 通过 prompt 参数分配工作。这种倒转的控制模型意味着agent 通过工具管理自己的团队，而不是由框架管理 agent。

MessageBus：通信基础设施

MessageBus（message_bus/_base.py:49）三种传输模式：Drain queue（单消费者，读取确认，用于每 session 收件箱）、Replay log（多消费者，游标跟踪，用于 SSE 事件重放）、Transient broadcast（即发即忘，用于唤醒信号）。InboxMiddleware（app/middleware/_inbox_middleware.py:24）在每次推理步骤前清空收件箱，将 HintBlock 对象注入 agent 上下文。

三种范式对比

维度CrewAIAutoGen/MAFAgentScope
协调方式角色委派对话工具调用
控制流自上而下（Crew）自下而上（GroupChat）元认知（agent 决定）
通信方式AgentTools 委派pub/sub 通过运行时MessageBus（3 种模式）
单 agent 能力是（基础 ReAct）是（AssistantAgent）是（完整 harness）
Worker 隔离共享内存运行时范围Session + Workspace
独特贡献Memory（RecallFlow）Actor 运行时团队即工具
设计启发：团队结构已知时用 CrewAI（更简单）。结构从对话中涌现时用 AutoGen/MAF。当 agent 应该自己决定团队结构时用 AgentScope。元认知模型最强大——也最难预测。
