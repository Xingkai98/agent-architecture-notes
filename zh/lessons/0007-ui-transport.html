第7课：UI 与传输层

TUI 框架、客户端/服务端分离以及多频道适配器 —— agent 如何与外界对话。

2026年6月 · 7 个主要源代码库 · 固定链接

每个 agent 都需要一个界面。对某些 agent 来说，这个界面是终端（Claude Code、Pi-mono、Codex、Opencode）。对另一些来说，是消息应用（OpenClaw 支持 24 个频道，Nanobot 支持 18 个）。再其他的则是桌面应用或 API 服务器。传输层的选择塑造了其上的一切：工具如何渲染结果、流式传输如何工作、错误如何呈现给用户。

在第2课中，我们了解了工具是如何定义、注册、分发和规范化的。本课向上移动一层：这些工具结果如何到达用户，以及用户的输入如何到达 agent。我们将纵览完整的界面范式频谱，详细比较六种 TUI 实现，分析客户端/服务端架构，并描绘多频道领域的地图。

界面范式频谱

Agent 以不断增加的复杂性和覆盖范围向用户暴露自身：

REPL/TUI API
  1. REPL / TUI： Agent 是一个终端应用程序。用户输入，agent 响应，循环是可见的。Claude Code、Pi-mono、Codex 和 Opencode 都属于此类。这是面向开发者的 agent 的主导范式。
  2. CLI 单次执行： Agent 运行单轮并退出。输入来自参数或 stdin，输出到 stdout。适用于脚本和 CI。Codex 和 Pi-mono 都支持此模式。
  3. Daemon： Agent 作为长期运行的进程，通常无头运行。Codex 的 app-server 和 OpenClaw 的 gateway 都以这种方式运作——agent 进程与界面分离。
  4. Multi-channel（多频道）： 一个 agent，多个界面。OpenClaw（24 个频道）和 Nanobot（18 个频道）是此类的典范。单个 agent 循环可以同时接收来自 Telegram、Discord、Slack、飞书和 Matrix 的输入。
  5. API： Agent 暴露一个 HTTP 端点（通常兼容 OpenAI /v1/chat/completions）。Nanobot 的 api/server.py 是最清晰的示例——一个薄的 aiohttp 封装，允许任何 HTTP 客户端驱动 agent。

TUI 技术对比

六个代码库实现了终端 UI，但它们在技术选择上截然不同。框架的选择不仅仅是实现细节——它决定了流式输出如何渲染、滚动如何工作、按键绑定如何定义，以及项目承担了多少维护负担。

Claude Code：自定义 React + Ink fork

Claude Code 的 TUI 建立在 Ink（React 的终端版本）的自定义 fork 之上。该 fork 位于 src/ink/，包含一个 251KB 的 ink.tsx，包括自定义 fiber reconciler 和终端布局引擎。两个最大的组件——Messages.tsx（147KB）和 VirtualMessageList.tsx（148KB）——展示了一个终端 agent 积累了多么庞大的 UI 逻辑。虚拟滚动是必需的：agent 对话产生数百条消息，朴素地渲染所有消息会使终端不堪重负。

Ink fork 是一个多年的承诺。Claude Code 无法简单升级到上游 Ink——该 fork 为了支持自定义布局、虚拟滚动和流式工具渲染而产生了分歧。这是"TUI 即应用"理念的代价。

Codex：ratatui + crossterm

Codex 使用 Rust 原生终端库：ratatui 用于基于 widget 的渲染，crossterm 用于终端控制。主 TUI 位于 codex-rs/tui/src/tui.rs 和 app.rs（58KB）。Ratatui 提供保留模式的 widget 树和布局约束——这是一种不同于 React 声明式 diffing 的范式。Crossterm 处理原始模式、光标控制和输入事件。

Pi-mono：从零开始的自定义 TUI

Pi-mono 在 packages/tui/src/tui.ts（53KB）中的 TUI 共有零个框架依赖。它从零开始实现组件：box、editor、image、input、loader 和 markdown。没有 Ink、没有 ratatui、没有 prompt_toolkit——只有原始的 ANSI 转义序列和终端 IO。这给出了最大的控制和最小的膨胀，但代价是需要重新实现每个 widget 行为。

Hermes-agent：prompt_toolkit

Hermes-agent 在 cli.py 中使用 Python 的 prompt_toolkit，组合了 Application、Layout、TextArea 和 KeyBindings。Prompt_toolkit 开箱即用地提供了成熟的 widget 系统，支持语法高亮、自动补全和按键绑定 DSL——无需从零开始构建。

Opencode：OpenTUI + Solid.js

Opencode 在 packages/tui/src/app.tsx（36KB）中的 TUI 建立在 OpenTUI 之上，配合 Solid.js 实现响应式渲染。这是一种新颖的组合：终端 UI 配合细粒度响应式 JSX。Solid 基于信号（signal）的响应式意味着终端输出只有变化的部分会重新渲染，无需虚拟 DOM。

Nanobot：prompt_toolkit + rich

Nanobot 在 cli/commands.py（67KB）中将 prompt_toolkit 用于交互式 CLI，配合 rich 进行格式化输出。Rich 提供表格、面板、语法高亮代码块和 markdown 渲染——全部针对终端输出样式设计。

客户端/服务端分离

三个代码库将 agent 进程与 UI 进程分离——这一设计选择使得远程操作、无头部署和多前端成为可能：

Opencode：Web Worker 服务端 + RPC + TUI 客户端

Opencode 在 Web Worker 中运行 agent 核心，通过 RPC 与 TUI 客户端通信。TUI 可以连接到本地 worker（进程内）或远程服务器（通过 HTTP）。这意味着同一个 agent 可以驱动终端 UI、Web UI 或 IDE 插件——全部通过相同的 RPC 接口。

Codex：App Server + TUI 客户端

Codex 分为 app-server/（agent 进程）和 app-server-client/（TUI）。两者可以在进程内运行（零网络开销），也可以让 TUI 通过 HTTP 连接到远程 app-server。这与 language server 的模式相同：服务器做工作，客户端处理显示。

OpenClaw：Gateway 协议

OpenClaw 在 packages/gateway-protocol/ 中定义了一个正式的 gateway 协议，并配有对应的 gateway-client/ 包。该协议规定了消息格式、认证和传输——任何能够使用该协议的客户端都可以驱动 agent。这比 Codex 的即兴客户端/服务端分离更进一步：协议是版本化的、文档化的和可复用的。

Multi-channel（消息机器人）

两个代码库远远超越了终端 UI——它们首先是消息平台，其次才是 agent：

OpenClaw：24 个频道

OpenClaw 支持 24 个频道：Discord、Slack、Telegram、飞书、Matrix、WhatsApp、Signal、IRC、iMessage、LINE、Zalo 等。频道在每个扩展的 package.json 中声明式注册。频道插件合约为以下方面标准化了适配器：配置、认证、出站消息发送、连接状态、设备配对和安全策略。构建时的频道发现会生成 dist/channel-catalog.json——一份所有可用频道的清单，无需运行时扫描。

关键的架构洞察：每个频道是一个适配器，而非一个 agent。只有一个 agent 循环。来自所有频道的消息通过统一的消息总线路由。agent 不知道也不关心消息来自哪个频道。

Nanobot：18 个频道

Nanobot 支持 18 个频道，每个都有可观的实现：Telegram（64KB）、飞书（78KB）、Discord（33KB）、Slack（30KB）、Signal（55KB）、Matrix（43KB）、微信（63KB）等。所有频道遵循带有三个核心方法的 BaseChannel ABC：start()、stop() 和 send()。消息总线处理频道与 agent 核心之间的发布/订阅路由。

# Nanobot 频道合约（概念性的）：每个频道实现这三个方法
class BaseChannel(ABC):
    async def start(self) -> None: ...
    async def stop(self) -> None: ...
    async def send(self, target, message) -> None: ...

# 每个频道是适配器——agent 循环从不导入 Telegram 或 Discord。
# 消息以标准化类型到达总线；总线将其路由到 agent。
# 响应通过总线返回；频道适配器为其平台格式化它们。

Hermes-agent：桌面 + CLI

Hermes-agent 采取了不同的方法：一个桌面 Electron 应用位于 apps/desktop/，使用 React + Vite 构建，加上一个 cli.py 用于终端使用。桌面应用提供图形化聊天界面，而 CLI 服务于脚本和高级用户。这是整个集合中唯一提供 GUI 的代码库。

单一界面 agent

四个代码库坚持使用单一界面——没有频道、没有桌面应用、没有 API 服务器：

  • Claude Code： 仅终端（React+Ink）。支持结构化 IO 用于 IDE 集成（VS Code、JetBrains），但 agent 本身是一个终端应用。
  • Pi-mono： 仅终端（自定义 TUI）。无 API、无频道、无桌面。终端是全部产品界面。
  • Codex： 仅终端（ratatui）配合远程控制 daemon。daemon 支持无头操作，但用户交互通过 TUI 进行。
  • Opencode： 仅终端（OpenTUI）。Web Worker 架构允许 Web UI，但主要界面是终端。

值得关注的设计选择

四个值得研究的实现细节：

Claude Code 的 Ink fork： TUI 就是应用程序本身。Messages.tsx 的 147KB 不是一个"视图层"——它是核心用户体验。工具结果、流式文本、权限提示和模式指示器全部通过 React 组件渲染。该 fork 的存在是因为没有现成的终端 UI 库能以这种规模处理流式 agent 对话。

Pi-mono 的从零开始 TUI： 仅 53KB 且零依赖，Pi-mono 的 TUI 是一种声明。没有框架变动，没有升级跑步机，没有来自上游的破坏性变更。代价是每个功能——语法高亮、回滚、调整大小处理——都必须自行构建和维护。对于重视极简主义的项目，这种权衡是有意义的。

OpenClaw 的频道目录： dist/channel-catalog.json 在构建时从扩展清单生成。这意味着 agent 在启动前就知道其可用的频道——无需运行时扫描、无动态导入、无竞争条件。这是一个小小的细节，却消除了一整类启动错误。

Nanobot 的 OpenAI 兼容 API 服务器： api/server.py 是一个薄的 aiohttp 封装，暴露了 /v1/chat/completions。这意味着任何可以调用 OpenAI API 的工具——langchain、openai-python、curl——都可以驱动 Nanobot，而不需要知道它不是在跟 OpenAI 对话。agent 成为了一个可直接替换的替代方案。

设计启发

启发 1： 终端 agent 需要虚拟滚动——消息无限增长。Claude Code 的 VirtualMessageList（148KB）存在的原因是朴素渲染 500+ 条消息会使终端冻结。每个 TUI agent 最终都会遇到这个问题，无论使用什么框架。
启发 2： 多频道 agent 需要统一的消息总线——每个频道是适配器，而非 agent。OpenClaw 和 Nanobot 都独立地收敛到这个模式。agent 循环处理一种消息类型；频道适配器在它与平台之间进行转换。不要为每个频道编写一个 agent。
启发 3： TUI 框架的选择是多年的承诺——自定义 fork 给你控制权但消耗维护成本。Claude Code 的 Ink fork 无法跟踪上游。Pi-mono 的从零开始 TUI 需要手工构建每个 widget。Ratatui 升级需要 Rust 重构。请仔细选择你维护负担的界面。

我们没有涵盖的内容

  • Web UI 实现 —— 一些代码库有 Web 仪表板，但没有作为主要界面进行深入分析。
  • 流式传输协议 —— server-sent events、WebSocket 和 chunked transfer encoding 如何将 agent 输出传输给客户端。部分涵盖在第1课（流式 vs. 批处理）。
  • IDE 集成细节 —— Claude Code 的结构化 IO 模式、Codex 的 VS Code 扩展和 Opencode 的编辑器插件内部如何工作。
  • 移动界面 —— 七个代码库都没有发布移动应用，尽管消息频道（Telegram、WhatsApp）提供了事实上的移动访问。

  • 如果你想要了解终端的虚拟滚动：追踪 Claude Code: src/components/VirtualMessageList.tsx —— 这个 148KB 的文件是在终端中仅渲染可见消息的最终实现。
