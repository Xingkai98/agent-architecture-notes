第3课：上下文管理

AI agent 如何管理其上下文窗口 — compact、token 预算以及树模型。

2026年6月 · 9个主要源代码库 · 永久链接

上下文窗口是每个 agent 的最严苛资源约束。上下文太少则 agent 忘记它在做什么；太多则 LLM 变慢、变贵、变混乱。每个代码库都不同地解决这个问题，但它们面对同一个难题：如何保留重要的、丢弃次要的。

在第2课中，我们涵盖了从工具定义到规范化的全过程。本课深入相反的方向：agent 如何将工具结果和对话历史压缩回上下文窗口的容量之内。

四种基本策略

纵观九个代码库，浮现出四种截然不同的 compact 策略：

主动式（调用之前） 被动式（错误之后）
  1. 主动 compact（claude-code、hermes-agent）： 在每次 API 调用之前检查 token 数量。如果超过阈值，先 compact。这是最安全的方式 — LLM 永远不会看到超载的上下文。
  2. 被动 compact（pi-mono、openclaw、hermes-agent）： 等待提供商返回 413 prompt-too-long 错误，然后 compact 并重试。实现更简单，但浪费一次 API 调用。
  3. 每轮合并（nanobot）： 每轮运行一次轻量级合并检查；如果需要，在继续之前运行最多 5 轮摘要。
  4. 委托给提供商（opencode、codex）： 将 compact 留给外部包（@opencode-ai/core）或可配置的 shell 钩子。没有内置的 LLM 摘要器。

最复杂的方案：claude-code 的七层架构

Claude Code 拥有九个代码库中最精密的上下文管理。七个不同层次协同工作：

第一层：Token 预算（守门人）

每次 API 调用前，src/query/tokenBudget.ts 检查对话是否在容量之内。阈值是上下文窗口的 90%（COMPLETION_THRESHOLD = 0.9）。它还检测收益递减：如果连续 3 次或更多续接各自增加的新 token 不足 500，系统将强制 compact — 对话正在原地打转而没有进展。

收益递减启发式虽微妙但重要。没有它，一个不断对同一文件做微小编辑的 agent 仅凭 token 计数永远不会触发 compact — 每轮增加很少。但经过 10+ 轮后，上下文中充满了近乎重复的工具结果。该启发式捕获了这一模式："你没有在添加有意义的新信息，所以现在就 compact。"

// src/query/tokenBudget.ts — 阈值和收益递减检查
const COMPLETION_THRESHOLD = 0.9;

function isDiminishingReturns(continuations: Continuation[]): boolean {
  return continuations.length >= 3 &&
    continuations.slice(-3).every(c => c.tokenDelta < 500);
}

第二层：上下文窗口解析

src/utils/context.ts — getContextWindowForModel() 解析当前模型的上下文窗口。它检查 [1m] 后缀（表示百万 token 模型），查询模型能力，并默认回退到 200K token。这是所有其他层引用的基础。

第三层：Token 计数

src/utils/tokens.ts — tokenCountWithEstimation() 从最后一个已知的 API 使用点向后遍历。此后的所有内容为估算值，而非精确计数。这提供了快速、合理准确的计数，而无需从头重新计算所有内容。

第四层：自动 compact（主动）

src/query.ts:454 — 每次 API 调用前，系统检查 token 阈值。如果超出，它调用 buildPostCompactMessages()，运行摘要并生成压缩后的对话。这是主要的 compact 路径。

第五层：被动 compact

src/services/compact/reactiveCompact.ts — 如果自动 compact 因某种原因错过了触发（竞态条件、意外的 token 激增），被动层捕获 413 prompt-too-long 错误。它运行一次性的 compact 并重试。没有重试循环 — 如果被动 compact 未能充分减少，错误会被传播出去。

第六层：微 compact

src/services/compact/microCompact.ts — 在完整 compact 运行之前，微 compact 会折叠连续的相同工具结果。如果一个工具被调用了 5 次且结果相同，则仅保留 1 条记录。这通常足以减少 token 数量，从而避免完整摘要。

第七层：上下文折叠、会话记忆及 snip

三种专门的 compact 类型：

  • 上下文折叠（src/services/contextCollapse/）：使用 commit-log 模型生成投影读取时视图 — 每次折叠是状态的"提交"，LLM 看到摘要而非原始历史。
  • 会话记忆 compact（src/services/compact/sessionMemoryCompact.ts）：实验性功能，使用会话记忆数据生成更高质量的摘要。
  • Snip compact（src/query.ts:401）：在对话发送前移除隐藏或有意 snip 的消息内容。

系统提示词组合

src/utils/systemPrompt.ts — buildEffectiveSystemPrompt() 按优先级从 5 个源合并提示词：override > coordinator > agent > custom > default。每一层都可以添加、覆盖或删除段。这本身不是 compact，但它控制每次对话顶部的固定 token 开销。

为什么这很重要：系统提示词是位于每次 API 调用顶部的固定开销。在 200K 窗口、90% 阈值下，10K token 的系统提示词仅留下 170K 用于实际对话 — 系统提示词永久消耗可用预算的 5%。分层组合配合 override 允许不同 agent 模式（coordinator、sub-agent、standalone）调整这一开销。

会话树模型（pi-mono、openclaw）

pi-mono 和 openclaw 共享一个基本不同的数据模型：历史是一棵由类型化条目组成的树，而非扁平数组。

// 简化的树结构
type SessionEntry =
  | { type: 'message', role, content }
  | { type: 'compaction', summary, entriesCompacted }
  | { type: 'branch_summary', branchId, summary }

// buildSessionContext() 遍历树 → 扁平消息列表
// beforeBranch → ... → compaction → afterCompaction → ... → current

该模型的关键属性：

  • compact 被记录，而非销毁： 已 compact 的条目保留在树中。它们只是不包含在活跃上下文中。如果需要，可以"取消 compact"。
  • CompactionSettings：{ enabled: true, reserveTokens: 16384, keepRecentTokens: 20000 } — 始终为 LLM 响应和最近的几轮对话保留空间。
  • 跨轮 compact： 当 token 截断点落在工具调用轮的中间时（部分工具已返回，部分未返回），该轮的前缀和后缀都会被分别摘要。模型永远不会看到半完成的工具周期。
  • 文件操作追踪： 文件操作（读取、写入、编辑）跨会话追踪并附加到 compact 条目中，使摘要知道哪些文件被接触过。
  • 增量摘要更新： UPDATE_SUMMARIZATION_PROMPT 将新消息合并到现有摘要中，而不是从头重新摘要。这更经济且保持连续性。

三层记忆上下文（hermes-agent）

Hermes-agent 将三个来源组合为单一上下文：

┌─────────────┐  ┌──────────┐  ┌──────────────┐
│  MEMORY.md   │  │ USER.md  │  │  SQLite FTS5 │
│  （项目）     │  │ （用户）  │  │  （搜索）     │
└──────┬───────┘  └────┬─────┘  └──────┬───────┘
       └───────────────┼───────────────┘
                       ▼
              ContextCompressor

agent/context_compressor.py 中的 ContextCompressor 运行一个五阶段算法：

┌─────────┐ ┌─────────────┐ ┌──────────────┐ ┌─────────────┐ ┌──────────────┐ │ 阶段 1 │ → │ 阶段 2 │ → │ 阶段 3 │ → │ 阶段 4 │ → │ 阶段 5 │ │ 修剪 │ │ 边界判定 │ │ 既有摘要 │ │ LLM 摘要 │ │ 组装 │ │ │ │ │ │ │ │ │ │ │ │ 修剪冗长 │ │ 哪些消息 │ │ 更新 vs. │ │ 生成 │ │ 前缀 + │ │ 工具输出 │ │ 保留原始 vs.│ │ 重新创建 │ │ 简洁 │ │ 摘要 + │ │ │ │ 被摘要 │ │ 摘要 │ │ 摘要 │ │ 最近消息 │ └─────────┘ └─────────────┘ └──────────────┘ └─────────────┘ └──────────────┘
  1. 修剪工具结果： 在一切之前先修剪冗长的工具输出。这是最便宜的缩减方式，通常自身就足够了。
  2. 确定边界： 找到截断点 — 哪些消息保留为原始文本，哪些被摘要。
  3. 搜索既有摘要： 如果摘要已经存在，则检索它，以便压缩器可以更新而非重新创建。
  4. 生成 LLM 摘要： 调用 LLM 生成已修剪条目的简明摘要。
  5. 组装： 将系统提示词 + 摘要（参考前缀）+ 最近消息组合成最终上下文。

hermes-agent 方案的关键启发式：

  • 防抖动： should_compress() 在最近 2 次压缩各自节省的 token 都不足 10% 时跳过 compact。compact 是有成本的 — 如果效果不好，就停止。
  • 参考前缀： 摘要带有明确的指令前缀，说明它是背景信息，而非活跃指令。这防止模型将陈旧摘要当作命令处理。
  • 回退确定性摘要： 如果 LLM 摘要调用失败，压缩器回退到基于规则的摘要（提取每条消息的首句，上限 8K 字符）。
  • 预检查压缩： 在工具循环开始之前而非期间运行。最多 3 轮。这意味着 agent 永远不会以超载的上下文开始一轮。

带硬限制的文件引用系统（hermes-agent）

Hermes-agent 的 @reference 系统支持 @file:、@folder:、@git: 和 @url: 指令。引用会展开为其内容，但受制于上下文窗口 50% 的硬限制 — 没有任何引用可以消耗超过一半的可用上下文。这防止单个大文件挤占对话历史。

50% 的限制并非随意设定。它保证对话历史始终至少与注入引用有同样多的空间，这意味着模型可以始终以同等权重的历史上下文来"思考"注入的内容。90% 的限制会让引用占据主导；20% 的限制会让引用对大型文件变得无用。50% 是平衡的分割点。

@folder: 指令是四种指令中最危险的一个 — 递归展开目录树可能拉入数千个文件。Hermes-agent 通过先按文件大小估计展开条目，按相关性排序（使用 FTS5 索引），并在 50% 预算耗尽时停止来处理。

显式状态机（nanobot）

Nanobot 将上下文管理作为其轮次循环中的一等状态：

// nanobot/agent/loop.py — TurnState 枚举
RESTORE → COMPACT → COMMAND → BUILD → RUN → SAVE → RESPOND → DONE

每个组件有明确的角色：

  • AutoCompact（nanobot/agent/autocompact.py）：基于空闲会话 TTL 的归档。如果会话闲置，保留最后 8 条消息并摘要其余部分。这与每轮合并正交 — 它处理用户离开数小时后再回来的情况。
  • Consolidator（nanobot/agent/memory.py）：maybe_consolidate_by_tokens() 每轮运行。如果上下文超出预算，最多运行 5 轮合并，每轮调用 LLM 压缩最旧的未压缩消息。多轮设计处理单次摘要遍次不足以充分缩减的情况 — 它持续进行直到上下文适应预算。
  • ContextBuilder（nanobot/agent/context.py）：按以下顺序组装最终上下文：身份 → 引导文件 → 工具契约 → 记忆 → 技能 → 历史（上限 120 条消息）→ 会话摘要。顺序很重要：身份和工具契约是固定开销；历史是最易变的，也是最先被截断或摘要的内容。

钩子驱动与委托式方案

两个代码库采取了根本不同的方法 — 它们自己不实现 compact：

codex（钩子驱动）： compact 由 codex-rs/hooks/src/events/compact.rs 中的可配置钩子驱动。PreCompactRequest 和 PostCompactRequest 结构体携带关于要 compact 什么和已 compact 了什么的上下文。实际的摘要由用户配置的 shell 命令完成 — codex 没有内置的 LLM 摘要器。上下文片段由类型化标记定界，以便钩子脚本可以识别各部分。

这种设计与 codex 对沙箱的哲学一致（第2课）：框架提供事件和契约，但策略完全由用户定义。PostCompactRequest 携带 compact 后的结果和之前/之后的 token 计数，让钩子决定是否接受或重做 compact。

opencode（委托）： compact 完全委托给外部的 @opencode-ai/core 包和 provider 层。OpenCode 本身不运行摘要 — 它依赖 provider 的原生上下文处理。这让 OpenCode 的代码库保持小巧，但意味着 compact 行为不透明且不可跨 provider 移植。

权衡很明确：codex 和 opencode 有零行摘要代码需要维护、调试或安全审查。但它们也对摘要质量零控制。如果钩子脚本或 provider 产生糟糕的摘要，没有回退、没有补救。

中间件驱动的上下文管理（deepagents、deer-flow）

DeepAgents 和 DeerFlow 引入了一种根本不同的模式：上下文管理作为显式命名的中间件，而非 agent 循环内部的隐式逻辑。这与第1课和第2课中看到的"显式中间件 vs 隐式耦合"模式相同 — 但应用到了上下文管理上。在现有的七个代码库中，compact、token 计数和内容修剪是在 agent 循环内部调用的函数。而在 DeepAgents 和 DeerFlow 中，它们是具有清晰接口和可配置触发器的独立中间件类。

DeepAgents：SummarizationMiddleware 和 FilesystemMiddleware

DeepAgents 在两个专用的中间件类中编码上下文管理：

SummarizationMiddleware（middleware/summarization.py:1003）包装 model_call，在超过 token 阈值时执行 LLM 驱动的摘要。当模型有已知的 max_input_tokens 时，默认触发器为 ("fraction", 0.85)，否则为 ("tokens", 170000)。这使得 DeepAgents 具有与 claude-code 和 hermes-agent 相同的主动姿态 — 在调用之前检查，而非在错误之后。如果 LLM 摘要调用本身因 ContextOverflowError 失败，它会回退到紧急摘要（截断旧的工具参数）。

FilesystemMiddleware（middleware/filesystem.py:692-749）自动将大型工具结果卸载到文件系统后端，在上下文中替换为预览。token 限制为工具结果 20,000、人工注入消息 50,000。这是一种自动内容门控机制 — LLM 看到的是预览而非原始转储，完整内容按需检索。它在功能上等同于 hermes-agent 的工具输出修剪（阶段 1）和 claude-code 的微 compact，但实现为可独立开启或关闭的独立中间件层。

DeepAgents 还在消息上使用 DeltaChannel（graph.py:67），通过仅存储增量将检查点增长从 O(N²) 降至 O(N)，每 50 步创建一次完整快照。这是一种结构优化 — 不改变 LLM 看到的内容，但大幅降低在有状态图中维护对话历史的存储成本。

graph.py:70-143 中的系统提示词组装顺序为：USER →（BASE 或 CUSTOM）→ SUFFIX，以空行连接。这是比 claude-code 的 5 优先级合并更简单的组合模型，但实现了相同的目标：每次对话顶部的固定开销可针对不同 agent 模式进行调整。

DeerFlow：19 个中间件管道与延迟工具过滤

DeerFlow 通过 19 个中间件链将中间件模式推向了更远：

ThreadDataMiddleware → UploadsMiddleware → SandboxMiddleware →
DanglingToolCallMiddleware → SummarizationMiddleware → TodoListMiddleware →
... → ClarificationMiddleware（共 19 个）

每个中间件处理一个特定的横切关注点。对于上下文管理，有三个特别相关：

  • DynamicContextMiddleware 将当前日期和记忆作为 <system-reminder> 注入到第一条 HumanMessage 中。这是 hermes-agent 三源上下文组装（MEMORY.md + USER.md + FTS5）的轻量替代方案 — 它注入动态状态而不占用永久的系统提示词空间。
  • SummarizationMiddleware 通过 config.yaml 可配置：触发器可以基于 token、消息数量或最大输入的比例。保留最近消息同时摘要较旧消息。agent/_config.py:56 中的 ContextConfig 暴露 trigger_ratio（默认 0.8）、reserve_ratio（默认 0.1）、compression_prompt 和 summary_template — 全部可用户配置而无需修改源码。
  • DeferredToolFilterMiddleware 在 tool_search 提升操作之前向 LLM 隐藏 MCP 工具 schema。这是一种先发制人的上下文节省策略：不将全部可用工具 schema 在每个系统提示词中发送（每轮可能消耗数千 token），仅被提升的工具可见。其余推迟到需要时。

延迟工具过滤是 DeerFlow 对上下文管理最独特的贡献。在 claude-code 或 hermes-agent 等代码库中，所有工具 schema 始终存在于系统提示词中。对于拥有 50+ 个 MCP 工具的 agent 来说，那是每轮 5,000–15,000 token 的固定开销。DeerFlow 的方法将其缩减为仅 agent 明确请求的工具 — 在功能上等同于工具定义的懒加载。

显式 vs 隐式：跨课程的中间件模式

这是我们代码库调查中第三次出现显式中间件模式：

  • 第 1 课（Agent 循环）： DeepAgents 和 DeerFlow 将循环外化为中间件，而其他 7 个代码库将循环逻辑直接嵌入 agent 中。
  • 第 2 课（工具系统）： DeepAgents 通过中间件（ToolNormalizationMiddleware）规范化工具，而其他代码库在工具注册内部临时处理规范化。
  • 第 3 课（上下文管理）： DeepAgents 和 DeerFlow 将 compact、内容门控和工具可见性变成命名的中间件类，而其他 7 个代码库在循环内部隐式处理这些关注点。

模式是一致的：中间件使上下文管理可组合（可以添加/移除/重新排序策略）、可测试（每个中间件是隔离的单元）和可观察（中间件名称准确告诉你正在应用哪个关注点）。代价是概念上的开销 — 一个 19 个中间件的链条比单个 buildPostCompactMessages() 函数更难端到端理解。但对于需要混合搭配上下文策略的复杂 agent，中间件方法比循环内部的临时条件判断更具可扩展性。

为什么树模型重要

扁平和树之间的区别不只是数据结构选择 — 它决定了哪些操作是可能的：

操作扁平数组（claude-code、hermes-agent、nanobot、deepagents、deer-flow、codex、opencode）树（pi-mono、openclaw）
撤销 compact必须保留旧数组副本或重新请求更改哪些节点为"活跃" — 无数据丢失
从对话中间分支复制数组到该点添加新分支节点 — O(1)
合并分支手动协调两个数组遍历两棵子树，生成合并摘要节点
以不同粒度重新摘要不能 — 原始消息已丢失删除旧的 compact 节点，从保留的子节点创建新节点

pi-mono 和 openclaw 选择树模型主要是为了支持分支 — 同一对话可以分叉到多个方向。但次要好处（可撤销、可重做的 compact）被证明同样有价值。

设计启发

跨代码库成立的模式：

如果你控制上下文窗口大小，主动优于被动。 Claude Code 和 hermes-agent 都在调用 LLM 之前检查 token。这避免浪费 API 调用，给用户更流畅的体验。被动只在 provider 控制窗口且你无法预测限制位置时才有意义。
如果你有多个上下文来源，树模型比扁平数组更清晰。 pi-mono 的会话树将消息、compact 和分支摘要存储为类型化节点。这使得为会话中的任何点重建上下文、撤销 compact 或创建分支变得简单。扁平数组迫使你变更或复制历史。
用户文件（MEMORY.md、USER.md）在摘要中必须是"纯参考"的。 Hermes-agent 的参考前缀是一个关键的安全措施：已 compact 的 MEMORY.md 内容应该通知模型，但绝不应覆盖用户的当前指令。在没有这一区分的情况下将记忆混入系统提示词会导致 agent 遵循陈旧的指令。
在进行 LLM 摘要之前修剪工具输出显著降低 compact 成本。 每个做 compact 的代码库都先修剪工具输出。Hermes-agent 的阶段 1（修剪）和 claude-code 的微 compact 都这样做。LLM 摘要是昂贵的 — 先修剪冗长的工具输出立刻就能回本。

compact 失败时会发生什么？

每个 compact 系统都可能失败 — LLM 摘要器返回垃圾结果、token 数量未充分下降、或 bug 破坏了摘要。各代码库如何处理：

  • claude-code： 如果被动 compact 未能将上下文减到限制以下，错误会传播给用户。一次性设计意味着没有无限重试循环。
  • hermes-agent： 确定性回退摘要（8K 字符，基于规则提取）保证即使 LLM 宕机也能生成某种摘要。防抖动检查也防止压缩器在无效 compact 上浪费 token。
  • pi-mono / openclaw： 如果 LLM 摘要失败，compact 条目不会被创建，原始消息保留在树中。agent 以超载的上下文继续 — 这可能在下次调用时触发 provider 413 错误，但至少数据被保留了。
  • nanobot： 多轮合并（最多 5 轮）提供自然的兜底。如果 5 轮后上下文仍然放不下，最旧的消息会被截断（非摘要）作为最后手段。
  • codex： 钩子脚本完全控制失败处理。PostCompactRequest 携带之前/之后的 token 计数 — 一个编写良好的钩子可以检测到缩减不足并使用不同参数重试。
  • opencode： 失败处理完全是 provider 的责任。如果 provider 的 compact 失败，agent 直接停止工作。
  • deepagents： 如果 LLM 摘要引发 ContextOverflowError，SummarizationMiddleware 回退到紧急摘要 — 截断旧的工具参数而非尝试 LLM 摘要。这是结构性回退，而非重试。
  • deer-flow： 可配置的 reserve_ratio（默认 0.1）确保始终为 LLM 响应预留空间，因此 compact 失败表现为截断的摘要而非崩溃的轮次。19 个中间件链意味着一个中间件（如 SummarizationMiddleware）的失败不会阻止其他中间件（如 TodoListMiddleware）运行。
