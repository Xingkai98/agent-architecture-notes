第8课：错误恢复与韧性

从崩溃到继续——9 个 agent 代码库如何分类、恢复并经受住失败。

2026年6月 · 9 个主要源代码库 · 固定链接

每个 agent 循环都会遇到错误。脆弱的 agent 与韧性的 agent 之间的区别不在于错误是否发生——而在于 agent 是否能够看到错误、分类错误、选择恢复策略，并在不丢失状态的情况下继续。本课追溯了全部 9 个代码库中的完整恢复流水线。

在第2课中，我们看到工具是 agent 循环的引擎。在第3课中，我们看到上下文是循环的记忆。本课涵盖当循环断裂时发生什么——以及每个代码库如何将其重新拾起。

错误类别

每个 agent 错误都可以归入五个类别之一。每个代码库以不同方式处理它们，但这些类别是通用的：

恢复理念的两个极端

在九个代码库中，出现了两个极端——但在两个最新条目中可以看到一条新的轴线：

调用者委派（pi-mono、openclaw） 完全自我恢复（claude-code、hermes-agent）
  1. 调用者委派： pi-mono、openclaw。agent 循环在错误时停止并向上发出原因信号。恢复——重试、重启、模型切换——是调用者的责任。框架提供钩子（config.transformContext、config.prepareNextTurn），但没有内置的重试逻辑。
  2. 完全自我恢复： claude-code、hermes-agent、nanobot。agent 循环包含带有 exponential backoff 的重试、模型回退链、溢出时的上下文压缩以及 checkpoint restore。循环本身就是一个带有错误转换的状态机。
  3. 中间件即边界： DeepAgents、DeerFlow。这些框架不是将恢复逻辑嵌入 agent 循环，而是将错误恢复作为一等中间件关注点。每种错误类型都有自己的中间件。每个 wrap_model_call() 和 wrap_tool_call() 充当错误边界——一个层中的失败可以被外层捕获和处理。DeerFlow 的 19 个中间件链意味着 19 个独立的错误边界：前所未有的细粒度。
  4. 混合模式： codex、opencode。一些错误在内部处理（SQLite 恢复、步骤限制），其他错误则上报给调用者。

Claude Code：最复杂的恢复系统

Claude Code 在 src/query.ts:241 中的 queryLoop() 是一个递归生成器，针对可变 State 逐轮运行 agent 循环。该循环具有12 个终止原因（结束循环的条件）和7 个恢复转换（使用修改后的状态重试当前轮次的条件）。

12 个终止原因

循环的每次退出都用行号记录。这些不是异常——它们是带有 reason 鉴别器的显式 return 语句：

7 个恢复转换

在达到终止原因之前，循环可能走上 7 条恢复路径中的一条。每条路径修改 State 并 yield 回循环顶部进行另一次尝试：

// claude-code: src/query.ts — queryLoop() 中的恢复转换

// 1. collapse_drain_retry — drain-and-collapse 后重试同一轮
// 2. reactive_compact_retry — 溢出触发的 compaction，然后重试
// 3. max_output_tokens_escalate — 首次 'max_tokens' 命中 → 升级预算
// 4. max_output_tokens_recovery — 后续命中 → 注入恢复消息
// 5. stop_hook_blocking — stop hook 说"不要停止"，继续循环
// 6. token_budget_continuation — 精确预算命中，追加继续请求
// 7. next_turn — 正常转换：收集结果，开始下一轮

当模型达到其输出 token 限制时，Claude Code 使用两阶段升级：

模型回退

API 错误在 src/services/api/withRetry.js 中被捕获。当主模型因可重试错误而失败时，会抛出 FallbackTriggeredError。调用者捕获此异常并切换到 fallbackModel——通常是一个更小、更快的模型，至少能在要求用户干预之前完成当前轮次。

设计观察： Claude Code 将所有代码库中最多的恢复逻辑打包到了一个函数中。结果是一个循环，其中每个退出和重试路径都在一个地方可见——没有分散在文件中的隐藏恢复逻辑。这是有意为之："压平循环，在一个函数中看到所有终止条件。"

Hermes-agent：分类优先的恢复，22 个失败原因

Hermes-agent 采取了根本不同的方法：不是带有内联恢复的平坦循环，而是使用专用的错误分类流水线，在任何恢复决策做出之前运行。

TurnRetryState：14 个一次性恢复守卫

agent/turn_retry_state.py:33 定义了 TurnRetryState dataclass——一个包含 14 个布尔标志的集合，每个标志作为一次性守卫，防止无限重试循环：

  • 5 个认证刷新守卫： refreshed_oauth、refreshed_api_key、rotated_org、switched_project、cleared_token_cache
  • 6 个格式/负载守卫： truncated_context、dropped_images、compressed_payload、switched_to_text_only、split_long_message、reduced_tool_count
  • 2 个传输/速率限制守卫： switched_endpoint、rate_limit_backoff_applied
  • 2 个重启信号： needs_fresh_turn、needs_model_fallback

每个守卫在首次采取其恢复操作时设置为 True。如果同样的错误再次出现，分类器看到守卫已经设置，会升级为更严重的失败（通常是模型回退或终止错误）。

回退链

Hermes-agent 在 agent 实例上维护一个有序回退链：agent._fallback_chain——一个 (provider, model) 对的列表。当分类器返回硬失败原因（model_not_found、billing、content_filter），或软失败的所有一次性守卫都已用尽时，agent 激活链中的下一个条目并将 retry_count 重置为 0。这意味着完整的恢复预算会在每次回退步骤中重置——新模型获得新的机会。

DeepAgents：中间件即错误边界

DeepAgents 对错误恢复采取了一种根本性的架构方法：每一段恢复逻辑都是一个中间件。DeepAgents 不是将错误处理内联在 agent 循环中，而是将模型调用和工具调用包装在一个中间件栈中，其中每一层捕获并处理特定的错误类别。

PatchToolCallsMiddleware：流式工具调用修复

middleware/patch_tool_calls.py:11 解决了一个特定于流式传输的问题：当流在工具调用中途被切断时，部分的工具调用可能导致 agent 崩溃。PatchToolCallsMiddleware 检测流式响应中不完整的工具调用负载，并在它们到达执行层之前完成或删除它们。这防止了一整类"格式错误响应"的崩溃——大多数代码库通过简单的重试来处理这些崩溃。

SummarizationMiddleware：紧急上下文压缩

当 LLM 提供商抛出 ContextOverflowError 时，SummarizationMiddleware 触发紧急摘要化——压缩对话历史以适应 token 预算——保持 agent 存活而不是崩溃。与 Claude Code 的多阶段 drain-and-collapse 流水线不同，这是中间件层的单阶段紧急干预。

ModelRetryMiddleware 和 ModelFallbackMiddleware

DeepAgents 集成了来自 langchain.agents 的两个中间件：

  • ModelRetryMiddleware —— 在单个中间件内对瞬态提供商错误（速率限制、服务器错误、超时）进行重试，将重试逻辑与其余栈隔离开。
  • ModelFallbackMiddleware —— 当重试耗尽时，切换到回退模型。回退决策是中间件的关注点，而不是循环的关注点。

AnthropicPromptCachingMiddleware

始终包含在中间件栈中，此中间件优雅地处理 prompt 缓存失败。如果 Anthropic API 拒绝缓存的 prompt 前缀（例如由于缓存驱逐），此中间件回退到非缓存请求，而不是将错误上抛到 agent 循环。

架构洞察：中间件就是错误恢复

在 DeepAgents 中，中间件栈中的每个 wrap_model_call() 和 wrap_tool_call() 都充当一个错误边界。当中间件层失败时，外层可以捕获并处理该失败。这与 Claude Code、hermes-agent 和 nanobot 的内联恢复方法根本不同。DeepAgents 不是在一个大循环中嵌入恢复逻辑，而是从独立的、可测试的中间件片段组合恢复——每个中间件负责恰好一种错误类型。

设计观察： DeepAgents 证明了错误恢复可以被组合而不仅仅是嵌入。中间件栈是声明性的：你可以通过阅读中间件列表看到每种恢复策略。添加新的恢复策略意味着添加一个中间件——无需更改现有代码。

DeerFlow：19 个中间件错误边界链

DeerFlow 将中间件方法推向了逻辑极端：一个19 个中间件链，每个中间件处理一个错误关注点。拥有 19 个独立的错误边界，DeerFlow 实现了任何其他代码库都无法比拟的细粒度。

LLMErrorHandlingMiddleware：提供商失败规范化

此中间件拦截所有提供商/模型调用失败，并将其规范化为可恢复的、面向助手的错误。LLMErrorHandlingMiddleware 不是让原始提供商异常通过栈传播，而是将其转换为下游中间件和工具可以推理的结构化错误。

ToolErrorHandlingMiddleware：工具异常作为消息

将工具执行异常转换为错误 ToolMessages，使运行可以继续而不是中止。当工具抛出异常时，agent 看到的是结构化的错误消息而不是崩溃——模型可以决定重试、绕过或报告失败。

DanglingToolCallMiddleware：孤立工具调用恢复

当 AIMessage 包含的 tool_calls 缺少对应的 ToolMessage 响应时（例如由于用户在中途打断），此中间件注入占位 ToolMessages——包括原始提供商工具调用负载——使对话保持结构有效。如果没有这个，孤立的工具调用将导致下一次模型调用因模式错误而失败。

LoopDetectionMiddleware：独一无二的

DeerFlow 是唯一将显式循环检测作为架构组件的代码库。LoopDetectionMiddleware 检测模型何时陷入重复的工具调用循环。当检测到循环时，它执行硬停止：清除响应中的结构化 tool_calls 和原始提供商工具调用元数据，然后强制模型生成最终文本答案。这防止了最令人沮丧的 agent 失败模式——无限工具调用循环，模型不断用相同的参数调用相同的工具。

SandboxAuditMiddleware：执行前安全网

在每次工具执行前进行安全日志记录。虽然主要是安全功能，但它充当执行前安全网——危险的工单在执行前被阻止，防止恢复成为必要。

SubAgentExecutor 超时

子 agent 获得默认 900 秒超时。如果子 agent 超出其预算，会发出 task_timed_out 事件，父 agent 可以决定是重试、回退还是报告。这是委派的超时处理——父 agent 保持控制权。

Worker 级恢复：checkpoint 与回滚

worker.py:164-199 实现了运行前 checkpoint 快照。在每次运行之前，worker 捕获其状态。如果运行失败，worker 可以恢复到运行前的 checkpoint——有效地撤销失败尝试中的所有状态变更。这是整个 worker 的回滚，而不仅仅是上下文或工具级恢复。

设计观察： DeerFlow 的 19 个中间件链意味着 19 个独立的错误边界——在所有 9 个代码库中前所未有的细粒度。每个中间件都是单一职责的错误处理器。结果是一个恢复架构，你可以在其中添加、删除或重新排序恢复策略，而无需触及 agent 循环。LoopDetectionMiddleware 尤其独一无二：没有其他代码库将循环检测作为一等组件融入其架构。

Codex：SQLite 损坏恢复

Codex 因其架构独有的恢复机制而与众不同：agent 的运行时状态存储在 SQLite 数据库中，而 SQLite 损坏是长期运行 agent 会话中的一种真实失败模式。

备份并重新开始

codex-rs/state/src/runtime/recovery.rs:71 实现了一个干净的恢复策略：

// codex-rs: codex-rs/state/src/runtime/recovery.rs:71
fn backup_runtime_db_for_fresh_start(corrupt_path: &Path) -> Result {
    // 1. 创建备份目录：db-backups/sqlite-{timestamp}-{seq}/
    // 2. 将损坏的 DB 文件移入备份目录
    // 3. 返回备份路径（以便调用者可以检查它）
    // 4. 调用者在原始路径创建新的 DB
    // 5. 之前的状态丢失，但损坏的 DB 被保留用于取证

检测函数 is_sqlite_corruption_error() 遍历错误链（Rust 的 Error::source()），查找 SQLite 错误码 11（SQLITE_CORRUPT——数据库磁盘映像损坏）和 26（SQLITE_NOTADB——文件不是数据库）。这是一次性恢复，不重试——如果新的 DB 也损坏，agent 硬失败。

Nanobot：提供商重试、长度恢复与 checkpoint restore

Nanobot 在三个不同的层次实现恢复：

提供商重试

nanobot/providers/base.py:817 —— _run_with_retry() 具有两种模式：

  • 标准模式： 3 次尝试，延迟分别为 1 秒、2 秒、4 秒。用于正常的 API 调用。
  • 持久模式： 无限重试，60 秒延迟上限。用于长期运行的 agent 会话，用户期望 agent 不断尝试直到提供商恢复。

长度恢复

nanobot/agent/runner.py:528 —— 当 LLM 响应具有 finish_reason == "length"（模型在响应中达到了其输出 token 限制）：

# nanobot/agent/runner.py:528
# 1. build_length_recovery_message() 构造：
#    "你之前的回复因长度限制而被截断。
#     请从你中断的地方继续。"
# 2. 该消息作为用户轮次注入
# 3. 最多 3 次尝试——然后循环终止

空响应重试

空的 LLM 响应（无内容且无工具调用）最多重试 2 次。重试消息明确要求模型产生输出。

流中断恢复

如果流式连接中断，on_stream_recover 回调从最后收到的 token 开始一个新的流段。模型从中断处继续生成，而不是丢弃整个轮次。

Checkpoint restore

nanobot/agent/loop.py:1725 —— 在崩溃或重启后，加载运行中的轮次 checkpoint。任何在崩溃时正在进行的工具调用会获得合成的错误结果：

# nanobot/agent/loop.py:1725
# 恢复的工具调用 → 合成错误：
# {"error": "工具调用因 agent 重启而中断。结果不可用。"}
# LLM 将其视为错误工具结果，并可以决定重试还是继续。

Opencode：Effect-TS 类型化错误与步骤限制

Opencode 使用 Effect-TS 在类型系统中编码错误：

// opencode: packages/core/src/session/runner/index.ts:20
type RunError =
  | LLMError               // 提供商级失败
  | Model.Error            // 模型特定错误
  | MessageDecodeError     // 无法解析 LLM 输出
  | ContextSnapshotDecodeError  // 损坏的上下文快照
  | StepLimitExceededError // 超过 MAX_STEPS

步骤限制

MAX_STEPS = 25。当轮次计数达到 25 时，引发 StepLimitExceededError。这是一个硬终止——不重试，不升级。调用者必须决定如何处理。

上下文溢出恢复

compactAfterOverflow() 执行单次压缩并重试。如果一次压缩后上下文仍然溢出，错误向上传播。没有像 Claude Code 的 drain-and-collapse 流水线那样的多阶段升级。

错误时的工具清理

failUnsettledTools() 在轮次中发生提供商错误时被调用。任何已排队但尚未完成的工具调用被标记为失败并附上错误消息，确保对话状态保持一致——模型看到它的工具调用没有被执行。

Pi-mono、OpenClaw：最小化错误处理

这两个 TypeScript 代码库采取了尽可能简单的方法：

  • 无内置重试。 当 stopReason === "error" 或 "aborted" 时循环停止。
  • 工具执行错误 在 executeToolCalls() 中被捕获并作为错误工具结果返回——LLM 看到带有 isError: true 的 ToolResultMessage。
  • 恢复被委派给调用者，通过两个配置钩子：
    • config.transformContext —— 允许调用者在每轮之前修改上下文（例如压缩、截断）
    • config.prepareNextTurn —— 在每轮之前调用，调用者可以注入恢复消息或切换模型

理念：框架不应该做出恢复决策。调用者了解自己的用例和容错能力。给他们钩子，而不是策略。

设计启发

研究全部 9 个代码库后，浮现出五个原则：

1. 恢复路径应该是可见的。 压平循环，在一个函数中看到所有终止条件。Claude Code 的 queryLoop() 是典范——12 个终止原因和 7 个恢复转换，全在一个文件中，全带有行号。当恢复逻辑分散在多个文件中（重试在一个文件，压缩在另一个，回退在第三个），就不可能推理完整的状态机。
2. 一次性守卫防止无限恢复循环。 每个恢复路径需要一个标志，防止它对同一个错误激活两次。Hermes-agent 的 14 个 TurnRetryState 布尔标志是对这一原则最清晰的实现。Claude Code 的 max-tokens 恢复（升级一次，然后注入最多 3 次）是另一个例子。没有这些守卫，一个持久错误会永远循环。
3. 工具错误应该对 LLM 可读。 将工具失败格式化为模型可以推理的消息。每个代码库都这样做，但 nanobot 的 checkpoint restore 走得最远：孤立的工具调用获得解释为什么结果不可用的合成错误消息。模型然后可以决定是重试工具调用还是绕过缺失的数据。
4. 恢复越复杂，循环就应该越平坦。 Claude Code 拥有最多的恢复路径（7 个）和最平坦的循环（全在 queryLoop() 中）。这不是巧合。当恢复逻辑深度嵌套或分散在模块间时，添加新的恢复路径有可能破坏现有的路径。一个平坦、可见的循环使每个转换都明确，每个交互都可观察。
5. 当错误空间很大时，组合中间件的错误恢复优于嵌入式的错误恢复。 DeepAgents 和 DeerFlow 表明，当你有许多错误类型时（流式传输中断、工具失败、上下文过载、循环检测、缓存失败、模型回退），从独立的中间件片段组合恢复比将其全部嵌入一个循环中更易于维护。每个中间件都是单一职责的错误处理器——隔离可测试、顺序可组合。DeerFlow 的 19 个中间件链证明了这可以扩展到两位数的错误关注点而不会在自身的重量下崩溃。然而，中间件组合失去了平坦循环的可见性优势——你无法在一个函数中看到所有恢复转换。
