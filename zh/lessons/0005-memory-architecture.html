第5课：记忆架构

working、short-term、long-term —— 每个 agent 必须管理的三层记忆体系，以及九个代码库如何实现（或不实现）。

2026年6月 · 9 个主要源代码库 · 固定链接

记忆是将无状态 API 调用与一个随时间学习并改进的 agent 区分开来的关键。每个 agent 都面临同样的根本问题：LLM 上下文窗口是有限的，但对话、文件和经验是无界的。解决方案始终是分层架构 —— 但实现方式从完整的三层编排到"我们直接往 JSONL 里写数据就完事了"不等。

在第3课中，我们介绍了 agent 如何管理 LLM 上下文窗口。在第4课中，我们分析了权限控制。本课将视角扩展到更广泛的问题：agent 记住什么，存储在哪里，以及如何在正确的时间检索到正确的记忆？

记忆的三个层级

每个 agent 记忆架构都可以分解为恰好三个层次：

Working Memory（工作记忆） 上下文内
仅当前对话
约 100K token
→
Short-Term Memory（短期记忆） 会话范围内
消息、文件、笔记
数小时到数天
→
Long-Term Memory（长期记忆） 持久化存储
跨会话、跨天数
数周到数月

Working memory 就是 LLM 上下文窗口本身——模型当前能"看到"的消息。Short-term memory 是会话历史，能够在单轮对话间存活但无法跨会话持久。Long-term memory 是持久化的：跨会话、跨天数，最终跨月存储。

困难的问题不在于存储——而在于检索。存储一切很简单。在正确的时间将正确的记忆注入到正确的上下文窗口中，才是整个游戏的关键所在。

hermes-agent：定义性的三层架构

Hermes-agent 是这个集合中最复杂的记忆系统。它实现了全部三个层级，配备了后台守护进程、全文搜索、skill 生命周期管理，以及 cron 驱动的建议提案。其他所有代码库要么省略了某个层级，要么以更简单的形式实现它。

MemoryManager：prefetch → inject → sync

agent/memory_manager.py:252 中的 MemoryManager 在每一轮中编排一个三阶段流水线：

# agent/memory_manager.py:252 — MemoryManager 编排
class MemoryManager:
  def __call__(self, agent: Agent, ctx: AgentContext, turns: list[Turn]):
    # Phase 1: Prefetch — 在 LLM 调用前拉取相关记忆
    relevant = self._prefetch_memories(ctx, turns)
    # Phase 2: Inject — 将记忆插入到 system prompt 中
    ctx = self._inject_memories(ctx, relevant)
    # Phase 3: Sync — 后台线程进行新记忆提取
    self._start_sync_thread(agent, turns)

prefetch 阶段是最关键的。在每次 LLM 调用之前，MemoryManager 使用带 FTS5 全文搜索的 SQLite 查找与当前对话相关的记忆。它从最近的轮次中提取关键词，并与已存储的记忆条目进行匹配。

inject 阶段将检索到的记忆插入到 system prompt 中的一个仅参考块中——agent 可以看到它们，但不能将其视为活动指令。这个区分至关重要：用户管理的文件（如 MEMORY.md）必须是提供信息，而非发号施令。

sync 阶段启动一个后台守护线程（通过 background_review.py），回放对话并向 LLM 询问："这次交流中是否应该保存任何 skill 或 memory？" 这是异步发生的，永远不会阻塞主 agent 循环。

存储：SQLite 配合 FTS5 + 中日韩三元语法

hermes_state.py 中的存储层使用 SQLite 并启用 WAL 模式，以支持写入期间并发读取。Schema 包括 FTS5 全文搜索，支持标准分词和中日韩三元语法（trigram）支持（适用于中文、日文、韩文文本）。触发器在 insert/update/delete 时保持 FTS 索引与主表同步。

# hermes_state.py — FTS5 配合中日韩三元语法支持
CREATE VIRTUAL TABLE memories_fts USING fts5(
  title, content, tags,
  tokenize='trigram'  -- 中日韩 + 标准文本搜索
);
CREATE TRIGGER memories_ai AFTER INSERT ON memories BEGIN
  INSERT INTO memories_fts(rowid, title, content, tags)
  VALUES (new.rowid, new.title, new.content, new.tags);
END;

人类可读层位于 ~/.hermes/memories/，以 MEMORY.md 和 USER.md 文件的形式存在。这些是基础真相 —— SQLite 数据库是派生的索引，而非真相来源。用户可以直接编辑这些文件，agent 在下一次加载时予以尊重。

后台审查与 skill 策展

两个后台系统随时间推移维护记忆健康：

Background Review（后台审查）（agent/background_review.py）：一个守护线程，在每一轮后回放对话并向 LLM 提出一个问题："是否应该从此对话中保存任何 skill 或 memory？"这是提取机制 —— 由 LLM 决定什么值得保留，而非启发式规则。

Skill Curator（skill 策展器）（agent/curator.py）：管理已保存 skill 的生命周期。Skill 自动在状态间转换：active（最近使用过）→ stale（30天未使用）→ archive（90天未使用）。这防止了记忆膨胀，并保持活跃集的相关性。

Cron Suggestions（cron 建议）（cron/suggestions.py）：一个经验驱动的系统，分析过去的交互并提议新的 cron 作业。如果 agent 注意到一个模式——"你总是在周一早上要求 standup 摘要"——它可以建议将其自动化。

Claude Code：基于文件的记忆层级

Claude Code 采用面向文件的方法。记忆不是使用数据库，而是以众所周知路径下的 markdown 文件形式组织，按严格的优先级顺序加载。

四层 CLAUDE.md 加载

src/utils/claudemd.ts 加载器从四个层次解析 CLAUDE.md 文件，按越来越具体的顺序合并：

  1. Managed（托管层）： /etc/claude/claudemd/ —— 系统级，管理员控制
  2. User（用户层）： ~/.claude/CLAUDE.md —— 个人级，跨项目
  3. Project（项目层）： 仓库根目录中的 CLAUDE.md，加上 .claude/CLAUDE.md —— 团队共享
  4. Local（本地层）： CLAUDE.local.md —— 每个开发者独立，已 .gitignore

较高层级设定策略；较低层级添加具体内容。项目级 CLAUDE.md 无法覆盖托管层的指令，但可以添加项目特定的指令。

MEMORY.md 系统

src/memdir/ 模块管理 MEMORY.md 文件，有严格限制：最多 200 行或 25KB，以先触发者为准。它支持 @include 指令，用于将大型记忆拆分为多个文件，并在 agent 积累足够上下文后自动写入记忆文件。

Session memory（会话记忆）

src/services/SessionMemory/sessionMemory.ts 为每个会话维护自动生成的 markdown 笔记。这些笔记记录会话期间发生的事情——接触过的文件、做出的决策、遇到的问题——并将其持久化到磁盘供以后审查。

Memory consolidation（记忆整理）

Claude Code 中的 consolidation 由 token 阈值触发：至少 10,000 token 才能启动，且两次更新之间至少间隔 5,000 token。一个 forked subagent 从对话中提取关键信息——它作为独立进程运行，因此永远不会阻塞主 agent。consolidation agent 被赋予聚焦的 prompt：提取事实、决策和模式，而非原始对话。

Nanobot：文件 I/O 配合 Consolidator

Nanobot 的记忆系统是基于文件的，但增加了一个增量 consolidation 机制，这是 Claude Code 较简单的阈值方法所缺乏的。

MemoryStore：四个文件 + 历史记录

nanobot/agent/memory.py:40 定义了带有五个持久化产物的 MemoryStore：

# nanobot/agent/memory.py:40 — MemoryStore
class MemoryStore:
  def __init__(self, root_dir: Path):
    self.memory_md    = root_dir / "MEMORY.md"     # 持久记忆
    self.user_md      = root_dir / "USER.md"       # 用户偏好
    self.soul_md      = root_dir / "SOUL.md"       # agent 身份
    self.history_jsonl = root_dir / "history.jsonl" # 整理后的日志
    self.cursor        = 0                          # 最后一次整理的位置

SOUL.md 文件在所有 9 个代码库中是独一无二的——它定义了 agent 的个性和行为约束，与事实记忆（MEMORY.md）和用户偏好（USER.md）分开。这种三部分划分（身份、事实、偏好）是一种干净的分离，其他代码库将其混在更少的文件中。

Consolidator：增量 LLM 摘要

memory.py:602 中的 Consolidator 名副其实：它将旧消息增量摘要到 history.jsonl 中。它使用基于游标的方法——通过 cursor 字段跟踪哪些消息已经被整理，因此每次运行只处理新消息。

consolidation 边界算法在用户轮次边界处选择安全的切割点。它永远不会在 agent 响应的中间切分对话——切割总是在用户消息和下一条 agent 消息之间。这保持了摘要中的对话连贯性。

AutoCompact：空闲会话归档

nanobot/autocompact.py:17 提供了 AutoCompact，一种基于 TTL 的归档机制。如果会话在配置的时间之外处于空闲状态，AutoCompact 触发 consolidation 并归档完整的对话记录。这防止了长期运行但不常使用的 agent 出现磁盘膨胀。

基于 Git 的版本控制

SOUL.md、USER.md 和 MEMORY.md 通过 GitStore 封装进行版本控制。对这些文件的每次更改都会用描述性消息提交，为用户提供 agent 记忆如何演变的审计跟踪。history.jsonl 不进行版本控制——它被视为缓存。

Codex：SQLite + 两阶段提取

Codex 使用关系型方法，包含四个独立的 SQLite 数据库和两阶段记忆提取流水线。

四个数据库

Codex 维护四个 SQLite 数据库，每个有不同的关注点：

  • State DB： 运行时状态、工具结果、对话元数据
  • Logs DB： 结构化日志条目，用于调试和可观测性
  • Goals DB： 跨会话的持久化目标跟踪
  • Memories DB： memories_1.sqlite —— 长期记忆存储

两阶段提取流水线

codex-rs/state/src/runtime/memories.rs:28 实现带有两阶段提取流水线的 MemoryStore：

// codex-rs/state/src/runtime/memories.rs:28
// Stage 1: 线程级提取 —— 每个对话线程运行一次
// Phase 2: 全局 consolidation —— 跨线程合并，6 小时冷却
pub struct MemoryStore {
  db: SqlitePool,
  last_global_consolidation: Instant,  // 6 小时冷却
  // ...
}

Stage 1 从单个对话线程中提取记忆——决定了什么，学到了什么。这在足够多的轮次后频繁运行。

Phase 2 执行全局 consolidation——将来自多个线程的记忆合并到统一的知识库中。它在 6 小时冷却周期内运行，以避免冗余的 LLM 调用。冷却时间是每进程的（非持久化），因此在重启时重置。

两阶段设计体现了一个关键洞察：线程级提取成本低，可以频繁运行，但跨线程 consolidation 成本高，应该批量处理。

Minimal memory：pi-mono、openclaw 和 opencode

pi-mono：仅 JSONL 会话

pi-mono 在所有代码库中拥有最简单的记忆方案：JSONL 会话文件。仅此而已。没有 MEMORY.md，没有 consolidation，没有 skill 学习，没有后台提取。每个会话被写入 .jsonl 文件以供重放，但没有机制让信息跨越会话边界。为什么？pi-mono 被设计为一个研究框架，记忆是消费者的责任——框架提供循环、钩子和工具，应用开发者决定要记住什么。

openclaw：SDK 可用，非默认

openclaw 提供了带有 SQLite 存储和可选 embedding 语义搜索的 Memory Host SDK。然而，它默认不被使用。该 SDK 可供想要接入的消费者使用，但框架本身的运行不包含任何长期记忆。这反映了 openclaw 的设计理念：提供能力，不强制集成。

opencode：仅 localStorage

OpenCode 的 packages/app/src/utils/persist.ts 将 UI 状态持久化到 localStorage——如面板宽度、排序顺序和筛选偏好。完全没有 agent 记忆系统。持久化纯粹是针对面向用户的 UI，而非 agent 的知识。这与 OpenCode 定位为轻量编码助手而非持久化 agent 是一致的。

记忆作为中间件：DeepAgents 与 DeerFlow

两个代码库因一个统一的架构决策而脱颖而出：它们将记忆作为一等中间件关注点，而非附加在 agent 循环上的临时功能。DeepAgents 和 DeerFlow 都将记忆实现为拦截请求/响应管道的中间件——记忆在 LLM 调用之前加载，在调用之后提取，且从不阻塞主循环。

这是值得关注的关键架构模式：当记忆是中间件时，它在每一轮中都会被保证执行。当记忆是 agent 循环中的临时代码时，很容易被跳过、遗忘或绕过。中间件通过设计强制执行记忆卫生——你不可能意外遗漏 prefetch 或忘记 consolidation，因为中间件层是无条件运行的。

"存在于 agent 循环中的记忆是可选的。存在于中间件中的记忆是强制性的。" 将记忆逻辑嵌入 agent 的 turn 函数中还是将其注册为中间件，这个架构选择决定了记忆是一等关注点还是事后才考虑的东西。DeepAgents 和 DeerFlow 都选择了中间件，因此都拥有更丰富的记忆系统。

DeepAgents：通过 CompositeBackend 实现分层存储

DeepAgents 采用后端抽象的方法来处理记忆。它定义了一个 CompositeBackend，按命名空间前缀将记忆操作路由到三个专用后端——每个后端具有不同的持久化保证和生命周期。

三个后端，一个路由器

backends/composite.py 中的 CompositeBackend 充当类似 URL 路由器的记忆操作分发器，按命名空间前缀进行路由：

# backends/composite.py — 基于前缀路由到三个后端
class CompositeBackend:
  def route(self, namespace):
    prefix = namespace[0] if namespace else ""
    if prefix == "memories":
      return self.store_backend     # 跨会话，持久化
    elif prefix == "data":
      return self.filesystem_backend # 磁盘支持，持久化
    else:
      return self.state_backend      # 线程内部，临时性

三个后端服务于不同的目的：

  • StateBackend（backends/state.py:38）：线程内部的临时存储，使用 LangGraph 状态通道。数据通过 LangGraph 检查点在单个线程内持久化，但对其他线程不可见。最适合：对话草稿数据、不需要跨会话边界的运行中计算结果。
  • StoreBackend（backends/store.py:172）：通过 LangGraph 的 BaseStore 实现跨会话持久化存储，命名空间隔离。支持两种编码格式：v1（字符串列表）和 v2（字符串 + 编码字段，用于更丰富的元数据）。最适合：事实、偏好、必须跨会话边界存活的学习知识。
  • FilesystemBackend：磁盘支持的存储，用于大型或人类可读的数据。最适合：文件、文档、用户可能想要直接查看或编辑的任何产物。

这种命名空间前缀路由是一种干净的架构模式：调用者通过命名空间指定意图，CompositeBackend 透明地路由到正确的存储层。代码库中没有分散的 if-else 链，没有魔法常量——只有命名空间驱动的、所有权清晰的分发。

MemoryMiddleware：AGENTS.md 注入

middleware/memory.py:180-441 中的 MemoryMiddleware 处理记忆注入到 agent 上下文中。在 before_agent()（第 303 行）中，它通过配置的后端加载 AGENTS.md 文件。然后在 modify_request()（第 371 行）中将它们以 <agent_memory>...</agent_memory> XML 标签的形式注入到 system prompt 中。加载每个会话仅执行一次——DeepAgents 将 AGENTS.md 视为会话内稳定的上下文，而非每轮都需要重新读取的内容。

DeerFlow：完整记忆生命周期

DeerFlow 可以说是所有九个代码库中最复杂的记忆系统。它实现了完整的记忆生命周期：检测 → 排队 → 提取 → 写入 → 注入，每一步都由 LLM 驱动的智能所支持，而非启发式规则。

记忆结构：上下文、历史和置信度评分的事实

agents/memory/storage.py:24 中的记忆模型将知识组织为三层：

# agents/memory/storage.py:24 — 记忆结构
{
  "context": {
    "workContext": "...",       # 用户的职业上下文
    "personalContext": "...",   # 个人兴趣、偏好
    "topOfMind": "..."          # 当前相关的内容
  },
  "history": {
    "recentMonths": "...",      # 压缩的近期历史
    "earlierContext": "...",    # 更早的、更概括的历史
    "longTermBackground": "..." # 基础的、很少变化的内容
  },
  "facts": [
    {
      "id": "uuid",
      "content": "事实文本",
      "category": "...",
      "confidence": 0.85,       # 0–1 置信度评分
      "createdAt": "iso8601",
      "source": "产生此事实的对话轮次"
    }
  ]
}

每条事实都带有一个置信度评分（0–1）。这是关键的设计决策：高置信度的事实被积极注入；低置信度的事实被保留或修剪。基于置信度的修剪意味着记忆系统自我清理——从未被强化的事实自然而然地消失，无需 cron 任务或手动策展。

存储：基于 mtime 缓存的每用户 JSON

agents/memory/storage.py:62 中的 FileMemoryStorage 将记忆存储为每个用户一个 JSON 文件，位于 {base_dir}/users/{user_id}/memory.json，或按 agent 划分的 users/{user_id}/agents/{agent_name}/memory.json。文件通过mtime 失效机制缓存在内存中——如果文件的修改时间发生变化，缓存会被刷新。这对于单写入者场景是安全的，同时避免了数据库的复杂性。

记忆提取流水线：检测、排队、提取、注入

  1. 检测（agents/middlewares/memory_middleware.py:28）：MemoryMiddleware 将对话流过滤为仅用户输入 + 最终 AI 回复——没有中间工具调用，没有思考追踪。它检测纠正和强化信号（例如，"其实，我的意思是……"或"记住……"）作为触发立即记忆更新的信号。
  2. 排队（agents/memory/queue.py:28）：MemoryUpdateQueue 提供每线程去重，可配置 debounce（默认 30 秒）。后台处理使用 threading.Timer，在入队时捕获 user_id。快速连续的消息不会触发冗余的提取运行——队列会合并它们。
  3. 提取（agents/memory/updater.py:380）：MemoryUpdater 是 LLM 驱动的。update_memory()（第 539 行）是同步路径，在异步上下文中被卸载到线程池。_apply_updates()（第 600 行）处理上下文变更、事实添加/删除和基于置信度的修剪。LLM 决定什么是事实、属于什么类别以及置信度是多少——没有正则表达式，没有关键词匹配。
  4. 注入：通过 DynamicContextMiddleware，前 15 条事实加上上下文以 <memory> 标签注入到 system prompt 中，受 token 计数预算控制（tiktoken 或基于字符的备选方案），默认 max_injection_tokens 上限为 2,000。

为什么 DeerFlow 的设计值得关注

基于置信度的事实模型和自动修剪意味着 DeerFlow 的记忆是自我维护的。从未被强化的低置信度事实被修剪；积累支持证据的高置信度事实则保留。这是所有九个代码库中唯一将记忆置信度视为具有自动生命周期后果的一等维度的系统——hermes-agent 的 curator 管理 skill 生命周期（基于时间），但 DeerFlow 管理事实生命周期（基于证据）。

带去重的 debounced 队列也是独特的。其他代码库要么在每一轮都运行 consolidation（hermes-agent），要么在固定的冷却时间内运行（Codex）。DeerFlow 的 debounce 是自适应的——它仅在对话稳定后才运行提取，在不丢失信息的情况下减少 LLM 成本。

设计启发

"记忆只有在正确的时间可检索时才是有用的——prefetch 是全部关键。" 存储记忆是简单的部分。困难的部分是在 agent 需要时，恰好将正确的记忆呈现到上下文窗口中。Hermes-agent 的 prefetch 阶段、Claude Code 的分层 CLAUDE.md 加载以及 Codex 的 SQL 查询都解决同一个问题：检索时机。
"Consolidation 应该在后台进行，永远不要阻塞主循环。" 每个具有 consolidation 的代码库（hermes-agent、claude-code、nanobot、codex）都是异步运行的。Hermes-agent 使用守护线程。Claude Code fork 了一个 subagent 进程。Codex 使用 6 小时冷却门控。主 agent 循环永远不应该等待 consolidation 运行——用户的轮次时间是神圣的。
"用户管理的文件（MEMORY.md）在上下文中必须是'仅参考'——永远不要视为活动指令。" Hermes-agent 将记忆注入到 system prompt 的参考块中。Claude Code 将 CLAUDE.md（指令）与 MEMORY.md（知识）分开。这个区别很重要：用户写指令；agent 写记忆。混淆它们会导致 agent 将其自身过时的记忆当作命令来执行。
"陈旧度会衰减——没有生命周期管理的记忆会变成噪音。" Hermes-agent 的 curator 是唯一具有明确生命周期管理的系统：active → stale（30天）→ archive（90天）。没有它，长期运行的 agent 会积累杂物——旧事实、已解决的 TODO 和稀释检索信号的不相关上下文。
"安全的切割点对连贯性很重要。" Nanobot 的 consolidator 仅在用户轮次边界处分割。Codex 将线程内的 consolidation 与全局 consolidation 分开。两者都认识到任意截断会产生不连贯的摘要。切割点必须尊重对话结构。
"记忆作为中间件——保证在每一轮中执行。" DeepAgents 和 DeerFlow 都将记忆作为一等中间件关注点。当记忆是中间件时，它在每一轮中都会被保证执行——你不可能意外跳过 prefetch 或忘记 consolidation。当记忆是 agent 循环中的临时代码时，很容易被绕过。记忆代码的架构位置（中间件层 vs. turn 函数体）直接决定了记忆系统的可靠性。

为什么 pi-mono 没有记忆

pi-mono 中记忆的缺失是一个设计决策，而非遗漏。pi-mono 是一个研究框架——它提供 agent 循环、钩子和工具接口，并将记忆留给消费者。这与"原语，而非功能"的理念一致：给开发者构建块，而非预设好的记忆架构。

但这有实际成本：每个 pi-mono 消费者必须构建自己的记忆系统。如果不这样做，他们的 agent 就没有持久知识。这对于一次性编码任务来说是可以接受的；对于长期运行的 agent 来说则是致命的。

  • 如果你想要最复杂的记忆系统：追踪 hermes-agent/agent/memory_manager.py 和 hermes_state.py —— 带有 FTS5 的 prefetch-inject-sync 流水线是黄金标准。
