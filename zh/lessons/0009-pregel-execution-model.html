第9课：Pregel 执行模型

图，而非循环——LangGraph 如何用声明式状态机替代命令式控制流。

课程二：Agent 基础设施 · 2026年6月 · 永久链接

课程一中的每个 agent 都围绕一个循环构建。while (should_continue)、for iteration in range(max)、状态机——全是命令式的。本课介绍一种根本不同的方法：定义图，编译它，让运行时执行它。

Pregel 是 Google 于 2010 年引入的大规模图处理系统。其核心思想——Bulk Synchronous Parallel（批量同步并行）——自然地映射到 agent 工作流：每个节点是一个计算，边定义依赖关系，在"超级步"之间由屏障同步状态。

LangGraph 的 Pregel 运行时是 deepagents、deer-flow 和 LangChain 的 create_agent 底层的执行引擎。当你调用这些框架的 agent.astream() 时，你运行的并非 while 循环——而是每次一个超级步地遍历图。理解这个运行时，就是理解现代 agent 生态系统半数以上运行的"操作系统"。

从循环到图

课程一中，我们看到了四种循环架构：

A：嵌套双循环
pi-mono, openclaw
B：扁平 While-True
claude-code, hermes-agent
C：状态机
nanobot, codex
D：委托式图
deepagents, deer-flow

架构 D 委托给 LangGraph。那么 LangGraph 实际做了什么？它将 StateGraph 定义编译为 CompiledStateGraph（继承自 Pregel），然后使用批量同步并行模型执行。每次迭代是一个超级步，包含三个阶段：

1. 规划（Plan）
prepare_next_tasks() 根据 channel 版本确定哪些节点被触发
→
2. 执行（Execute）
PregelRunner 通过 asyncio.gather 并发运行所有选定节点
→
3. 更新（Update）
apply_writes() 将结果提交到 channel，增加版本号

规划：prepare_next_tasks() 位于 pregel/_algo.py:392，消费待处理的 Send 任务（PUSH 任务）并检查 trigger_to_nodes 以获取 channel 驱动的触发器（PULL 任务）。对被触发的每个节点，_proc_input() 在第 1348 行根据当前 channel 状态构建该节点的输入。

执行：PregelRunner 位于 pregel/_runner.py:135，并发运行所有任务。同步循环使用 executor.submit()，异步使用 asyncio.gather()。不共享 channel 依赖的节点并行执行——这就是 BSP 中"批量同步"的含义。

更新：apply_writes() 位于 pregel/_algo.py:232，按路径排序任务，更新 versions_seen，将写入应用到 channel，增加 channel 版本号，并通知未更改的 channel。每个 channel 都有一个 reducer（归并器）来控制并发写入如何合并。

Channel：状态原语

在课程一的 agent 循环中，状态是一个可变对象——struct、dict、消息列表。在 LangGraph 中，状态分布在 channel 之中。每个 channel 都有一个类型，定义其合并行为：

ChannelReducer用途
LastValue最后的写入胜出系统提示、模型名称、标志位
Topic追加到列表消息历史（配合 add_messages reducer）
EphemeralValue读取一次后清除跳转信号、瞬态标志
BinaryOperatorAggregate应用二元运算（求和、取最大值等）Token 计数器、迭代计数
DeltaChannel每 N 步快照一次大消息历史（O(N²)→O(N) checkpoint）

DeltaChannel 架构上最有趣。朴素的 checkpoint 会在每一步序列化整个消息列表——O(N²) 存储。DeltaChannel 每 50 步才快照一次，并通过向上走回最近的快照并重放写入来重建中间状态。这正是 deepagents 在 graph.py:67 中为 DeepAgentState.messages 使用的机制。

条件边：图的决策点

StateGraph 分两步构建。首先用 add_node("name", callable)（graph/state.py:662）添加节点。然后连接它们：add_edge("a", "b") 用于无条件转换，add_conditional_edges("a", branch_fn, path_map)（第 969 行）用于分支。条件边是 LangGraph"图即程序"性质显现的地方。branch_fn 接收当前状态并返回一个字符串——下一个要执行的节点。这就是标准 agent 图的工作原理：

# 每个基于 LangGraph 的 agent（deepagents、deer-flow、LangChain create_agent）
# 都编译成这个拓扑结构：

StateGraph(AgentState)
  .add_node("model", call_llm)         # 思考
  .add_node("tools", ToolNode(tools))  # 行动
  .add_edge(START, "model")
  .add_conditional_edges(
      "model",
      lambda state: "tools" if state["messages"][-1].tool_calls else END,
      {"tools": "tools", END: END}
  )
  .add_edge("tools", "model")          # 观察 → 回到思考
  .compile()

compile() 方法位于 graph/state.py:1164，验证图、解析 channel，将每个节点包装为 PregelNode（带有 channel 读/写器），并返回 CompiledStateGraph。此后 invoke() / stream() / astream() 都通过 Pregel 循环，提供三项命令式循环没有的能力：独立分支的免费并行、确定性重放和无需手写保存/恢复逻辑的框架级持久化。代价是不透明——调试需要理解 channel 版本系统和触发图。

设计启发：如果你的 agent 有复杂的条件分支或超过三个工具，图模型通过 channel/reducer 模式所防止的 bug，比它在调试中增加的复杂度要多。简单线性循环则用 while 循环更直接。
