第4课：权限模型

从"完全信任 LLM"到"什么也不信任" — 七个 agent 代码库如何决定一个工具能做什么、不能做什么。

2026年6月 · 7个主要源代码库 · 永久链接

AI agent 中的权限模型回答两个问题：LLM 能请求这个工具吗？以及harness 能安全执行它吗？这是两个不同的问题，代码库在生命周期的不同时间点给出答案。钩子、ACL 和沙箱之间的区别在于检查何时发生以及谁来执行。

在第2课中，我们看到工具调度是每个 agent 循环中的中心阶段。本课深入到守卫该调度的权限检查 — 每个代码库如何防止 LLM 做不该做的事。

权限光谱

每个代码库都位于这条连续谱的某个位置：

无权限 完全沙箱化
  1. 无权限： nanobot、hermes-agent — 工具无条件运行。任何守卫（check_fn、SSRF 检测）是可用性或安全网，而非安全边界。
  2. 基于钩子的委托： pi-mono、openclaw — 框架提供一个 beforeToolCall 钩子；调用者实现其想要的任何权限逻辑。框架本身是中立的。
  3. ACL（访问控制列表）： claude-code、opencode — 每个工具有一个权限级别（allow、ask、deny）。规则在注册时（LLM 能看到什么）和执行时（实际运行什么）评估。
  4. 完全沙箱化： codex — OS 内核强制边界。工具在 Landlock、Windows 沙箱或 macOS 沙箱内运行。沙箱可以在重试时升级。

Claude Code — 权限钩子 + ACL

Claude Code 拥有七个代码库中最完善的 ACL 系统。权限检查是每个工具上一等的生命周期方法。

逐工具权限检查

src/Tool.ts 中的每个工具都有一个 checkPermissions() 方法。buildTool() 工厂提供一个安全的默认值 — 如果工具作者不覆盖它，工具默认为询问用户。这意味着新工具默认是安全的。

// claude-code: src/Tool.ts — 每个工具自带权限检查
buildTool({
  name: ToolNames.BASH,
  // ... schema、调用处理器 ...
  // buildTool 为以下提供默认值：
  // checkPermissions(input, context) — 默认：询问用户
  // isConcurrencySafe、isReadOnly、isEnabled、
  // validateInput、renderToolUseMessage、renderToolResultMessage
})

权限过滤管道

到达 LLM 的工具列表不是全部已注册工具的集合。依次应用三个过滤器：

// 合并内置工具 + MCP 工具并进行去重。

权限级别与持久化

每个工具可设为三种级别之一：allow（不询问直接运行）、ask（每次提示用户）或 deny（直接阻止）。--dangerously-bypass-permissions 标志绕过所有检查 — 为自动化环境提供的显式逃生舱。

权限决定在三个作用域中持久化：会话级（当前调用有效）、项目级（存储在 .claude/settings.json）和用户级（存储在 ~/.claude/settings.json）。这给用户对权限授权持续时间的精细控制。

Codex — 多层沙箱

Codex 是唯一一个拥有真正沙箱方案的代码库。当 Claude Code 问"应该运行这个工具吗？"，Codex 问"操作系统能强制这个工具无法逃脱吗？"

编排器管道

codex-rs/core/src/tools/orchestrator.rs 中的 ToolOrchestrator 使每个工具通过一个四阶段管道：

┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ 审批 │ → │ 沙箱 │ → │ 执行 │ → │ 重试 │ │ APPROVAL │ │ SELECTION │ │ EXECUTE │ │ （提升沙箱） │ │ │ │ │ │ │ │ │ │ 用户提示 │ │ Landlock / │ │ 在沙箱内 │ │ 如果沙箱 │ │ 或自动 │ │ Win / Mac │ │ 运行工具 │ │ 阻止，以更高 │ │ 批准 │ │ 沙箱选择 │ │ │ │ 级别重试 │ └──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘

审批： 执行前，编排器检查工具是否需要用户批准。网络访问有自己独立的审批系统，支持两种模式：立即（现在询问）和延迟（允许工具运行但阻止网络使用直到批准）。

沙箱选择： 编排器为平台选择适当的沙箱：Linux 上的 Landlock（内核级文件系统限制）、Windows 沙箱或 macOS 沙箱。每个沙箱有受限和提升两种变体。

提升重试： 这是其独特功能。如果工具失败是因为受限沙箱阻止了某个操作（例如在 workspace 外写入文件），编排器可以以提升的沙箱重试 — 用户无需重新提示。沙箱变成了动态属性而非静态墙壁。

并行执行安全

codex-rs/core/src/tools/parallel.rs 定义了 ToolCallRuntime，它管理并发的工具执行。每个工具的沙箱是独立的，因此并行工具不能互相干扰 — 这是基于钩子和基于 ACL 的系统无法保证的属性。

Opencode — 基于 ACL

Opencode 使用 Effect-TS 将权限规则编码到类型系统中。权限检查在两个时间点发生：物化时（工具定义发送给 LLM 之前）和执行时（工具实际运行时）。

子 agent 权限派生

// opencode: packages/opencode/src/agent/subagent-permissions.ts
// deriveSubagentSessionPermission():
//   父级 deny 规则 + 子 agent allow 规则
//   = 有效的子 agent 权限集
//
// 子 agent 只能使用满足以下条件的工具：
//   1. 父级 NOT 明确 deny，AND
//   2. 子 agent 自身的 allow 规则允许

工具执行包装

packages/core/src/tool/tool.ts 提供了 withPermission()，它包装工具执行。包装器在工具的 execute() 被调用之前检查权限规则。如果被 deny，工具永远不会运行 — LLM 收到一个错误结果。

packages/core/src/tool/registry.ts 在物化时 — 即工具列表发送给 LLM 之前 — 应用权限规则。如果一个工具被 deny，LLM 甚至无法在 function list 中看到它。这是一个关键的设计选择：如果 LLM 不能请求它，它就不能社交工程用户去批准它。

危险工具的默认 deny

两个工具在子 agent 中默认 deny：todowrite 和 task。Wildcard.match 实用工具用于工具名称的 deny 规则匹配，支持 glob 模式，以便单条规则可以覆盖工具族（例如 mcp__* 拒绝所有 MCP 工具）。

Pi-mono、OpenClaw — 基于钩子的委托

pi-mono 和 openclaw 都不自带内置权限系统。相反，框架在 agent-loop.ts 中公开一个 beforeToolCall 钩子。调用者 — 嵌入 agent 框架的应用程序 — 通过提供钩子函数来实现权限检查。

// pi-mono / openclaw: packages/agent/src/agent-loop.ts
// 框架在调度前调用 beforeToolCall(toolCall, context)。
// 如果钩子抛出异常，工具被跳过，错误被包装为
// 带 isError: true 的 ToolResultMessage。
//
// 框架自身对什么被允许没有任何意见。
// 所有权限逻辑存在于消费者应用程序中。

当工具抛出异常（无论是来自权限钩子还是工具自身），agent 循环捕获该异常并将其包装为错误结果消息。LLM 看到该错误并可以决定如何继续。这意味着权限模型优雅降级 — 被拒绝的工具只是 LLM 可以推理的另一个错误。

Hermes-agent、Nanobot — 极简 / 无

这两个 Python 代码库有最薄的权限层。将它们纳入光谱是为了展示"无权限模型"在实践中是什么样子。

Hermes-agent：check_fn 作为可用性守门

Hermes-agent 注册表中的每个 ToolEntry 可以有一个 check_fn — 但这是一个可用性守门，而非安全边界。它回答"这个工具的依赖是否可用？"（例如 Docker 是否在运行？），而不是"LLM 是否被允许使用这个工具？"。逐工具的 max_result_size_chars 设置是一个资源限制，而非权限检查。

# hermes-agent: tools/registry.py:77
class ToolEntry:
  name: str
  check_fn: Callable | None  # "Docker 安装了吗？" — 而非"LLM 能用这个工具吗？"
  max_result_size_chars: int # 资源上限，非安全措施

Nanobot：SSRF 检测与 workspace 边界

Nanobot 的 runner.py 有两个嵌入在工具执行中（而非权限层中）的安全检查：

  • SSRF 检测： 在工具发起 HTTP 请求之前，nanobot 检查目标是否为私有 IP。如果是，工具被阻止，LLM 收到一个递进提示。
  • Workspace 边界违规： workspace 根目录外的文件操作被检测到，并用递进提示分类 — 从温和的"你在 workspace 之外"到硬性阻止。

这些被归类为递进提示，而非权限拒绝。LLM 被轻推而非被阻止 — 一个与基于 ACL 或基于沙箱的方法根本不同的哲学。

核心权衡：安全 vs. 复杂度

权限模型是安全保障与实现复杂度正面交锋的地方：

更多安全保障 = 更多代码、更多故障模式、更多用户摩擦。 Codex 的沙箱提升意味着工具可以自动重试，但它需要平台特定的沙箱实现（三种不同的 OS 沙箱，每个有自己的 API 和故障模式）。Claude Code 的三级作用域权限持久化给用户精细的控制，但也意味着权限状态可能令人困惑 — "为什么这个工具又问我？我允许的是会话还是项目？"

更少安全基础设施 = 更简单的代码，但用户（或嵌入应用程序）承担所有风险。 Pi-mono 的基于钩子的方法在框架中有零行权限代码。那是优雅的 — 直到 LLM 运行 rm -rf / 而没有人连接钩子。

设计启发： 在注册/物化时的权限检查防止 LLM 请求它不能做的事情。在执行时的权限检查保护 harness 免受 LLM 做不该做的事。最强的系统（claude-code、opencode、codex）两者都做。最弱的（hermes-agent、nanobot）两者都不做 — 它们用可用性守门或用提示轻推，信任 LLM 会守规矩。根据你的威胁模型选择光谱上的位置：如果 LLM 能访问 shell，执行时强制不是可选项。
