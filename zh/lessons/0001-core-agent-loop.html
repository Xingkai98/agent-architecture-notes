第1课：Agent 核心循环

思考、行动、观察、重复——每个 AI agent 的基础架构。

2026年6月 · 8 个主要参考来源 · 永久链接

打开任意一个 agent 代码库。第一个要问的问题是：循环在哪里？每个 agent——无论是在终端中运行、在 Slack 频道中、还是在 Docker 容器中——都围绕同一个架构重心构建：在调用 LLM、执行工具和将结果回传之间循环的执行循环。

CoALA（Sumers 等人，2023）是第一篇将其形式化为认知架构的论文：一个具有模块化记忆存储和结构化动作空间的通用决策循环。请参阅 arXiv:2309.02427。

本课逐行映射了七个生产级 agent 代码库中的循环——四个编程 agent（claude-code、codex、opencode、pi-mono）和三个通用 agent（hermes-agent、openclaw、nanobot）。我们将看到两种基本架构浮出水面，并理解推动团队选择其中一种的权衡。

通用骨架

剥离 CLI、聊天频道、插件和沙箱。每个 agent 循环都归结为：

┌─────────────────────────────────────────────────────┐ │ while (should_continue): │ │ │ │ // THINK phase │ │ response = await llm.stream({ │ │ system: system_prompt, │ │ messages: conversation_history, │ │ tools: available_tools, │ │ }) │ │ │ │ // ACT phase │ │ if response.has_tool_calls(): │ │ for each tool_call: │ │ result = execute(tool_call) │ │ conversation_history.append(result) │ │ │ │ // OBSERVE (results already in history) │ │ // Loop continues — next LLM call sees results │ │ │ │ if response.is_text_only(): │ │ return response.text // terminal state │ │ │ └─────────────────────────────────────────────────────┘

这就是 ReAct 模式（Yao 等人，2022）——推理与行动交错进行。LLM 推理要做什么，agent 去执行，结果被回传用于下一轮推理。仅此而已。

但仔细审视七个真实实现，这个简单的循环分化为两种根本不同的架构。

两种架构

架构 A：嵌套双循环

使用者：pi-mono、openclaw（opencode 也可能有部分继承）。

外层循环处理后续消息（用户中断、在 agent 原本将停止后到达的排队命令）。内层循环运行实际的思考-行动-观察回合：

// pi-mono: packages/agent/src/agent-loop.ts, runLoop() // openclaw: packages/agent-core/src/agent-loop.ts, runLoop() // // (这两个实现结构完全相同——openclaw 的是 fork 版本。) while (true) { // OUTER: follow-up loop while (hasMoreToolCalls // INNER: tool turn loop || pendingMessages > 0) { message = await streamAssistant() // THINK toolCalls = executeTools(message) // ACT hasMoreToolCalls = !allTerminate // OBSERVE → decide } followUps = getFollowUpMessages() if (followUps.length === 0) break // terminal state pendingMessages = followUps }

为什么采用这种设计？后续消息是一种常见模式：用户在运行中途打断（"不行，换个方法"），或者子 agent 完成后其结果需要被注入。通过将后续消息处理分离到自己的外层循环中，内层循环可以纯粹专注于思考-行动-观察周期，无需在每次迭代时检查外部中断。

pi-mono 和 openclaw 共享这种架构，因为 openclaw 的 agent-core 包是 pi 的 agent 包的直接后裔。代码谱系很重要。

架构 B：扁平 While-True 循环

使用者：claude-code、hermes-agent。单个 while (true) 循环，将所有关注点——工具执行、后续消息注入、恢复——作为单次迭代内的分支处理。

// claude-code: src/query.ts, queryLoop() // hermes-agent: agent/conversation_loop.py, run_conversation() while (true) { // Phase A: Context prep (compaction, budget checks) prepareContext(state) // Phase B: THINK — stream LLM response for await (event of streamModel(state)) { if (event.type === 'tool_use') startToolExecution(event) // may run during streaming } // Phase C: No tools? → check stop hooks, token budget, return if (!needsFollowUp) { runStopHooks() if (ok) return { reason: 'completed' } } // Phase D: ACT — execute remaining tools if (needsFollowUp) { await executeAllTools() // Phase E: OBSERVE — collect results, refresh, continue state = nextState(state, toolResults) continue // back to top of while(true) } }

为什么采用这种设计？当恢复逻辑非常复杂时——提示过长、max_output_tokens、响应式压缩、模型回退、中止处理——扁平循环加上显式阶段比嵌套循环更易于推理，因为嵌套循环的不同恢复路径在不同层级退出。Claude Code 的循环有十二种不同的终止原因（query.ts 第 1050–1720 行）；每个都是扁平序列中正确位置的 return { reason: '...' }。

第三种变体：分层状态机

使用者：nanobot。nanobot 不是使用嵌套循环或扁平 while-true，而是使用具有 7 个状态的显式状态机（RESTORE → COMPACT → COMMAND → BUILD → RUN → SAVE → RESPOND → DONE）。在 RUN 状态中，一个单独的 for iteration in range(max_iterations) 循环处理思考-行动周期。

使用者：codex（Rust）。四个不同的循环层级——submission_loop、task loop、turn loop、sampling loop——每个都有各自的终止条件。turn loop 调用 run_sampling_request()，后者在内部流式传输 LLM 响应，通过 FuturesOrdered 清空进行中的工具 future，并确定 needs_follow_up。

使用者：opencode。使用 Effect-TS fibers 而非原始的 async/await。run() 函数是一个 Effect.fn 生成器，在 while (openActivity) 外层守卫内部有一个 for (let step = 0; step < MAX_STEPS; step++) 循环。

根本权衡：嵌套深度

为什么 codex 有 4 层循环而 claude-code 只有 1 层？这不是偶然——它反映了不同的优先级：

更多嵌套 = 更多关注点分离。Codex 的四层各自处理一件事：submission_loop 管理层会话生命周期，task loop 管理任务生命周期（中止/重启），turn loop 管理思考-行动循环，sampling loop 管理重试。每一层都可以独立测试和推理。

更少嵌套 = 更容易看到所有终止路径。Claude Code 的单个扁平循环使全部十二种退出条件在一个函数中可见。当恢复逻辑像 Claude Code 中那样复杂时（提示过长 → 折叠耗尽 → 响应式压缩 → 升级最大 token → 注入恢复消息 → 最终返回错误），扁平化循环使得更难遗漏某个恢复路径。

设计启发：如果恢复逻辑简单（错误时重试、信号时中止），嵌套循环工作良好。如果恢复逻辑涉及跨多个维度的级联回退（上下文大小、token 预算、模型可用性、停止钩子），则将循环扁平化，使所有路径在一个地方可见。

另一个关键维度：工具何时执行？

两个阵营：

  1. 批量执行（pi-mono、openclaw、nanobot、hermes-agent）：等待 LLM 流完成，收集所有工具调用，然后执行。实现简单，但本可以在 5 秒前就开始运行的工具却闲置等待模型完成生成。
  2. 流式执行（claude-code、codex）：在模型在流中一发出工具调用时就立即执行。工具与模型响应的其余部分并发运行。减少延迟，但需要仔细的状态管理——当工具结果在模型完成流式输出之前到达时会发生什么？

Claude Code 的 StreamingToolExecutor 在 tool_use block 的 content_block_start 触发时立即启动工具，但跟踪进行中的结果，以便在回合周期的适当时机将其返回给循环。

这关联到一个更深层次的设计问题——关于工具执行模型本身——在第2课：工具系统中详述。
