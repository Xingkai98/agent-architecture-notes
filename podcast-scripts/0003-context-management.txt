Lesson 0003: Context Management Lesson 0003: Context Management How AI agents manage their context window — compaction, token budgets, and the tree model. June 2026 · 7 primary source codebases · In Lesson 0002 , we covered tool definition through normalization. This lesson zooms into the opposite direction: how agents squeeze tool results and conversation history back down to fit the context window. The four fundamental strategies Across the seven codebases, four distinct compaction strategies emerge: Proactive (before the call) Reactive (after the error) Proactive compaction (claude-code, hermes-agent): Check token count before every API call. If over threshold, compact first. The safest approach — the LLM never sees an overfull context. Reactive compaction (pi-mono, openclaw, hermes-agent): Wait for the provider to return a 413 prompt-too-long error, then compact and retry. Simpler to implement but wastes one API call. Per-turn consolidation (nanobot): Run a lightweight consolidation check every turn; if needed, run up to 5 rounds of summarization before proceeding. Delegate-to-provider (opencode, codex): Leave compaction to an external package () or configurable shell hooks. No built-in LLM summarizer. The most sophisticated: claude-code's 7-layer approach Claude Code has the most elaborate context management of all seven codebases. Seven distinct layers work together: Layer 1: Token budget (gatekeeper) Before every API call, checks whether the conversation fits. The threshold is 90% of the context window (). It also detects diminishing returns : if 3 or more continuations each add less than 500 tokens of new content, the system forces a compaction — the conversation is spinning without progress. The diminishing-returns heuristic is subtle but important. Without it, an agent that keeps making tiny edits to the same file would never trigger compaction by token count alone — each turn adds little. But across 10+ turns, the context fills with near-duplicate tool results. The heuristic catches this pattern: "you're not adding meaningful new information, so compact now." Layer 2: Context window resolution — resolves the active model's context window. It checks for the suffix (denoting million-token models), consults model capabilities, and defaults to 200K tokens. This is the foundation that all other layers reference. Layer 3: Token counting — walks backward from the last known API usage point. Everything after that is estimated rather than precisely counted. This gives a fast, reasonably accurate count without re-counting everything from scratch. Layer 4: Auto-compact (proactive) — before every API call, the system checks the token threshold. If exceeded, it calls which runs summarization and produces a compacted conversation. This is the primary compaction path. Layer 5: Reactive compact — if auto-compact somehow misses the trigger (race condition, unexpected token surge), the reactive layer catches 413 prompt-too-long errors. It runs a single-shot compaction and retries. There's no retry loop — if reactive compaction fails to reduce enough, the error propagates. Layer 6: Micro-compact — before the main compaction runs, micro-compact collapses identical consecutive tool results. If a tool was called 5 times with the same result, only 1 entry is kept. This often reduces token count enough to avoid full summarization. Layer 7: Context collapse, session memory, and snip Three specialized compaction types: Context collapse (): produces a projected read-time view using a commit-log model — each collapse is a "commit" of the state, and the LLM sees a summary rather than raw history. Session memory compact (): experimental, uses session memory data for a higher-quality summary. Snip compact (): removes hidden or intentionally-snipped message content before the conversation is sent. System prompt composition — merges prompts from 5 sources in priority order: . Each layer can add, override, or remove sections. This isn't compaction per se, but it controls the fixed token overhead at the top of every conversation. Why this matters: the system prompt is fixed overhead that sits at the top of every API call. A 10K token system prompt on a 200K window with 90% threshold leaves only 170K for actual conversation — the system prompt consumes 5% of the available budget permanently. Layered composition with overrides lets different agent modes (coordinator, sub-agent, standalone) tune this overhead. The session-tree model (pi-mono, openclaw) pi-mono and openclaw share a fundamentally different data model: history is a tree of typed entries, not a flat array. Key properties of this model: Compaction is recorded, not destructive: compacted entries remain in the tree. They're just not included in the active context. You can "uncompact" if needed. : — always preserves room for the LLM response and the most recent turns. Split-turn compaction: when the token cut point falls in the middle of a tool-call turn (some tools returned, some didn't), both the prefix and suffix of that turn are summarized separately. The model never sees a half-finished tool cycle. File operation tracking: file operations (reads, writes, edits) are tracked across sessions and appended to compaction entries, giving the summary awareness of which files were touched. Iterative summary updates: merges new messages into an existing summary rather than re-summarizing from scratch. This is cheaper and preserves continuity. The 3-layer memory context (hermes-agent) Hermes-agent combines three sources into a single context: The at runs a 5-phase algorithm: Prune tool results: trim verbose tool output before anything else. This is the cheapest reduction and often sufficient on its own. Determine boundaries: find the cut point — which messages stay as raw text and which get summarized. Search for prior summary: if a summary already exists, retrieve it so the compressor can update rather than re-create. Generate LLM summary: call the LLM to produce a concise summary of the pruned entries. Assemble: combine system prompt + summary (reference-only prefix) + recent messages into the final context. Key heuristics in hermes-agent's approach: Anti-thrashing: skips compaction if the last 2 compressions each saved less than 10% of tokens. Compaction has a cost — if it's not paying off, stop doing it. Reference-only prefix: the summary is prefixed with explicit instructions that it is background information , not active instructions. This prevents the model from treating stale summaries as commands. Fallback deterministic summary: if the LLM summarization call fails, the compressor falls back to a rules-based summary (extract first sentence of each message, cap at 8K chars). Preflight compression: runs before the tool loop starts, not during. Up to 3 passes. This means the agent never starts a turn with an overfull context. File reference system with hard limits (hermes-agent) Hermes-agent's system supports , , , and directives. References expand to their content, but subject to a 50% context window hard limit — no reference can consume more than half the available context. This prevents a single large file from crowding out conversation history. The 50% limit is not arbitrary. It guarantees that conversation history always has at least as much room as injected references, which means the model can always "think about" the injected content with equal-weight history context. A 90% limit would let references dominate; a 20% limit would make references too small to be useful for large files. 50% is the balanced split. The directive is the most dangerous of the four — recursively expanding a directory tree can pull in thousands of files. Hermes-agent handles this by expanding entries with file-size estimates first, sorting by relevance (using the FTS5 index), and stopping when the 50% budget is exhausted. The explicit state machine (nanobot) Nanobot makes context management a first-class state in its turn loop: Each component has a clear role: (): idle-session TTL-based archiving. If a session has been idle, it keeps the last 8 messages and summarizes the rest. This is orthogonal to per-turn consolidation — it handles the case where the user walks away and comes back hours later. (): runs per-turn. If the context exceeds the budget, it runs up to 5 consolidation rounds, each round calling the LLM to compress the oldest uncompressed messages. The multi-round design handles the case where a single summarization pass doesn't reduce enough — it keeps going until the context fits. (): assembles the final context in this order: identity → bootstrap files → tool contract → memory → skills → history (capped at 120 messages) → session summary. The order matters: identity and tool contract are fixed overhead; history is the most volatile and the first thing to get capped or summarized. Hook-driven and delegate-based approaches Two codebases take a fundamentally different approach — they don't implement compaction themselves: codex (hook-driven): Compaction is driven by configurable hooks in . The and structs carry context about what to compact and what was compacted. The actual summarization is done by user-configured shell commands — codex has no built-in LLM summarizer. Context fragments are delimited by typed markers so the hook scripts can identify sections. This design is philosophically consistent with codex's approach to sandboxing (Lesson 0002): the framework provides the event and the contract , but the policy is entirely user-defined. A carries the compacted result and the token counts before/after, letting the hook decide whether to accept or redo the compaction. opencode (delegate): Compaction is delegated entirely to the external package and to the provider level. OpenCode itself doesn't run summaries — it relies on the provider's native context handling. This keeps the OpenCode codebase small, but means compaction behavior is opaque and non-portable across providers. The tradeoff is clear: codex and opencode have zero lines of summarization code to maintain, debug, or security-review. But they also have zero control over summarization quality. If the hook script or provider produces a bad summary, there's no fallback and no recourse. Why the tree model matters The difference between a flat array and a tree is not just a data structure choice — it determines what operations are possible: Operation Flat array (claude-code, hermes-agent, nanobot) Tree (pi-mono, openclaw) Undo compaction Must keep old array copy or re-request Change which nodes are "active" — no data loss Branch from mid-conversation Copy the array up to that point Add a new branch node — O(1) Merge branches Manual reconciliation of two arrays Walk both subtrees, produce merged summary node Re-summarize with different granularity Can't — the original messages are gone Delete old compaction node, create new one from preserved children pi-mono and openclaw chose the tree model primarily to support branching — the same conversation can fork into multiple directions. But the secondary benefit (undoable, redoable compaction) turned out to be just as valuable. Claude Code and hermes-agent both check tokens before calling the LLM. This avoids wasted API calls and gives the user a smoother experience. Reactive only makes sense when the provider controls the window and you can't predict where the limit is. If you have multiple context sources, the tree model is cleaner than flat arrays. pi-mono's session tree stores messages, compactions, and branch summaries as typed nodes. This makes it trivial to reconstruct context for any point in the session, undo compactions, or fork branches. Flat arrays force you to mutate or duplicate history. User files (MEMORY.md, USER.md) must be "reference-only" in summaries. Hermes-agent's reference-only prefix is a critical safety measure: compacted MEMORY.md content should inform the model but never override the user's current instructions. Blending memory into the system prompt without this distinction causes the agent to follow stale directives. Tool output pruning before LLM summarization significantly reduces compaction costs. Every codebase that does compaction prunes tool outputs first. Hermes-agent's phase 1 (prune) and claude-code's micro-compact both do this. LLM summarization is expensive — trimming verbose tool output first pays for itself immediately. What happens when compaction fails? Every compaction system can fail — the LLM summarizer returns garbage, the token count doesn't drop enough, or a bug corrupts the summary. How each codebase handles this: claude-code: if reactive compaction fails to reduce context below the limit, the error propagates to the user. The single-shot design means there's no infinite retry loop. hermes-agent: the deterministic fallback summary (8K chars, rule-based extraction) guarantees some summary is produced even if the LLM is down. The anti-thrashing check also prevents the compressor from burning tokens on ineffective compactions. pi-mono / openclaw: if the LLM summarization fails, the compaction entry is not created and the original messages stay in the tree. The agent proceeds with an overfull context — which may trigger a provider 413 error on the next call, but at least the data is preserved. nanobot: the multi-round consolidation (up to 5 rounds) provides a natural backstop. If after 5 rounds the context still doesn't fit, the oldest messages are truncated (not summarized) as a last resort. codex: hook scripts control failure handling entirely. The carries before/after token counts — a well-written hook can detect insufficient reduction and retry with different parameters. opencode: failure handling is entirely the provider's responsibility.
