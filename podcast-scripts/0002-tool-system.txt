Lesson 0002: Tool System — define, register, dispatch, normalize Lesson 0002: Tool System Define, register, dispatch, normalize — the four stages every agent tool passes through. June 2026 · 7 primary source codebases · In Lesson 0001 , we saw that every agent is built around a loop. This lesson zooms into the inner engine of that loop: how tools work . We'll trace all four stages across all seven codebases, and identify the three fundamental design philosophies that divide them. The four stages Every tool interaction follows this pipeline: Three design philosophies Across the seven codebases, three distinct philosophies emerge: Maximally minimal Fully orchestrated Minimalist: pi-mono, openclaw, nanobot. Tools are simple objects with an execute function. No central registry or discovery machinery — tools live in arrays or dicts. The framework doesn't even ship with built-in tools; tools are provided by the consumer. Type-safe / DI: opencode (Effect-TS). Tools are defined with runtime-validated schemas, registered via dependency injection into scoped containers, and dispatched with automatic cleanup. The type system guarantees correctness. Fully orchestrated: claude-code, codex, hermes-agent. Tools go through a multi-stage pipeline: permission checks, sandbox selection, execution, retry, result normalization, and (for claude-code) UI rendering. Each stage is a separate concern with pluggable implementations. Stage 3: Dispatch — how tools are executed Execution timing As covered in Lesson 0001, two camps: Batch (after stream): pi-mono, openclaw, nanobot, hermes-agent — wait for LLM stream to complete, collect tool calls, execute together Streaming (during stream): claude-code — executes tools as the LLM emits them Hybrid: codex — drains in-flight tool futures during streaming; opencode — configurable Concurrency model Every codebase has a concept of which tools can run together: Codebase Concurrency Model Implementation pi-mono Sequential or Parallel (Promise.all) / openclaw Sequential or Parallel (Promise.all) Same pattern as pi-mono, per tool claude-code Concurrent-safe tools parallel, others serial partitions by hermes-agent Sequential (sync context + async bridge) handles coroutines with per-thread event loops nanobot Sequential with flag / properties codex Parallel via Tokio + FuturesOrdered gates concurrent execution opencode Sequential settling via Effect fiber Each call is its own Effect Sandboxing Only one codebase has a serious sandboxing story: codex: The (orchestrator.rs) provides a multi-layered sandbox: Landlock (Linux), Windows sandbox, macOS sandbox, elevated/restricted token. On sandbox denial, the orchestrator can retry with escalated sandboxing — if a restricted sandbox blocks a file write, the tool can be retried with the elevated sandbox. Network access approval is a first-class concept with immediate and deferred modes. All other codebases delegate sandboxing to external mechanisms: pi-mono/openclaw use hooks, hermes-agent uses environment-based terminals, claude-code uses permission checks on tool access (not execution sandboxing). Stage 4: Normalize — how results feed back After execution, tool results must become messages the LLM can consume. This is where the most divergence occurs: Size truncation (all): Every codebase has a maximum result size. Claude Code's persists large results to disk and replaces them with truncated previews. Hermes-agent's truncates per-tool. Nanobot exempts from offloading to prevent persist-to-disk → read-from-disk loops. Error normalization (all): All codebases catch tool exceptions and convert them to error messages that the LLM can understand. Codex has an entire with approval, sandbox selection, and retry escalation. Why? More tool infrastructure = more safety guarantees, but more code to maintain. Codex's sandbox retry logic means a tool that fails due to permission denial in the restricted sandbox can automatically retry with elevated permissions — the user doesn't need to re-run. But it also means a multi-stage pipeline with 10+ files just for tool orchestration. Less tool infrastructure = more flexibility, but more burden on the consumer. pi-mono gives you and says "go." You can implement any tool system you want. But you also have to — the framework won't help you with permission checks, sandboxing, result normalization, or error recovery. Design heuristic: If you control the execution environment and trust your tools, start minimalist — an array of objects with an execute function. If you need sandbox isolation, permission gating, and observability, the orchestrated approach (codex, claude-code) pays off — but expect the tool system to become the largest subsystem in your codebase.
