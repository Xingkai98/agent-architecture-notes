Lesson 0007: UI / Transport Layers — TUI, client/server, multi-channel Lesson 0007: UI / Transport Layers TUI frameworks, client/server splits, and multi-channel adapters — how agents talk to the world. June 2026 · 7 primary source codebases · In Lesson 0002 , we saw how tools are defined, registered, dispatched, and normalized. This lesson moves one layer up: how those tool results reach the user , and how the user's input reaches the agent. We'll survey the full interface paradigm spectrum, compare six TUI implementations in detail, examine client/server architectures, and map the multi-channel landscape. The interface paradigm spectrum Agents expose themselves to users along a spectrum of increasing complexity and reach: REPL/TUI API REPL / TUI: The agent is a terminal application. The user types, the agent responds, the loop is visible. Claude Code, Pi-mono, Codex, and Opencode all live here. This is the dominant paradigm for developer-facing agents. CLI one-shot: The agent runs a single turn and exits. Input comes from args or stdin, output goes to stdout. Useful for scripting and CI. Codex and Pi-mono both support this mode. Daemon: The agent runs as a long-lived process, often headless. Codex's and OpenClaw's gateway both operate this way — the agent process is separate from the interface. Multi-channel: One agent, many surfaces. OpenClaw (24 channels) and Nanobot (18 channels) are the exemplars here. A single agent loop can receive input from Telegram, Discord, Slack, Feishu, and Matrix simultaneously. API: The agent exposes an HTTP endpoint (often OpenAI-compatible ). Nanobot's is the clearest example — a thin aiohttp wrapper that lets any HTTP client drive the agent. TUI technologies compared Six codebases implement terminal UIs, but they make radically different technology choices. The framework selection is not just an implementation detail — it determines how streaming output renders, how scrolling works, how key bindings are defined, and how much maintenance burden the project carries. Claude Code: Custom React + Ink fork Claude Code's TUI is built on a custom fork of Ink (React for terminals). The fork lives at with a 251KB that includes a custom fiber reconciler and terminal layout engine. The two largest components — (147KB) and (148KB) — show how much UI logic a terminal agent accumulates. Virtual scrolling is mandatory: agent conversations produce hundreds of messages, and naively rendering them all would overwhelm the terminal. Codex: ratatui + crossterm Codex uses Rust-native terminal libraries: for widget-based rendering and for terminal control. The main TUI lives at and (58KB). Ratatui provides a retained-mode widget tree with layout constraints — a different paradigm from React's declarative diffing. Crossterm handles raw mode, cursor control, and input events. No Ink, no ratatui, no prompt_toolkit — just raw ANSI escapes and terminal IO. This gives maximal control and minimal bloat, at the cost of reimplementing every widget behavior. Hermes-agent: prompt_toolkit Hermes-agent uses Python's in , composing , , , and . Prompt_toolkit provides a mature widget system with syntax highlighting, auto-completion, and key binding DSL out of the box — no need to build from scratch. Opencode: OpenTUI + Solid.js Opencode's TUI at (36KB) is built on OpenTUI with Solid.js for reactive rendering. This is a novel combination: terminal UI with fine-grained reactive JSX. Solid's signal-based reactivity means only changed parts of the terminal output re-render, without a virtual DOM. The TUI can connect to a local worker (in-process) or a remote server over HTTP. This means the same agent can drive a terminal UI, a web UI, or an IDE plugin — all through the same RPC interface. Codex: App Server + TUI client Codex splits into (the agent process) and (the TUI). The two can run in-process (zero network overhead) or the TUI can connect to a remote app-server over HTTP. This is the same pattern as a language server: the server does the work, the client handles display. OpenClaw: Gateway protocol OpenClaw defines a formal gateway protocol at with a corresponding package. The protocol specifies message formats, authentication, and transport — any client that speaks the protocol can drive the agent. This is a step beyond Codex's ad-hoc client/server split: the protocol is versioned, documented, and reusable. Multi-channel (messaging bots) Two codebases go far beyond terminal UIs — they are messaging platforms first, agents second: OpenClaw: 24 channels OpenClaw supports 24 channels: Discord, Slack, Telegram, Feishu, Matrix, WhatsApp, Signal, IRC, iMessage, LINE, Zalo, and more. Channels are registered declaratively in each extension's . The channel plugin contract standardizes adapters for: configuration, authentication, outbound message sending, connection status, device pairing, and security policies. Hermes-agent: Desktop + CLI Hermes-agent takes a different approach: a desktop Electron app at built with React + Vite, plus a for terminal use. The desktop app provides a graphical chat interface while the CLI serves scripting and power users. This is the only codebase in the set that ships a GUI. Single-interface agents Four codebases commit to a single interface — no channels, no desktop app, no API server: Claude Code: Terminal-only (React+Ink). Supports structured IO for IDE integration (VS Code, JetBrains) but the agent itself is a terminal application. Pi-mono: Terminal-only (custom TUI). No API, no channels, no desktop. The terminal is the entire product surface. Codex: Terminal-only (ratatui) with a remote control daemon. The daemon enables headless operation, but user interaction is through the TUI. Opencode: Terminal-only (OpenTUI). The Web Worker architecture allows a web UI, but the primary interface is the terminal. Interesting design choices Four implementation details worth studying: Claude Code's Ink fork: The TUI IS the application. at 147KB is not a "view layer" — it is the core user experience. Tool results, streaming text, permissions prompts, and mode indicators all render through React components. The fork exists because no off-the-shelf terminal UI library handles streaming agent conversations at this scale. Pi-mono's from-scratch TUI: At 53KB with zero dependencies, Pi-mono's TUI is a statement. No framework churn, no upgrade treadmill, no breaking changes from upstream. The cost is that every feature — syntax highlighting, scrollback, resize handling — must be built and maintained. For a project that values minimalism, this tradeoff makes sense. OpenClaw's channel catalog: is generated at build time from extension manifests. This means the agent knows its available channels before starting — no runtime scanning, no dynamic imports, no race conditions. It is a small touch that eliminates an entire class of startup bugs. Nanobot's OpenAI-compatible API server: is a thin aiohttp wrapper that exposes . This means any tool that can call the OpenAI API — langchain, openai-python, curl — can drive Nanobot without knowing it is not talking to OpenAI. The agent becomes a drop-in replacement. Claude Code's VirtualMessageList (148KB) exists because naive rendering of 500+ messages freezes the terminal. Every TUI agent hits this eventually, regardless of framework. Heuristic 2: Multi-channel agents need a unified message bus — each channel is an adapter, not an agent. OpenClaw and Nanobot both converge on this pattern independently. The agent loop processes one message type; channel adapters translate to and from it. Don't write an agent per channel. Heuristic 3: The TUI framework choice is a multi-year commitment — custom forks give control but cost maintenance. Claude Code's Ink fork can't track upstream. Pi-mono's from-scratch TUI needs every widget built by hand. Ratatui upgrades require Rust refactors. Choose the surface of your maintenance burden carefully.
