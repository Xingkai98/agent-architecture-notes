June 2026 · 8 primary source codebases · In Lesson 0002 , we saw that tools are the engine of the agent loop. This lesson zooms out one level: when one agent becomes many . We'll trace sub-agent spawning, context isolation, resource budgeting, and result collection across all eight codebases, then extract the design heuristics that hold regardless of framework. The spawn spectrum Sub-agents exist on a continuum of isolation. The further right you go, the stronger the guarantee — and the higher the overhead: In-process Container No codebase in our set uses containers for ad-hoc sub-agents (though several use them for the host sandbox). The real battle is between in-process cloning and child-process spawning — and the choice shapes everything downstream. Three sub-agent modes Claude Code supports three distinct sub-agent patterns: Synchronous (blocking): The parent pauses until the child completes. Used when the parent needs the child's answer to continue. This is the default mode. Asynchronous (fire-and-forget): The parent dispatches the child and continues immediately. The child's result is delivered later via a callback or message bus. Used for parallel exploration tasks. Fork: At , the fork subagent inherits the parent's conversation context and uses a byte-identical API prefix. This allows the LLM provider to share prompt cache entries between parent and child — a significant cost optimization for deeply nested delegation. Swarm mode At , implements multi-agent orchestration using file-based mailboxes. Each agent writes to and reads from a designated file, creating an asynchronous message-passing system entirely within one process. The file system acts as a durable, inspectable message bus. Pi-mono — child process model Pi-mono takes the strongest isolation approach short of containers: each sub-agent is a separate OS process. Subagent extension At , the complete subagent extension is implemented as a standalone module. It demonstrates that sub-agents in pi-mono are extensions , not built into the framework core — the framework provides the spawn primitive (child_process), and the extension defines the delegation protocol. Three execution modes Single: Spawn one child, wait for completion, return the result. Parallel: Spawn up to 8 children with a concurrency limit of 4. Results are collected as they arrive. Failed children don't block siblings. Chain: Spawn children sequentially, each receiving the previous child's output via a placeholder in its prompt. Enables pipeline-style decomposition: "Find relevant files" → "Read those files" → "Summarize findings." Process-level isolation uses with and flags. The child writes JSON lines to stdout; the parent reads them as a stream. Per-task output is capped at 50KB to prevent any single child from flooding the parent with data. Opencode — child sessions Opencode models sub-agents as child sessions, reusing the same session machinery that powers the top-level agent. TaskTool At , the creates a child session via . The child session is a fully independent agent conversation with its own context window, tool set, and abort controller — but it shares the same process and the same LLM connection. Permission derivation At , the child's permission ruleset is derived from the parent's. This is not a simple copy: the derivation function strips permissions that are meaningless for sub-agents (e.g., if nesting is disabled) and adds restrictions (e.g., read-only file access for worker children). Foreground vs. background Opencode supports two execution modes directly analogous to Claude Code's sync/async split: Foreground (blocking): The parent waits for the child session to complete before continuing. Background (fire-and-forget): The child session runs independently; the parent can poll or be notified on completion. OpenClaw — ACP-based delegation OpenClaw uses the Agent Communication Protocol (ACP) as its delegation transport, making sub-agents addressable across process and network boundaries. Spawn parameters At , defines the spawn contract: Multi-agent routing OpenClaw's ACP layer provides 24 channels for inter-agent communication. Each agent subscribes to channels it cares about; the ACP router delivers messages by channel subscription, not by agent identity. This decoupling means agents can be added, removed, or replaced without reconfiguring the routing table. Every design decision in a sub-agent system flows from the isolation boundary. In-process cloning (claude-code, codex) is fast and cache-friendly but requires disciplined context separation. Process spawning (pi-mono) is safe by construction but pays IPC overhead on every delegation. Thread pools (hermes-agent) sit in the middle — safe-enough with Python's GIL, but the isolation is conventional, not enforced by the OS. "Budgets must be separate: a sub-agent shouldn't exhaust the parent's token allowance." Hermes-agent's per-child IterationBudget is the cleanest expression of this principle. Nanobot achieves it implicitly through asyncio cooperative scheduling. Claude Code relies on the parent to manage budgets (via AbortController linkage). The key failure mode is the same across all codebases: an unconstrained sub-agent burns tokens exploring dead ends, and the parent has no recourse except timeout-killing the child. "The fork pattern (claude-code, codex) trades isolation for cache efficiency." When a child inherits the parent's conversation prefix (fork_context), the LLM provider can reuse cached key-value pairs from the parent's inference. This saves substantial API cost — but it means the child can "see" the parent's conversation. If the parent was discussing sensitive data, the child inherits that data in its context. Forking is an optimization, not a default — use it when cache savings outweigh isolation concerns. Exercise: Sub-agent design decisions Q2.
