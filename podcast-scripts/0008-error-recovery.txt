Lesson 0008: Error Recovery — resilience patterns across AI agent codebases Lesson 0008: Error Recovery From crash to continuation — how 7 agent codebases classify, recover from, and survive failure. June 2026 · 7 primary source codebases · In Lesson 0002 , we saw tools as the engine of the agent loop. In Lesson 0003 , we saw how context is the loop's memory. This lesson covers what happens when the loop breaks — and how each codebase picks it back up. Error categories Every agent error falls into one of five categories. Each codebase handles them differently, but the categories are universal: Two poles of recovery philosophy Across the seven codebases, two poles emerge: Caller-delegated (pi-mono, openclaw) Full self-recovery (claude-code, hermes-agent) Caller-delegated: pi-mono, openclaw. The agent loop stops on error and signals the reason upward. Recovery — retry, restart, model switching — is the caller's responsibility. The framework provides hooks (, ) but no built-in retry logic. Full self-recovery: claude-code, hermes-agent, nanobot. The agent loop includes retry with exponential backoff, model fallback chains, context compaction on overflow, and checkpoint restore. The loop itself is a state machine with error transitions. Hybrid: codex, opencode. Some errors are handled internally (SQLite recovery, step limits), others are surfaced to the caller. Nanobot: Provider retry, length recovery, and checkpoint restore Nanobot implements recovery at three distinct layers: Provider retry — has two modes: Standard mode: 3 attempts with delays of 1s, 2s, 4s. For normal API calls. Persistent mode: Unlimited retries, capped at 60s delay. Used for long-running agent sessions where the user expects the agent to keep trying until the provider recovers. Length recovery — when the LLM response has (the model hit its output token limit mid-response): Empty response retry Empty LLM responses (content-less with no tool calls) are retried up to 2 times. The retry message explicitly asks the model to produce output. Stream stall recovery If the streaming connection stalls, the callback starts a new stream segment from the last received token. The model resumes generating where it left off, rather than the entire turn being discarded. Checkpoint restore — after a crash or restart, the in-flight turn checkpoint is loaded. Any tool calls that were in progress at the time of the crash get synthetic error results : Opencode: Effect-TS typed errors and step limits Opencode uses Effect-TS to encode errors in the type system: Step limit . Context overflow recovery performs a single-shot compaction and retry. If the context still overflows after one compaction, the error propagates upward. No multi-stage escalation like Claude Code's drain-and-collapse pipeline. Tool cleanup on error is called when a provider error occurs mid-turn. Any tool calls that were queued but not yet settled are marked as failed with an error message, ensuring the conversation state remains consistent — the model sees that its tool calls were not executed. Recovery paths should be VISIBLE. Flatten the loop to see all terminal conditions in one function. Claude Code's is the exemplar — 12 terminal reasons and 7 recovery transitions, all in one file, all with line numbers. When recovery logic is scattered across files (retry in one, compaction in another, fallback in a third), it becomes impossible to reason about the complete state machine. 2. One-shot guards prevent infinite recovery loops. Every recovery path needs a flag that prevents it from activating twice for the same error. Hermes-agent's 14 boolean flags are the clearest implementation of this principle. Claude Code's max-tokens recovery (escalate once, then inject up to 3 times) is another. Without these guards, a persistent error will loop forever. 3. Tool errors should be LLM-readable. Format tool failures as messages the model can reason about. Every codebase does this, but nanobot's checkpoint restore takes it furthest: orphaned tool calls get synthetic error messages that explain why the result is unavailable. The model can then decide whether to retry the tool call or work around the missing data. 4. The more complex the recovery, the flatter the loop should be. Claude Code has the most recovery paths (7) and the flattest loop (all in ). This is not coincidence. When recovery logic is deeply nested or spread across modules, adding a new recovery path risks breaking existing ones. A flat, visible loop makes every transition explicit and every interaction observable. Exercise: Which recovery strategy? Q2.
